Routes, Data & Production

Volumes 1–3 gave you the toolbox: configs, runes, stores, utilities, and the entire component library. Volume 4 is where the application becomes a website — file-based routes, universal vs server load functions, progressive-enhancement form actions, a real database behind the contact form and lead capture, protected member areas, an admin console, an SEO sitemap that ships from the build, and the production deploy story.

Chapter 01

The SvelteKit Routing Mental Model

Volumes 1–3 built a workshop. We poured the foundation in Vol 1, hand-tuned the tools in Vol 2, and forged every component in Vol 3. None of it was a website yet. A website needs URLs that map to pages, a way to load data, a way to handle form submissions, and a way to separate code that may run on the client from code that must only run on the server. SvelteKit's answer to all four is a single mechanism: file-based routes built from a small alphabet of special files.

That alphabet is the first thing a Principal-L7 reader needs etched in memory before reading a single page of this volume. Get this right and every other chapter is a footnote.

The five file types you will see in every route folder

+page.svelte
The component that renders for this URL. Receives a data prop typed by ./$types. Runs on the server during SSR, then hydrates in the browser. This is the only file in the alphabet that is allowed to contain markup.
+page.ts
A universal load function. Runs on the server during SSR and in the browser during client-side navigation. Whatever it returns becomes data in +page.svelte. Universal load is for data that's safe to compute anywhere — module imports, URL params, fetch with no secret.
+page.server.ts
A server-only load function. Never shipped to the browser. Same return value contract as +page.ts but you can touch $env/dynamic/private, the database, cookies, sessions, and anything else the public internet must not see. Also where actions live — the named handlers that respond to form POSTs.
+layout.svelte / +layout.ts / +layout.server.ts
Same three roles but for a persistent wrapper shared by every nested route. The layout renders once, its children snippet renders the matched page, and crucially layouts don't unmount as you navigate between sibling pages — that's why navbars, cart drawers, and toast queues live there.
+server.ts
An HTTP endpoint. Exports GET, POST, PUT, PATCH, DELETE, OPTIONS, or HEAD as functions returning Response. This is how /sitemap.xml and /robots.txt exist as URLs without a Svelte component.

Plus one error handler:

+error.svelte
Rendered when a load function or action throws. The nearest +error.svelte wins, walking up the directory tree. We have one at the root of src/routes/ — that's our project-wide 404.
Why this matters · the split

The single most important architectural distinction in SvelteKit is universal vs server. +page.ts is "this can run anywhere". +page.server.ts is "this is server-only — the file is never even bundled for the browser." If you import a database client into a .ts file, your build either fails or — worse — leaks credentials into the client bundle. The $lib/server/* convention plus the .server.ts filename suffix are SvelteKit's two enforcement mechanisms. We use both throughout this volume.

How a URL becomes a render

The lifecycle of a request to /indicators/revolution-ranger looks like this:

REQUEST GET /indicators/revolution-ranger │ ▼ ┌──────────────────────────────────────────────────────────┐ │ src/hooks.server.ts → handle({ event, resolve }) │ │ • parse cookies, populate event.locals.user │ └──────────────────────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────┐ │ Layout loads (root → /indicators → /indicators/[slug]) │ │ • +layout.server.ts (server-only, sequential) │ │ • +layout.ts (universal, in parallel) │ └──────────────────────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────┐ │ Page load │ │ • +page.server.ts (server-only, sees locals) │ │ • +page.ts (universal, runs on server here) │ │ → returns { product } │ └──────────────────────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────┐ │ Render: layout(s) → +page.svelte │ │ • props: { data: { product } } │ │ • produces HTML for the wire │ │ • bundles client JS for hydration │ └──────────────────────────────────────────────────────────┘ │ ▼ RESPONSE 200 OK · text/html · (then JS hydrates the runes)

Then the client takes over. On a subsequent navigation — say the user clicks a link to /courses/options-101 — SvelteKit does not reload the page. Instead it re-runs the load functions in the browser, mounts the new +page.svelte, and the layout stays put because /courses/options-101 shares the same root layout.

Glossary · SSR vs CSR vs SPA vs MPA

SSR (server-side rendering): the first response is real HTML the browser can paint immediately. CSR (client-side rendering): JS runs on subsequent navigations and re-renders without a full page reload. SPA: a CSR-only app — no server render. MPA: every navigation is a fresh page load. SvelteKit defaults to SSR + CSR, the best of both: first paint is HTML, every click after is fast.

The page-options dial

Every route can be tuned with three module-level exports:

  • export const prerender = true — build the page at build time into static HTML, ship it from the CDN, never hit a server again.
  • export const ssr = false — skip server rendering, ship an empty shell, mount in the browser. Useful for client-only pages (auth-gated dashboards where there's nothing to render until JS runs).
  • export const csr = false — ship the server-rendered HTML, do not hydrate. Pages turn into pure documents. Useful for legal terms, sitemap fragments, robots.

We use prerender for the SEO endpoints and leave SSR on for the rest. The trade-off is laid out in detail in chapters 16 and 18.

The Real Root Layout

In Volume 1 Chapter 9 we wrote a placeholder root layout — a one-liner that did nothing but render its children. That was deliberate. We needed something compiling while we built the rest. Now we replace it with the production version: an app shell that mounts the Navbar, Footer, AnnouncementBar, CartDrawer, and Toast queue from Vol 3, with one critical twist — it has to not render the marketing chrome on routes that own their own chrome.

sveltesrc/routes/+layout.svelte
<script lang="ts">	import '../app.css';	import type { Snippet } from 'svelte';	import { page } from '$app/state';	import Navbar from '$lib/components/layout/Navbar.svelte';	import Footer from '$lib/components/layout/Footer.svelte';	import AnnouncementBar from '$lib/components/layout/AnnouncementBar.svelte';	import CartDrawer from '$lib/components/layout/CartDrawer.svelte';	import Toast from '$lib/components/ui/Toast.svelte';	import JsonLd from '$lib/components/seo/JsonLd.svelte';	import { organizationLd, websiteLd } from '$lib/utils/jsonld.js';	type Props = { children: Snippet };	let { children }: Props = $props();	// Hide app shell on auth + dashboard + admin routes (they have their own chrome)	const isBareRoute = $derived(		page.url.pathname.startsWith('/dashboard') ||			page.url.pathname.startsWith('/admin') ||			page.url.pathname === '/login' ||			page.url.pathname === '/signup' ||			page.url.pathname === '/forgot-password' ||			page.url.pathname === '/reset-password'	);</script><JsonLd data={[organizationLd(), websiteLd()]} />{#if !isBareRoute}	<AnnouncementBar />	<Navbar />	<main id="main-content">		{@render children()}	</main>	<Footer />{:else}	{@render children()}{/if}<CartDrawer /><Toast />

What's happening, line by line

Line 2 · import '../app.css'
Pulling the global stylesheet from Vol 1 Ch 6. Importing it here, in the root layout, guarantees it ships with every page exactly once. Importing it in +page.svelte instead would be wrong: it would re-emit per page in dev and confuse Vite's CSS handling.
Line 4 · import { page } from '$app/state'
SvelteKit 2.12+ runes API. page.url, page.params, page.data, and page.status are reactive — read them anywhere and your derived state updates as the user navigates. The old $app/stores still works but is the pre-runes API; new code uses $app/state.
Lines 5–9 · the layout-only imports
These five components — Navbar, Footer, AnnouncementBar, CartDrawer, Toast — are the things that need to be persistent. The cart drawer must keep its open/closed state as the user clicks between marketing pages; toast notifications shouldn't disappear when navigating; the navbar shouldn't re-mount and lose its scroll-lock state. SvelteKit's layout contract guarantees that: as long as the layout file boundary doesn't change, those components are not unmounted across navigations.
Lines 10–11 · JSON-LD imports
Two factories from Vol 2 Ch 12. organizationLd() emits Schema.org markup that tells Google "this is the TradeFlex Trading organization, here are our socials." websiteLd() emits the SearchAction so a Google sitelinks search box can show. Both ship from the root layout — every page inherits them automatically.
Lines 13–14 · the props shape
A layout receives a single prop: children, typed as Snippet (the rune-mode replacement for slots). We destructure it the same way every $props() call works.
Lines 16–24 · the bare-route derivation
This is the heart of the file. The marketing site has its own chrome. The dashboard has its own chrome. The admin has its own chrome. The auth pages are full-screen designs. If the root layout naively wrapped everything in Navbar + Footer, those would render twice on dashboard/admin pages. Solution: derive a boolean from page.url.pathname, then conditionally render the chrome. $derived from Vol 2 Ch 3 — it re-evaluates only when its dependencies change, and the only dependency here is the URL.
Line 27 · <JsonLd />
Pass an array of LD objects; the component renders one <script type="application/ld+json"> tag per object inside <svelte:head>. Children layouts and pages call <JsonLd /> with their own arrays — Svelte concatenates head content; nothing fights nothing.
Lines 29–38 · the conditional render
Bare routes get only {@render children()} — no AnnouncementBar, no Navbar, no <main> wrapper, no Footer. Their child layout (e.g. /dashboard/+layout.svelte) provides the chrome. Marketing routes get the full shell.
Lines 40–41 · CartDrawer + Toast — outside the conditional
This is the subtle bit. CartDrawer and Toast live always, on every route. If the user is on the dashboard, clicks "buy" on an upsell card, the cart drawer needs to open. If a form action throws on the admin page, the toast needs to show. They sit outside the {#if} for exactly that reason.
Why a derived for isBareRoute and not just inline checks?

You could inline page.url.pathname.startsWith('/dashboard') in the template. It would work. We pull it into a $derived for two reasons: (1) reading page.url.pathname multiple times in the template re-runs the access tracker on each access — once via $derived, the dependency is observed exactly once; (2) the predicate is six lines of OR — naming it isBareRoute tells the next engineer what the boolean means without making them parse the disjunction.

Pitfall · don't put the cart inside the conditional

When this layout was first built the CartDrawer was inside the marketing branch of the {#if}. Result: pressing the cart button on a dashboard upsell did nothing — there was no drawer mounted to receive the open command. The fix is what you see here: cart + toast are peers of the branch, not inside it.

The root +error.svelte

When any load function in the route tree throws — or any user lands on a URL that has no matching +page.svelte — SvelteKit walks up the directory tree looking for the nearest +error.svelte. Ours sits at the root and serves as both the 404 and the catch-all for runtime errors.

Three details earn discussion:

  • noindex on the SEO component. An error page must never enter Google's index. The Seo wrapper from Vol 2 Ch 12 emits <meta name="robots" content="noindex,follow"> when the prop is true.
  • page.error?.message with optional chaining. We use raw strings for href on the two top buttons because Vol 1's Button takes href: string; we use resolve() here because the link list is auto-generated and the type-check is free insurance against renaming a route and forgetting to update the error page.
TypeScript · the App.PageError shape

The app.d.ts file we wrote in Vol 1 Ch 5 declared interface PageError { message: string; code?: string }. That's the contract every error(status, payload) call in this project must honor. When the admin layout throws error(403, { message: 'Admin access required.', code: 'forbidden' }), the code field is available on page.error.code for the error page to inspect — though we don't currently branch on it.

Chapter 03

+layout.ts & +layout.server.ts — the load pair

The root layout component is mounted. Now: where does its data come from? Two files. They sit beside +layout.svelte in src/routes/, they have identical signatures, and they answer two different questions.

+layout.ts — universal load

Runs on the server during SSR and in the browser during client-side navigation. Bundled for the client. No secrets allowed. Can touch fetch, URL params, anything that's public.

+layout.server.ts — server-only load

Runs only on the server. The file is never even compiled into the client bundle. Can read cookies, locals, $env/dynamic/private, the database. The return value is serialized over the wire to the client.

The universal load

tssrc/routes/+layout.ts
import type { LayoutLoad } from './$types';export const prerender = false;export const ssr = true;export const load: LayoutLoad = () => {	return {};};

Eight lines, two structural decisions:

Line 3 · prerender = false
The default is "inherit". We set it explicitly to false at the root so the whole app is SSR by default. Individual routes that can be prerendered (sitemap, robots, legal terms) opt back in by declaring prerender = true in their own module. Setting the default at the root means we never accidentally prerender a dynamic route — the worst kind of staleness bug.
Line 4 · ssr = true
Also the default; we set it explicitly for documentation. The trade-off is in Ch 18.
Lines 6–8 · load = () => ({})
An empty load. Why have one at all? Because in SvelteKit, the presence of +layout.ts determines a per-route caching boundary. By exporting a (no-op) load, we tell the framework "yes, run a universal load step at the root" — which means children that throw redirect() get to bubble through this layer, and we have a place to hang future global universal data (a feature flag, a public env var) without altering the layout component.

The server load

tssrc/routes/+layout.server.ts
import type { LayoutServerLoad } from './$types';export const load: LayoutServerLoad = ({ locals, url }) => {	return {		user: locals.user,		pathname: url.pathname	};};

This is small but it's doing real work. Two return fields:

user: locals.user
event.locals is the per-request server context. hooks.server.ts from Vol 1 Ch 8 populates locals.user by parsing the tradeflex-mock-user cookie. The server load reads it and ships it to the client. From this point on, anywhere in the app, you can read data.user from a page's data prop, or page.data.user reactively, and know who is logged in.
pathname: url.pathname
Yes, we could read this from $page.url.pathname in any component — and we did in the root layout. We surface it explicitly in data because some child layouts use the server-rendered value to decide what to render before hydration. A redundancy with intent: optimize for legibility, not bytes.
Why is user in the server load, not the universal one?

locals only exists on the server — it's populated by the handle hook before any load runs. A universal load is shared across SSR and the browser; it has no locals binding because the browser doesn't have a hook system. The right place to surface request-scoped state is always +layout.server.ts. SvelteKit handles serialization: the user object is JSON-stringified, included in the initial HTML payload as a <script> tag, and rehydrated as data.user in the browser. We pay one trip in HTML; no fetch roundtrip on first paint.

How the two loads compose

Both layouts' loads run for every request. Universal loads execute in parallel with server loads at the same level. Their return values get merged into data — the server load's keys override the universal one's on collision. Then children inherit that merged data and can extend it further:

request /indicators/revolution-ranger │ ├─► +layout.server.ts (root) │ returns { user, pathname } │ ├─► +layout.ts (root) — parallel with server above │ returns {} │ ├─► (no /indicators/+layout.* — skipped) │ ├─► +page.ts (indicators/[slug]) │ returns { product } │ └─► merged: data = { user, pathname, product } └─► layout.svelte sees user, pathname └─► +page.svelte sees user, pathname, product

Note that the page-level load doesn't have to know about user — the layout already provided it, and the merge propagates downward. This is what makes "every page sees the logged-in user" a one-liner in the root server load and not a per-page repetition.

Pitfall · don't put the database in +layout.server.ts

It's tempting. "Just query the user's row from the DB on every request and put it in locals." Do not. Layout loads run on every page transition, including ones that have no business hitting the DB. Keep locals.user populated from a cheap cookie parse and let pages that actually need the row (the dashboard, the billing page) make their own targeted query. Vol 1 Ch 8's hooks.server.ts does it right: cookie → in-memory fake user → done.

The home page replaces Vol 1's placeholder. It is also the canonical template every other marketing route follows. Read this one closely; the others — /subscription, /about, /free-guide, /legal/* — are variations on the same composition pattern.

The home is structurally five things: SEO tags, hero, value pillars, a featured-products grid, proof stats, playbook, lead magnet, testimonials, FAQ, bottom CTA. What's happening

Every import is a Vol 2 or Vol 3 deliverable
Look at the list: Hero and TrustBar and FeatureGrid are the marketing primitives we shipped in Vol 3 Ch 14–15. ProductCard and PricingCard are from Vol 3 Ch 10–11. EmailCaptureForm is from Vol 3 Ch 13. INDICATORS, COURSES, DAY_TRADING_PLANS, GENERAL_FAQS are the fixtures from Vol 2 Ch 14. fadeUp and stagger are the GSAP attachments from Vol 2 Ch 13. None of these values change. No keystroke updates them. They are pure module-scope data. We do not wrap them in $state — that would be over-reactive (waste a proxy on an immutable array). Vol 2 Ch 2 covered this: use the lightest tool that gets the job done.

The home page has no +page.ts or +page.server.ts. All its data is imported statically at the module level — fixtures from $lib/data/*. SvelteKit treats a route without an explicit load as "no data needed; just render". If we later move the indicators to a CMS, we'd add a +page.ts that fetches them; right now the data is build-time content, and Vite tree-shakes the array imports into the page bundle directly.

The subscription page — same pattern, plus JSON-LD

The subscription page is a near-twin of the home in shape: hero with breadcrumbs, then the three pricing cards from DAY_TRADING_PLANS, then a compare table, then testimonials and FAQ. The one structural addition is JSON-LD for the service offering:

From Vol 2 Ch 12. Maps an array of price/cadence pairs to a Schema.org Service object with an offers array. Google understands this and can show price-aware results. We .map the plans into the lighter shape the factory expects — the factory doesn't need the full plan object, just three fields.

The keyed each from Vol 2 Ch 14. (plan.id) is the key — Svelte uses it to identify and reuse DOM nodes across renders. We use IDs, never indices, for the reason from Vol 2: when the array reorders, indices lie about identity and Svelte reuses the wrong components.

About, legal · pure static

The /about, /legal/terms, /legal/privacy, and /legal/refund routes follow the same template. All four are leaves: no +page.ts, no +page.server.ts, no params, no dynamic data. They could be prerendered — and in Ch 18 we discuss the trade-off — but for Phase 1 we leave them SSR-only so the build is faster and they re-fetch the (unchanging) layout chrome on demand.

The About page is interesting only in one respect: it emits personLd() for each team member, so Google's knowledge graph can connect Jordan Vega's profile to TradeFlex Trading the organization. The whole page is roughly 400 lines of markup and 200 lines of CSS — too long to reproduce here. The script block is forty lines of const arrays for principles, team, milestones, and stats; the rest is layout. No load function, no reactivity, no surprises.

Why bundle "legal" under one folder?

Three routes — /legal/terms, /legal/privacy, /legal/refund — share a logical grouping. We could have flattened them to /terms, /privacy, /refund. The nested folder costs one URL segment and buys us: (1) a single place in the file tree to grep, (2) a future /legal/+layout.svelte if we ever want shared chrome on legal pages, (3) the option to Disallow: /legal in robots.txt with a single line. The catalog pages — /indicators, /indicators/[slug], /courses, /courses/[slug] — are where SvelteKit's dynamic routes earn their keep. The detail-route load function · indicators

The kind !== 'indicator' guard
The product catalog mixes indicators and courses. If someone hits /indicators/options-101 (a course slug), we 404 — we don't render the course under the wrong URL prefix. Cleaner than 200ing on a mis-routed product.

Throws a typed HTTPError that SvelteKit catches. Result: nearest +error.svelte renders, status code 404, no leaked stack traces. Always use this helper instead of throw new Error(...) for HTTP errors.

The courses sibling file is identical except kind !== 'course'. One pattern, two routes.

Why universal (+page.ts) not server (+page.server.ts)?

Because getProductBySlug is a synchronous lookup against a static array. No secret, no DB. Universal load means: during SSR, the function runs on the server; during client-side navigation (e.g., user clicks from /indicators to a detail page), it runs in the browser and skips the server round-trip entirely. This is why catalog pages feel instant after first paint.

Why this matters · the load-where-you-can rule

Default to universal (+page.ts) until you can't. Switch to server-only (+page.server.ts) only when you need: a secret, the database, cookies, or anything else the browser must not see. Misclassifying once will eventually leak credentials. Misclassifying the other way just costs a network round-trip. Bias toward universal.

The catalog list pages

The listing routes (src/routes/indicators/+page.svelte, src/routes/courses/+page.svelte) have no load function at all. They import the data directly from $lib/data/products and iterate. That works because:

  • The pages aren't dynamic — same render every time
  • SvelteKit will still server-render them on first visit, hydrate on the second, and use the same module-level data both times

This is the lightest possible route: a single .svelte file, no .ts sibling. Use this pattern whenever data is module-level constants.

The cart route

Chapter 06

The cart route

The cart page is interesting because it has no server data at all. The cart store from Vol 2 Ch 8 lives in the browser (with localStorage persistence). The page just renders it. No load function, no DB query — but a lot of UI: line items, quantity steppers, subtotal math, the "Continue" button intentionally disabled with a Phase 2 tooltip.

Why no load function?

Three reasons:

  1. The cart lives in localStorage, which doesn't exist on the server. The server can't render the cart's contents anyway.
  2. SSR-rendering an empty cart, then hydrating with the real cart from localStorage, would produce a visible flash of "your cart is empty" before the items appear.
  3. The cart store imports $app/environment and gates its localStorage code behind if (browser) — so it's safe to render during SSR (it just shows empty until the browser takes over).

What the page does instead

src/routes/cart/+page.svelte imports the cart singleton and renders against its reactive fields:

svelteexcerpt
import { cart } from '$lib/stores/cart.svelte';import { formatPrice } from '$lib/utils/money';{#if cart.isEmpty}	<EmptyState title="Your cart is empty" ... />{:else}	{#each cart.lines as line (line.id)}		<!-- line item: name, quantity stepper, remove button -->	{/each}	<p>Subtotal: {formatPrice(cart.subtotalCents)}</p>{/if}

Three derived values from Vol 2 — cart.isEmpty, cart.lines, cart.subtotalCents — drive the entire view. Add or remove an item from anywhere else in the site, and this page updates automatically because they're the same singleton.

Why this matters · stateful pages don't need a server

The cart is a perfect example of when SvelteKit's "no load function" is the correct answer. There's no server data, so we don't pretend to have any. The page renders the empty state during SSR (it's static HTML), then the browser hydrates and the cart store immediately replaces it with the saved cart. One flash-free transition. Anyone instinct to add a +page.server.ts with cookies/DB persistence should stop and ask: does this need server state? Often no — localStorage + a rune store is enough.

The Phase-2 stub

The page has a "Continue to checkout" button that is explicitly disabled. Hovering shows a tooltip "Checkout in Phase 2 — payment processor not yet integrated." This is honest engineering: instead of a button that looks functional but errors when clicked, we show users exactly what state the feature is in. Cheap to write, expensive to forget.

Walk-through of the capture action

Named actions: actions.capture
SvelteKit's form actions can be named. The form uses <form action="?/capture" method="POST">. The same file also exports subscribe for the footer newsletter — different action, same DB table, different post-submit behavior.
Rate limit first
Before anything expensive (DB write), check that the client hasn't sent too many. Five per minute per IP. If exceeded, fail(429, { error }) — SvelteKit serializes this back to the client where the form's $page.form data exposes it.
safeParse with Zod
Browser-side validation (the form does it too) is for UX. Server-side Zod is the source of truth. The schema (from Vol 2 Ch 11) enforces format, length, and the honeypot rule. safeParse returns a discriminated union — { success: true, data } or { success: false, error }.
The honeypot check
if (parsed.data.website) return fail(400, ...) — if a bot filled the hidden field, reject silently with a generic message. No "honeypot detected" reveal — that would let attackers know to skip the field.
crypto.randomUUID() for IDs
Modern Web Crypto API. Returns a v4 UUID. The lead_ prefix is a project convention — every ID is human-recognizable as to its kind. Helps with debugging: lead_a1b2c3... vs msg_d4e5... at a glance.
.toLowerCase() on the email before insert
"Alex@Example.com" and "alex@example.com" are the same address. Normalize once at the boundary. Saves later deduplication migrations.
redirect(303, ...)
303 is the POST/Redirect/GET pattern — after a successful POST, send the browser to a new URL via GET. Prevents "are you sure you want to resubmit?" if the user reloads.
The success URL carries the email as a query param
The success page can greet the user by email without persisting any session state. The companion +page.ts on the success route reads it back via url.searchParams.

Companion: the rate limiter

The checkRateLimit import is from a small server-only module we added to defend the form endpoints. Here's the full file:

typescriptsrc/lib/server/rateLimit.ts (excerpt)
const buckets = new Map<string, number[]>();export function checkRateLimit(key: string, opts: RateLimitOptions): RateLimitResult {	const now = Date.now();	const cutoff = now - opts.windowMs;	const existing = buckets.get(key);	const timestamps = existing ? existing.filter((t) => t > cutoff) : [];	if (timestamps.length >= opts.limit) {		const oldest = timestamps[0] ?? now;		return { ok: false, retryAfterMs: Math.max(0, oldest + opts.windowMs - now) };	}	timestamps.push(now);	buckets.set(key, timestamps);	return { ok: true };}

A sliding-window limiter backed by an in-memory Map. Each key (e.g., 'lead:1.2.3.4') holds the recent timestamps for that bucket. On every call we prune timestamps older than windowMs, count what's left, and either accept or return the retry-after delay until the oldest still-active timestamp expires.

Pitfall · in-memory limits don't scale

This Map lives in process memory. Two server instances = two separate buckets, doubling the effective rate. Fine for a single-instance deploy (Phase 1). For multi-instance production, swap the Map for Redis (the file comment says exactly that). The interface stays identical; only the storage backend changes.

Two routes that ship XML and plain text instead of HTML: /sitemap.xml and /robots.txt. They use SvelteKit's +server.ts file type, which exports HTTP method handlers (GET, POST, etc.) returning Response objects. Both are prerendered at build time — a single static file each, served from CDN edge, no server compute per request.

The sitemap

Anatomy

Folder name sitemap.xml/
The route is /sitemap.xml because the folder is literally named that. SvelteKit's file router doesn't care that "xml" looks like a file extension — it treats the whole segment as a route name. The actual logic lives in +server.ts inside.
export const prerender = true
The whole point. At build time, SvelteKit calls our GET once, captures the response body, and writes sitemap.xml to the static output directory. In production this file is served by the CDN with zero server compute. Per the Svelte docs in Vol 1's Chapter 4 reference: "any two users hitting it directly must get the same content" — sitemaps are identical for every user. Perfect fit.
GET = async () => new Response(xml, { headers: ... })
The function name maps to the HTTP method. SvelteKit also supports POST, PUT, PATCH, DELETE, OPTIONS, HEAD. The return must be a Response object (or a Promise<Response>) — same Web platform Response the browser fetch API hands back.
The STATIC_ROUTES array
Ten hand-curated routes with priorities (1.0 home page, 0.95 main funnel page, 0.3 legal) and changefreq hints. Google treats these as advisory, not binding — they help the crawler prioritize.
Dynamic entries from the product catalog
We map each indicator and course into a URL entry. The entire catalog is reachable at build time because Vol 2's products.ts is a static module — no DB lookup needed. Add a new product to the fixture and the sitemap regenerates next build.
Hand-rolled XML
No XML library. The output is fifteen lines of template literals. Simpler than pulling in xmlbuilder2 for one route. The schema URL (sitemaps.org/schemas/sitemap/0.9) is the formal sitemap protocol — search engines validate against it.

The robots file

typescriptsrc/routes/robots.txt/+server.ts
import { SITE_URL } from '$lib/utils/seo.js';export const prerender = true;export const GET = async () => {	const body = `User-agent: *Allow: /Disallow: /dashboardDisallow: /adminDisallow: /cartDisallow: /loginDisallow: /signupDisallow: /forgot-passwordDisallow: /reset-passwordSitemap: ${SITE_URL}/sitemap.xml`;	return new Response(body, {		headers: { 'Content-Type': 'text/plain; charset=utf-8' }	});};
Eight Disallow lines
Tell crawlers to skip our private routes. Search engines won't index the dashboard or admin (even if they did somehow reach those URLs, the admin layout would return 403). The login/signup pages are also Disallowed — they're tools, not content.
The Sitemap: line at the end
Points crawlers to the sitemap above. Crawlers reach /robots.txt first and follow this link. The full SITE_URL is used because robots.txt values are URL-absolute by spec.
Content-Type: text/plain
Important: robots.txt is plain text, not HTML. Some servers serve it as HTML by mistake, which silently breaks crawler parsing.
Why prerender both?

Neither file changes between requests. Prerendering means SvelteKit calls the GET once at build time, captures the output, and ships static files. The runtime cost is zero — no Lambda cold start, no server compute, no rate limiting. Both files live next to your CSS and JS in the CDN. Crawlers, which hit them constantly, never touch your origin.

Chapter 17

Drizzle · schema, client, seed

Three small files implement the entire database layer. Drizzle uses TypeScript-as-schema: the table definitions live in code, the migrations are generated from diffs, and the query builder enforces row types from the same source. One place to change a column; every consumer compiles or breaks.

1. The schema · src/lib/server/db/schema.ts

typescriptsrc/lib/server/db/schema.ts
import { sql } from 'drizzle-orm';import { sqliteTable, text, integer, index } from 'drizzle-orm/sqlite-core';/* ─────────────── Leads (email captures) ─────────────── */export const leads = sqliteTable(	'leads',	{		id: text('id').primaryKey(),		email: text('email').notNull(),		source: text('source').notNull().default('free-guide'),		createdAt: integer('created_at', { mode: 'timestamp_ms' })			.notNull()			.default(sql`(unixepoch() * 1000)`)	},	(t) => [index('leads_created_at_idx').on(t.createdAt)]);/* ─────────────── Users (schema-only stub) ─────────────── */export const users = sqliteTable('users', {	id: text('id').primaryKey(),	email: text('email').notNull().unique(),	name: text('name').notNull(),	passwordHash: text('password_hash'),	avatarUrl: text('avatar_url'),	role: text('role', { enum: ['member', 'admin'] }).notNull().default('member'),	createdAt: integer('created_at', { mode: 'timestamp_ms' })		.notNull()		.default(sql`(unixepoch() * 1000)`)});/* ─────────────── Products, plans, cart, contact messages ─────────────── *//* same shape — see source for full content */export const contactMessages = sqliteTable(	'contact_messages',	{		id: text('id').primaryKey(),		name: text('name').notNull(),		email: text('email').notNull(),		subject: text('subject').notNull(),		body: text('body').notNull(),		createdAt: integer('created_at', { mode: 'timestamp_ms' })			.notNull()			.default(sql`(unixepoch() * 1000)`)	},	(t) => [index('contact_messages_created_at_idx').on(t.createdAt)]);export type Lead = typeof leads.$inferSelect;export type NewLead = typeof leads.$inferInsert;export type ContactMessage = typeof contactMessages.$inferSelect;
sqliteTable('leads', { ... }, (t) => [index('leads_created_at_idx').on(t.createdAt)])
The third argument to sqliteTable is a function that returns table-level constraints — indexes, unique compounds, foreign keys. We index created_at on both leads and contact messages because the admin reads ORDER BY created_at DESC LIMIT 200. Without the index, that query scans the whole table; with it, the engine reads the tail in constant time.
integer('created_at', { mode: 'timestamp_ms' })
Drizzle's storage hint. SQLite stores integers; the mode: 'timestamp_ms' tells the ORM to marshal them to/from JavaScript Date objects automatically on the way in and out. The schema gets ergonomic Dates without losing the cheap integer storage.
.default(sql`(unixepoch() * 1000)`)
SQL-side default. SQLite's unixepoch() returns seconds since epoch; multiplied by 1000, milliseconds since epoch — matches what JavaScript's Date.now() uses. Means new rows get a timestamp at insert time without the app having to provide one.
text('role', { enum: ['member', 'admin'] })
Drizzle's tagged enum. At the DB level it's a CHECK constraint (role IN ('member', 'admin')). At the TypeScript level it's a string union. One declaration, two enforcements.
$inferSelect and $inferInsert
The TypeScript types for what a row looks like when you read it (id: string; email: string; source: string; createdAt: Date) and what you must provide to insert one (id: string; email: string; source?: string; createdAt?: Date — defaults make some fields optional). Used in our server loads to type the return arrays.

terminal

Pitfall · never edit a shipped migration

Once a migration file is checked in and applied anywhere (CI, prod, a teammate's laptop), treat it as immutable. Editing it later means a fresh checkout will apply the new version, but anyone who already applied the old one is now in an inconsistent state. The right move is always: write a new migration that does the additional change. Drizzle's migrator tracks which migrations have been applied; the workflow assumes append-only.

Chapter 18

Production · adapters & deploy

The last chapter. Everything you've built — twelve volumes' worth of code — now has to ship to a real host that answers HTTPS at a domain. SvelteKit's answer is adapters: small plugins that take your build output and produce platform-specific deployment artifacts. Pick the right one and the rest is configuration.

What an adapter actually does

At build time SvelteKit produces a generic build manifest: which routes are server-rendered, which are prerendered, which are static, what code each handler needs. An adapter consumes that manifest and emits the shape the target platform expects:

  • adapter-node → a standalone build/ directory with a node build entry that boots an Express-style server.
  • adapter-vercel → a .vercel/output/ tree of serverless and edge function bundles plus static assets.
  • adapter-netlify → Netlify Functions + a publish/ static dir.
  • adapter-cloudflare → a Cloudflare Workers script with KV/D1 bindings exposed.
  • adapter-static → just HTML, CSS, JS files. No server. Requires every route to be prerenderable.
  • adapter-auto → sniffs the deploy environment and chooses one of the above automatically.

Why we ship Phase 1 with adapter-auto

Vol 1 Chapter 6 installed @sveltejs/adapter-auto. It's the right default while the deployment target is still undecided: push to Vercel, deploy works; push to Netlify, deploy works; the build figures it out. The downside is that the chosen adapter resolves at deploy time — your local pnpm build doesn't tell you exactly what will run in production.

Once you've committed to a host, swap in the specific adapter for two reasons: (1) build-time validation that the platform's constraints are met, (2) access to platform-specific config (e.g., Vercel's edge/serverless split, Cloudflare's bindings).

Prerender what you can

The cheapest request is the one your server doesn't handle. Every route that doesn't need per-request data should be prerendered. In Vol 4 we already did this for /sitemap.xml and /robots.txt. The same applies to about and legal pages — they could earn export const prerender = true with no code change:

typescriptsrc/routes/legal/terms/+page.svelte (add a sibling +page.ts)
// src/routes/legal/terms/+page.tsexport const prerender = true;

Same for /legal/privacy, /legal/refund, /about. They become static files at build time, served from CDN edge with zero compute. The catch: any page with a form action cannot be prerendered (the form needs a server to POST to). That's why /free-guide and /contact stay SSR — they have actions.

SSR vs CSR vs prerender · the decision tree

Does the page have a form action? ├── Yes → SSR (default). Cannot prerender. └── No → Does any two users see identical content? ├── Yes → prerender = true. Free, fast, CDN. └── No → SSR + cache headers. Per-request server work, but cacheable at the edge via Cache-Control. Does the page only work in the browser (uses window/document at top level)? └── Yes → ssr = false. Renders an empty shell, hydrates in browser. (Rare; we don't use this in Phase 1.)

Environment variables in production

The two we set in Vol 1's .env file (DATABASE_URL, PUBLIC_SITE_URL) translate to platform-specific env config:

  • Vercel: Project Settings → Environment Variables. Mark DATABASE_URL and DATABASE_AUTH_TOKEN as Secret.
  • Netlify: Site → Environment variables. Same.
  • Railway/Fly/Render: their respective env-config UIs.

The PUBLIC_-prefixed variables (we have PUBLIC_SITE_URL) end up in the client bundle. Anything without the prefix stays server-only. SvelteKit enforces this at compile time — try to import DATABASE_URL from $env/static/public and the build errors.

The Phase 2 production checklist

  1. Pick and install a specific adapter (Vercel or Node, most likely).
  2. Provision a Turso libsql database, set DATABASE_URL and DATABASE_AUTH_TOKEN as host env vars.
  3. Run pnpm db:generate locally, commit the migration files, ensure CI runs pnpm db:push on deploy.
  4. Add export const prerender = true to the legal and about pages.
  5. Wire real auth — argon2id password hashing, signed session cookies, login rate limiting.
  6. Replace mock cookie-based admin promotion with role lookup on the real users table.
  7. Swap in-memory rate limiter for Redis (or Cloudflare KV).
  8. Wire Stripe (or Paddle, LemonSqueezy) for actual checkout.
  9. Set up Sentry / Datadog / OpenTelemetry for production error tracking.
  10. Production smoke tests: end-to-end Playwright suite hitting Vercel preview deploys on every PR.
The Principal-L7 mindset · ship early, ship visible

What we've built across four volumes is a complete, observable, secure Phase 1. The UI works, the routes resolve, the forms persist, the admin reads back what the public wrote. Two real DB tables and ten more scaffolded for Phase 2. The seed is idempotent. The rate limiter defends the only two endpoints that touch the DB. This is what shipping looks like — not "feature-complete," but "honest about what's done and what's stubbed, with the stubs marked clearly enough that Phase 2 is mechanical."

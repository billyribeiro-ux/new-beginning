Foundations

In this first volume we'll build the bedrock of a production-grade trading platform: the terminal scaffold, the configuration files that govern every line of code we'll ever write, the TypeScript primer that explains why we type our code, and the design tokens that turn raw CSS into a coherent visual language. By the end, pnpm dev will run a dark, gold-accented shell with all the bones in place.

Think of these twelve chapters as laying foundations for a building. None of it looks like much yet — but every floor we add in Volumes 2, 3, and 4 stands on what we do here.

Chapter 01

The North Star

Before we type a single command, let's make sure we understand what we are building and why we are choosing each tool. A Principal Engineer never opens a terminal without a destination.

What we are building

TradeFlex Trading is a luxury-feeling, dark-canvas-and-gold web platform that sells day-trading indicators, courses, and a subscription. By the end of the series it will have:

  • A marketing site: home, indicators catalog with detail pages, courses catalog with detail pages, subscription plans, about, contact, cart, lead-magnet ("free guide"), and legal pages.
  • A full auth surface: login, signup, forgot-password, reset-password (UI only in Phase 1 — real auth is a future volume).
  • A logged-in member dashboard with billing, courses, indicators, downloads, notifications, profile, and subscription management.
  • An admin back-office with KPI overview, products, courses, plans, orders, customers, leads (live database), messages (live database), and settings.
  • SEO endpoints — /sitemap.xml and /robots.txt — that real search engines crawl.
  • A real SQLite database (via Drizzle ORM) that the lead-capture form and contact form write to, and that the admin reads from.

The stack — every choice, justified

One of the most expensive failure modes for a junior engineer is "I picked this because the tutorial used it." A Principal Engineer can explain, in one sentence, why every tool in their stack is there. Here's our roster:

Framework SvelteKit Gives us file-based routing, SSR (server-side rendering), form actions, API endpoints, and deployment adapters out of the box. We never have to wire up Express or React Router.

⚡ Why this matters

Notice three deliberate absences: no Tailwind, no Bootstrap, no React. Tailwind is a fine tool but it explodes class lists; we will use plain CSS with design tokens so that changing a brand color is a one-line edit. Bootstrap predates everything modern. React is the most popular framework in the world, but Svelte gives us a smaller bundle and a more direct mental model — and we are optimizing for this product, not career signaling.

Setting the Stage
Chapter 02

Setting the Stage

A senior engineer is recognizable not by what they type, but by what their environment looks like before they type anything. Deterministic versions, the right editor, a clean terminal. Let's make sure yours is ready.

1. Install (or verify) Node.js ≥ 20

Node.js is the JavaScript runtime that powers everything off the browser. Vite uses it, our dev server uses it, our tooling uses it. We need Node 20 or higher because SvelteKit dropped support for older versions.

Open your terminal and check:

terminal 
$ node --version
v20.11.1

$ npm --version
10.2.4

If your Node version is below 20, install the latest LTS from nodejs.org, or — better — install nvm (Node Version Manager) which lets you switch versions per-project.

📖 Terminology · LTS

LTS means "Long-Term Support." Node.js releases a new version every six months, but only even-numbered versions become LTS and receive security patches for 30 months. Always pick LTS unless you have a specific reason not to.

2. Install pnpm

You may know npm already — it's the default package manager that ships with Node. We will use pnpm instead. Same idea: it downloads JavaScript libraries from a registry into a node_modules folder. But pnpm:

  • Is roughly 2× faster than npm.
  • Saves disk space by storing each package version once globally and hard-linking it into each project.
  • Has strict dependency resolution that prevents the "phantom dependency" bug — where your code accidentally imports a transitive dependency that isn't in your package.json.

Install it with one command:

terminal 
$ npm install -g pnpm
added 1 package in 2s

$ pnpm --version
11.3.0

3. Editor: VS Code + the Svelte extension

You can use any editor, but I strongly recommend Visual Studio Code (free, made by Microsoft, the de-facto editor for web work). After installing it, open the Extensions panel and install:

Svelte for VS Code
Adds syntax highlighting, autocomplete, type-checking, and "go to definition" for .svelte files. Without this, working on Svelte feels like editing plain text.
Prettier — Code formatter
Auto-formats your code on save. Removes 90% of style nitpicks from code review.
ESLint
Shows linting errors directly in your editor as red squiggles. Catches bugs before you ever run the code.

In VS Code's settings (Cmd+, on Mac, Ctrl+, on Windows/Linux), enable "Format On Save" and set "Default Formatter" to Prettier.

⚡ Why this matters

A messy environment produces messy code. The five minutes you spend now on tooling will save you hours of "why doesn't autocomplete work" frustration later. Principal Engineers don't fight their tools — they configure them once, correctly.

4. Pick a project folder

Pick a folder where you keep code projects. I keep mine at ~/code. Open a terminal there:

terminal 
$ cd ~/code
$ pwd
/Users/you/code

That's where the next chapter begins.

Scaffolding from the Terminal

Now we create the project. We will not use a GUI or a "new project" wizard — we will use the terminal. This is how every real engineering team starts a project, and it forces us to understand each file that gets generated.

Create the directory

From inside your ~/code folder, run:

terminal 
$ mkdir tradeflex-trading
$ cd tradeflex-trading
📖 Terminology · mkdir, cd

mkdir means "make directory." It creates a folder. cd means "change directory." It moves your terminal's cursor into that folder, so subsequent commands operate inside it. Together, those two commands say: "create a new folder called tradeflex-trading and put me inside it."

Run the SvelteKit scaffolder

SvelteKit ships with a tool called sv that generates a new project. We invoke it through pnpm create:

terminal 
$ pnpm create svelte@latest .
create-svelte version 6.x
┌  Welcome to SvelteKit!
│

The . at the end means "scaffold into the current directory" (which is empty — we just made it). The tool will ask you a sequence of questions. Here is how to answer them and why:

Q1: Which Svelte app template?

Pick "SvelteKit minimal". The "demo app" template gives you a sample to-do app we'd just have to delete. Minimal gives us the absolute floor — a home page, a layout, and nothing else.

Q2: Add type checking with TypeScript?

Pick "Yes, using TypeScript syntax". The alternative — "JavaScript with JSDoc comments" — is fine but uses a less ergonomic syntax. We're going strict. The next chapter explains why.

Q3: What would you like to add?

You can use spacebar to toggle and enter to confirm. Pick:

  • ESLint — for linting
  • Prettier — for formatting

Skip Playwright, Vitest, and the rest for now. We'll add specific tools later, deliberately.

What just happened

After the scaffolder finishes, look at your folder:

terminal

Each of these files has a purpose. We will replace or extend almost every one of them over the next nine chapters — and you will know exactly why we changed what we changed.

Install dependencies

The package.json file lists the libraries the project needs. pnpm install reads it and downloads them into node_modules:

terminal 
$ pnpm install

You now have node_modules/ (the downloaded libraries — usually huge) and pnpm-lock.yaml (the exact versions, which we commit so collaborators get identical builds).

Initialize git

Git is version control — it tracks every change we make. Initialize it now so we can commit our scaffold:

terminal 
$ git init

$ git commit -m "chore: scaffold SvelteKit project"

Never commit node_modules/. The folder can be 500 MB and is reproducible from pnpm-lock.yaml at any time. The default .gitignore already excludes it — verify by running cat .gitignore.

That's the scaffold. From here we own every file. Chapter 4 is where the real engineering thinking starts — with TypeScript.

TypeScript — What & Why
Chapter 04

TypeScript — What & Why

This is the most important conceptual chapter of Volume 1. We're going to slow down and understand TypeScript thoroughly, because every .ts and .svelte file we write from now on relies on it. If you've ever felt that TypeScript was "JavaScript with extra annotations," that's about to change.

What JavaScript is — and what's wrong with it

JavaScript is the language that runs in every web browser. It's dynamically typed, which means a variable can hold any kind of value, and the type can change at runtime:

javascriptexample.js 
let price = 99;          // price is a numberprice = 'free';             // now it's a string — JavaScript doesn't mindprice = { amount: 99 };   // now it's an object — still fineconsole.log(price.toUpperCase()); // runtime crash 💥

That last line crashes — but only when your user runs the code in their browser. You never saw the bug because your editor never warned you. In a small script this is fine. In a 130-file production application, it's a recipe for shipping bugs.

What TypeScript adds

TypeScript is a layer on top of JavaScript that lets you describe what kind of value each variable holds. The TypeScript compiler reads your code, checks that you only do operations valid for those kinds, and then erases the type information to produce plain JavaScript that runs in any browser.

Here is the same example, in TypeScript:

typescriptexample.ts 
let price: number = 99;price = 'free';             // 🛑 compile error: Type 'string' is not assignable to type 'number'

The compile error appears in your editor immediately, before you ever run the code. Multiply that across every variable in a real application and you've eliminated an entire family of bugs.

𝓣 TypeScript Spotlight · "Erased at compile time"

The browser doesn't know about TypeScript. .ts files are compiled to .js files, with all the type annotations removed, before they're sent to the browser. This means TypeScript has zero runtime cost — every type check happens in your editor / build step. Your shipped bundle is plain JavaScript.

The grammar of TypeScript — by example

Here is essentially every TypeScript syntax you'll see in this project, in order of frequency. Don't memorize — just recognize.

1. Variable annotations

typescriptsnippets/types.ts 
let name: string = 'Alex';let age: number = 42;let active: boolean = true;let tags: string[] = ['gold', 'premium'];let maybeUser: string | null = null;  // union type: string OR null

2. Function signatures

typescriptsnippets/functions.ts 
function formatPrice(cents: number): string {  return `$${(cents / 100).toFixed(2)}`;}// Arrow form with two arguments and an inferred return type:const add = (a: number, b: number) => a + b;

3. Object shapes (interface and type)

typescriptsnippets/interface.ts 
interface Product {  id: string;  name: string;  priceCents: number;  tagline?: string;            // optional field — note the ?}const indicator: Product = {  id: 'ind_001',  name: 'Volume Profile Pro',  priceCents: 14900};

4. Union and literal types

A union | says "this can be one of several things." A literal type pins a string or number to one exact value:

typescriptsnippets/unions.ts 
type Role = 'member' | 'admin';   // only those two strings are validtype CartLineKind = 'product' | 'plan';function grantAccess(role: Role) {  if (role === 'admin') // ✅ TypeScript narrows the type inside the if    openAdminPanel();}

5. Generics

Generics let you parameterize a type, like a function for types. They look intimidating but are everywhere in modern code:

typescriptsnippets/generics.ts 
// A function that takes an array of anything and returns its first elementfunction first<T>(items: T[]): T | undefined {  return items[0];}const n = first([1, 2, 3]);     // n is number | undefinedconst s = first(['a', 'b']);     // s is string | undefined

Now: writing our tsconfig.json

Every TypeScript project has a tsconfig.json file that tells the compiler how strict to be. The scaffolder generated a basic one — we are going to make it stricter on purpose.

Open tsconfig.json at your project root and replace its entire contents with this:

jsontsconfig.json 
{	"extends": "./.svelte-kit/tsconfig.json",	"compilerOptions": {		"allowJs": true,		"checkJs": true,		"esModuleInterop": true,		"forceConsistentCasingInFileNames": true,		"resolveJsonModule": true,		"skipLibCheck": true,		"sourceMap": true,		"strict": true,		"noUncheckedIndexedAccess": true,		"noImplicitOverride": true,		"exactOptionalPropertyTypes": false,		"moduleResolution": "bundler",		"verbatimModuleSyntax": true,		"isolatedModules": true,		"target": "ES2022",		"module": "ESNext",		"lib": ["ES2022", "DOM", "DOM.Iterable"]	}}

Now, every option, line by line.

"extends": "./.svelte-kit/tsconfig.json"

SvelteKit auto-generates a tsconfig with all the Svelte-specific path aliases and JSX-equivalent settings. We extend from it so we inherit those defaults, then override what we want. The .svelte-kit/ folder is generated on first run — don't worry that it doesn't exist yet.

"allowJs": true and "checkJs": true

Allows us to mix plain .js files into the project (like config files) and type-checks them too. Even untyped JS gets the benefit of type inference.

"esModuleInterop": true

Smooths over CommonJS / ES Modules interop quirks. Old npm packages used module.exports; modern ones use export. This flag makes import foo from 'old-package' just work either way.

"forceConsistentCasingInFileNames": true

Prevents the "works on Mac, breaks on Linux" bug. macOS file systems are case-insensitive; Linux servers are case-sensitive. If you import ./Button but the file is named ./button.svelte, it works locally and fails in production. This flag makes the compiler reject that.

"resolveJsonModule": true

Lets you import data from './something.json' with full type inference. Useful for static fixtures.

"skipLibCheck": true

Don't type-check the types inside our dependencies. Saves a lot of compile time, and we can't fix those bugs anyway.

"sourceMap": true

Generates .map files alongside compiled output, so when an error happens in production the stack trace points to the .ts source, not the compiled .js.

"strict": true important

This single flag turns on a bundle of strictness rules: noImplicitAny (forbids untyped function parameters), strictNullChecks (you must explicitly handle null and undefined), strictFunctionTypes, and several more. It is the dividing line between "TypeScript as JavaScript with optional types" and "TypeScript as a real safety net."

𝓣 TypeScript Spotlight · strictNullChecks

Without strict null checks, a value typed as string could secretly be null, and you'd never know until it crashed in production. With strict on, you must declare string | null if a value can be absent, and the compiler forces you to handle the null case before you can call .toUpperCase() on it.

"noUncheckedIndexedAccess": true

By default, const x = arr[0] types x as T — even though the array might be empty and x would actually be undefined. This flag types it correctly as T | undefined, forcing you to handle the empty case. Catches real bugs.

"noImplicitOverride": true

When you override a method on a parent class, you must write override explicitly. Prevents accidental shadowing when the parent's method gets renamed.

"exactOptionalPropertyTypes": false

If this were true, the compiler would treat { name?: string } as different from { name?: string | undefined } — an annoying distinction that breaks too much real-world code. We turn it off.

"moduleResolution": "bundler"

Tells the compiler we use a bundler (Vite) that resolves imports — not Node's runtime resolution. Enables features like extensionless imports and conditional package exports.

"verbatimModuleSyntax": true strict

Imports are emitted verbatim — no implicit type-only imports. You must write import type { Foo } from '...' when you only need Foo for typing. This guarantees the compiled output exactly matches what you wrote, which is crucial for isolatedModules below.

"isolatedModules": true

Required by Vite. Each file must be compilable in isolation — no whole-program optimizations that need cross-file analysis. Bundlers compile files one at a time and in parallel; this flag ensures we write code compatible with that.

"target": "ES2022"

The JavaScript version to compile down to. ES2022 includes top-level await, class fields, and the ?. optional-chaining operator natively. Every modern browser supports it. Older targets would need polyfills.

"module": "ESNext"

Use the newest module syntax (import/export) without down-leveling. Vite handles browser compatibility.

"lib": ["ES2022", "DOM", "DOM.Iterable"]

Tells the compiler which built-in type definitions to include. ES2022 gives us all the JavaScript features. DOM gives us document, window, fetch, etc. DOM.Iterable adds iterator support on NodeList and friends so for (const el of document.querySelectorAll('div')) works.

⚡ Why this matters

This configuration is the single most important file in the project for catching bugs. We have just enabled five different strictness flags that, together, eliminate hundreds of potential runtime crashes — null reference errors, undefined array access, accidental type drift, untracked overrides. You will feel the difference within a day of coding under these settings.

Running the type checker

SvelteKit ships with a script called check that runs the TypeScript compiler over both .ts and .svelte files. We will run this constantly. After saving your tsconfig.json:

terminal 
$ pnpm check
svelte-check --tsconfig ./tsconfig.json
====================================
svelte-check found 0 errors and 0 warnings
====================================

That "0 errors, 0 warnings" line is the merge gate. If a junior engineer pushes a PR with type errors, CI rejects it.

devDependencies vs dependencies

📖 Terminology · dev vs prod dependencies

devDependencies are tools you only need while developing: the type checker, the linter, the bundler. dependencies are libraries that end up in your production bundle: the icon set, the ORM, the validator. The split matters because cloud hosts (Vercel, Railway) sometimes install only dependencies on production deploys to save time.

The ^ in front of versions

A caret means "compatible with this MINOR or PATCH version, but not the next MAJOR." So ^2.61.1 allows 2.61.2, 2.99.0, but not 3.0.0. This protects us from breaking changes while still getting bug fixes.

The pnpm-workspace.yaml file

One small companion file. Create pnpm-workspace.yaml at the project root:

yamlpnpm-workspace.yaml 
allowBuilds:  esbuild: trueminimumReleaseAgeExclude:  - '@sveltejs/kit@2.61.1'

Two things this does: (1) explicitly allows esbuild to run its install-time build script (pnpm blocks them by default for security), and (2) minimumReleaseAgeExclude exempts a specific package from a freshness check we'll enable globally later.

⚡ Why this matters

An install-time build script is code from a third-party package that runs on your machine just to install it. A malicious one could read your SSH keys. pnpm blocks all of them by default — you opt in per package. This is the modern, security-first defaults you'll see in every well-run repo.

Install the new dependency set so your pnpm-lock.yaml updates:

$ pnpm install

Vite & Svelte Config
Chapter 06

Vite & Svelte Config

Two config files own the bundler's behavior: vite.config.ts (how the bundler bundles) and svelte.config.js (how Svelte compiles components and which adapter we ship with).

vite.config.ts

Replace vite.config.ts with:

typescriptvite.config.ts 
import { sveltekit } from '@sveltejs/kit/vite';import { defineConfig } from 'vite';export default defineConfig({	plugins: [sveltekit()],	server: {		port: 5173,		strictPort: false	},	build: {		target: 'es2022',		cssMinify: 'lightningcss',		sourcemap: true	},	ssr: {		noExternal: ['gsap']	}});

Field-by-field

plugins: [sveltekit()]
Activates the SvelteKit plugin. This is what turns Vite into a "framework with routing" rather than just a bundler.
server.port: 5173
The dev server's port. 5173 is the SvelteKit default.
server.strictPort: false
If 5173 is busy, try 5174, then 5175, etc. With true, it would error instead — convenient when CI integration depends on a fixed port, but annoying in dev.
build.target: 'es2022'
Don't down-compile to ancient browsers. Every browser shipped after mid-2022 supports ES2022 features natively.
build.cssMinify: 'lightningcss'
Use the Rust-based LightningCSS minifier instead of the default. It's faster and produces smaller output. CSS that used to be 40KB minifies to 34KB.
build.sourcemap: true
Production gets sourcemaps too. When an error happens in production, the stack trace points to your real source files, not the compiled bundle.
ssr.noExternal: ['gsap']
This one matters. Vite by default externalizes dependencies during server-side rendering (uses Node's require instead of bundling them). GSAP doesn't work that way — it must be bundled even on the server side. This flag forces it.
⚠ Common pitfall

If you skip ssr.noExternal: ['gsap'], you'll get cryptic SyntaxError: Cannot use import statement outside a module errors the moment you try to use GSAP in a component. Hours of debugging avoided by one line of config.

svelte.config.js

Replace svelte.config.js with:

Field-by-field

Tells Svelte how to handle <script lang="ts"> and <style lang="postcss">. This single helper from the Vite plugin handles them all.

Why we don't force runes mode globally

Svelte 5 introduced runes, but it can still run old Svelte 4 components. There's a config option, compilerOptions.runes: true, that would force every component to use runes. We deliberately don't set it. Why?

Because node_modules/@tabler/icons-svelte still ships Svelte 4-style components internally. If we forced runes globally, those would error. Svelte 5 auto-detects runes mode per file based on whether you use $state, $props, etc. Every file we write will use runes; the icon library doesn't, and it doesn't have to.

SvelteKit's file-based mental model

SvelteKit doesn't make you configure a router. The router is the file system:

  • A folder under src/routes/ becomes a URL path.
  • A +page.svelte inside it becomes the page rendered at that path.
  • A +page.server.ts becomes server-only logic that runs before the page renders.

This file-based pattern is shared across modern web frameworks (Next.js, Nuxt, Astro). Once you internalize it, switching frameworks is mostly cosmetic.

Bootstrapping Every Request

Four files at the root of src/ are loaded for every page render: app.html, app.css, app.d.ts, and hooks.server.ts. They define the HTML envelope, the global stylesheet, the type contract for request data, and the server middleware. Get them right once.

Highlights worth understanding

data-theme="dark" on <html>
The theme is set on the root element. CSS can then write rules like [data-theme='dark'] .card { ... }. We set it inline so it's there before the first paint.
The inline <script> before %sveltekit.head%

data-sveltekit-preload-code="viewport"
SvelteKit preloads JavaScript bundles for any link that becomes visible in the viewport. Same idea, broader trigger.

Keyboard users press Tab on page load. The first focusable element is "Skip to content" — they hit Enter and jump past the navbar straight to the main content. WCAG-required.

The mockUser pattern — why we're not building auth yet

Building real authentication (password hashing, session tokens, CSRF protection, email verification) is several volumes of work. To unblock everything else, we attach a deterministic mock user. The dashboard and admin routes will read event.locals.user the same way they would with real auth — when we wire real auth in a future volume, the route code doesn't change.

The optional cookie tradeflex-mock-user=admin lets you demo the admin role without code changes: open DevTools, set the cookie, refresh.

Security headers — and the threats each prevents

X-Content-Type-Options: nosniff
Stops browsers from guessing the MIME type of a response. Without it, a file you served as text could be interpreted as JavaScript by an old Internet Explorer, enabling XSS.
Referrer-Policy: strict-origin-when-cross-origin
When users navigate from our site to another, we send only the origin (e.g. https://tradeflextrading.com) not the full URL. Protects user privacy — the destination site never sees the path they came from.
Permissions-Policy: camera=(), microphone=(), geolocation=(), interest-cohort=()
Explicitly disables browser APIs we never need. If a third-party script is ever compromised and tries to access the camera, the browser blocks it. The empty parens mean "no origins, including this one." interest-cohort=() opts out of Google's FLoC tracking.
X-Frame-Options: SAMEORIGIN
Prevents attackers from embedding our site in an <iframe> on their phishing site (a "clickjacking" attack). Only our own pages can frame our pages.
⚡ Why this matters

Four headers, fifteen lines of code, an entire class of web vulnerabilities mitigated. This is the kind of work that distinguishes a Principal Engineer's first commit from a junior's — security as a default, not a punchlist for "later."

The "Handle" type, briefly

The Handle type imported from @sveltejs/kit represents a function that takes { event, resolve } and returns a Response. The event contains everything about the request (URL, cookies, headers, locals); resolve(event) is what actually runs the route handler. Anything we do before calling resolve runs before the route. Anything after runs on the response. We use that to set headers after.

Chapter 10

Design Tokens

A design token is a named, semantic value: --gold-500 instead of #d4a24c. Tokens are the difference between a brand you can evolve in 30 seconds and a brand where rebranding takes three weeks. This is the single most important CSS file in the project.

The principle

Every component we ever write will reference tokens — never raw hex codes, never raw pixel values. When the brand designer says "make the gold a touch warmer," we change one line and the whole site shifts. When they say "ship a denser layout at 1920px," we change --container-max and every container reflows.

⚡ Why this matters

The single most common reason a small app becomes unmaintainable is that #d4a24c ends up hard-coded in 47 different files. By the time the brand evolves, replacing them is a half-day job with risk of breakage. Tokens make this a one-line edit.

The high-value bits

box-sizing: border-box on *
Padding and border are included in the declared width/height. Without this, "width: 100%; padding: 20px" overflows its parent by 40px. With it, everything just works.

Highlighted text is gold-on-dark. Tiny detail, but reinforces brand at every interaction.

CSS Architecture & Visual Engineering

Volumes 1–4 taught the language and structure of a SvelteKit + Svelte 5 app. This volume is the missing CSS layer — written from the perspective of having actually done the work, the failures included. Every chapter anchors to a specific commit in the project's git history. Every measurement comes from a real pnpm build of the repo, not generic textbook CSS.

The differentiator: this volume narrates the false starts. A cascade-ordering regression that broke production intermittently. Three failed attempts at single-source breakpoints before the working pipeline. A chart-label fix that solved a problem the user didn't have. An icon-library tree-shake that dropped client JS by 77% — once we figured out the wildcard-extension trap that wasn't in any doc. L7 isn't about not making mistakes; it's about catching them quickly, fixing them cleanly, and not pretending they didn't happen.

Fourteen chapters. Every src/lib/styles/*.css file. Every commit since affc7b5. Every measurement re-verified against the current build at write-time. If something in this volume disagrees with the code, the volume is wrong — open a PR.

Chapter 01

The cascade, specificity, and @layer

The CSS cascade is a sorting algorithm. Every browser runs it on every page, every time. For most projects it runs invisibly and developers never have to think about it. For this project it broke production intermittently — same source code, same build, same browser, different cascade. This chapter is the diagnosis, the cause, and the fix, with the citations that prove each step.

Anchor commits

4b61ad5 · css(infra): adopt @layer cascade architecture — the change that introduced the architecture (and the regression).

948dac6 · css(infra): inject @layer priority into EVERY CSS chunk (fixes intermittent layout breakage) — the fix.

The starting state · five files, no cascade strategy

Before either of those commits, the project had five global stylesheets imported in a flat list:

csssrc/app.css (before commit 4b61ad5)
/* TradeFlex Trading — global stylesheet entry */@import '$lib/styles/tokens.css';@import '$lib/styles/reset.css';@import '$lib/styles/typography.css';@import '$lib/styles/utilities.css';@import '$lib/styles/animations.css';

Six lines, no cascade architecture. Each file's rules win or lose by the CSS default rules: source order plus specificity. That's been the way for thirty years and it mostly works — but it has two specific failure modes that bite at scale:

  1. Specificity wars. A utility class like .container from utilities.css and a component-local rule like .card from a Svelte <style> block have the same specificity (0-1-0). Whichever loads later wins. Bundle order is non-deterministic in SvelteKit when route-specific chunks split out from the global stylesheet. Result: the same component renders differently on a cold load vs. a warm load.
  2. No escape hatch. You need a one-off override and your only tool is !important, which is a cliff — once you use it, the next override needs !important too, and you're in declaration-order roulette inside the !important bucket.

CSS Cascade Level 5 added @layer in 2022 specifically to solve both problems. Every modern evergreen browser supports it. The fix is to declare an explicit priority order once, then sort every rule into a named layer.

The architecture · reset → tokens → base → utilities → components → overrides

Six conceptual buckets, ordered from lowest to highest priority. Each global file assigns itself to a bucket. Component <style> blocks land in components. The bucket called overrides exists but stays empty — it's a named placeholder for the "I need this to win over everything" rule that hasn't been written yet. Anything outside any layer wins over everything inside one, so emergency rules go unlayered and the overrides name reserves the conceptual slot for documentation.

Why these names, in this order

The order reflects how rules want to apply. reset first — these are the universal element resets, they need to be overridden by everything else. tokens next — custom properties that everything reads. base — element-selector typography (h1, p) that componentry can override. utilities — .container, .stack, layout primitives. components — Svelte component-local rules, which must always beat utilities because the utility was generic and the component is specific to its context. overrides — escape hatch.

The intended rule, declared once in app.css:

csssrc/app.css (commit 4b61ad5 — the introduction)
@layer reset, tokens, base, utilities, components, overrides;@import '$lib/styles/tokens.css' layer(tokens);@import '$lib/styles/reset.css' layer(reset);@import '$lib/styles/typography.css' layer(base);@import '$lib/styles/utilities.css' layer(utilities);@import '$lib/styles/animations.css' layer(utilities);

The first line is the layer-priority statement. Per MDN's @layer reference:

Quoted from MDN

"The initial order in which layers are declared indicates which layer has precedence. As with declarations, the last layer to be listed will win if declarations are found in multiple layers."

"Having declared your layer names, thus setting their order, you can add CSS rules to the layer by re-declaring the name. The styles are then appended to the layer and the layer order will not be changed."

Locally — running pnpm dev on a single machine — this worked perfectly. The cascade was correct. Builds passed. CSS shipped. The change went out as commit 4b61ad5 and seemed done.

The regression · "sometimes the layout is broken"

Two days after 4b61ad5 shipped, the user reported a layout regression: on some page loads the design looked right; on others it looked subtly broken — specifically, utility classes were beating component-local rules. The symptom was inconsistent across reloads. Hard refreshes sometimes fixed it; sometimes didn't. Same browser, same URL, same build.

"Sometimes" is the diagnostic tell. Deterministic bugs are easy. Intermittent bugs almost always mean load-order non-determinism. Let me show what investigation revealed.

Hard evidence #1 · the chunk emission

SvelteKit + Vite split CSS into per-route + per-component chunks. The global stylesheet (app.css) becomes one chunk; each route brings its own component chunks. The browser loads them by <link rel="stylesheet"> tags in the HTML head. Spinning up a preview server and checking the actual emitted HTML for the home page:

terminal — order links appear in the SSR HTML head
$ curl -s http://localhost:4180/ | grep -oE '<link[^>]*\.css[^>]*>' | head -8
<link href="./_app/immutable/assets/LogoWordmark.DCIm7jWP.css" rel="stylesheet">
<link href="./_app/immutable/assets/Button.BZP9N7IF.css" rel="stylesheet">
<link href="./_app/immutable/assets/Tooltip.Ch3G4_NG.css" rel="stylesheet">
<link href="./_app/immutable/assets/0.CZn9h2r8.css" rel="stylesheet">
<link href="./_app/immutable/assets/AddToCartButton.C33DtGjW.css" rel="stylesheet">
<link href="./_app/immutable/assets/Badge.BF_udbdt.css" rel="stylesheet">
<link href="./_app/immutable/assets/PricingCard.DpZIDqdA.css" rel="stylesheet">

The global stylesheet — chunk 0.CZn9h2r8.css — sits in fourth place. Three component stylesheets (LogoWordmark, Button, Tooltip) load before it. That's the smoking gun. By the time the browser parses chunk 0 and sees the layer-priority statement, it has already seen @layer components { ... } in three other files.

Hard evidence #2 · the spec quote that explains why this breaks

Re-read the second MDN quote above: "The styles are then appended to the layer and the layer order will not be changed."

That sentence is the bug. When Button.BZP9N7IF.css arrives first containing @layer components { .btn { ... } }, the browser sees a layer it has never heard of and creates it. The first-encounter order becomes the priority order. components is now layer #1 with the lowest priority. When chunk 0 arrives later declaring @layer reset, tokens, base, utilities, components, overrides;, the four new layer names get appended after components, giving them higher priority. Result: utilities beats components, the inverse of the design.

The cascade is now upside-down — utility classes start winning over component-local rules. Every component that uses both (most of them) renders subtly wrong.

Hard evidence #3 · why "sometimes" and not "always"

Browsers can reorder which stylesheet they finish parsing first. Cache state, HTTP/2 multiplexing, network jitter — all play a role. On a cold load with a fast network and a small Button chunk, Button.css might finish first. On a warm load where chunk 0 was previously cached, chunk 0 finishes first. The order in which the browser creates the layers is therefore non-deterministic across reloads. Same code, different cascade, different render.

Pitfall · declaring layer order in one file is not enough

If the layer-priority statement only lives in one CSS file, you're betting that file will always parse first. That bet is not safe in any code-split bundler — Vite, webpack, esbuild, Rolldown. The browser only honors the order it first encounters layers in. By the time your "canonical" declaration arrives, the cascade is already locked.

The fix · declare priority in every chunk

Once the cause was clear the fix wrote itself. Re-declaring the same layer order is a no-op per spec (the second MDN quote: "the layer order will not be changed"). So if every CSS chunk opens with the priority statement, the first chunk the browser parses always establishes the correct order — regardless of which chunk it is.

The implementation needed two changes. First, move the layer-priority statement out of app.css and into a tiny file imported before everything else, unlayered, so its rules are visible to every layer that follows:

csssrc/lib/styles/breakpoints.css (created in commit 948dac6)
/* TradeFlex Trading — cascade foundation. * Imported FIRST and UNLAYERED in src/app.css. *//* Cascade priority (lowest → highest). */@layer reset, tokens, base, utilities, components, overrides;/* (Breakpoint @custom-media declarations also live here — see Ch 5.) */

Second, the same priority statement needs to ship in component stylesheets too — because those are the chunks that often parse before the global one. Svelte component <style> blocks become their own CSS chunks at build time. To get the foundation block into every one of them, a small Svelte preprocessor injects it:

javascriptsvelte.config.js (excerpted)
const CSS_FOUNDATION = `@layer reset, tokens, base, utilities, components, overrides;/* @custom-media declarations also live here — see Ch 5. */`;const cssPrePreprocessor = {  name: 'tradeflex-css-pre',  style({ content }) {    return {      code: `${CSS_FOUNDATION}\n@layer components {\n${content}\n}\n`    };  }};

Every Svelte <style> block now starts with the priority statement and wraps its own rules in @layer components { ... }. The result is verifiable in the build output:

terminal — proving every chunk now declares the order
$ grep -l "@layer reset" .svelte-kit/output/client/_app/immutable/assets/*.css | wc -l
64
$ ls .svelte-kit/output/client/_app/immutable/assets/*.css | wc -l
64

64 of 64. Every chunk. The first one the browser parses — whichever one that turns out to be — establishes the correct cascade order. The bug cannot recur because no path through the bundle skips the declaration.

What about Svelte's :global()?

An astute reader will notice the preprocessor I just showed has a subtle wrinkle — Svelte's :global(...) selector syntax. Lightning CSS (which Vite uses to process the injected block) doesn't recognize :global as a valid pseudo-class and warns ~60 times per build. The full pre-processor handles this with a mask/unmask trick — but that's a Ch 11 topic. For Ch 1 the relevant fact is just that the foundation injection is solid; the implementation details around tooling compatibility are in their own chapter.

Specificity in 2026 · :where(), :is(), and the bare * reset

Beyond layers, two other CSS Selectors Level 4 features affect cascade in this project. Both are worth knowing because the project's reset uses one and deliberately avoids the other.

:is(selectorA, selectorB) matches if the element matches any selector in the list, and contributes the highest specificity of the listed selectors. So :is(.button, #submit) has the specificity of #submit (1-0-0), even when matching .button. Useful for shortening selector lists; dangerous if you mix high-specificity selectors into the list.

:where(selectorA, selectorB) matches the same elements as :is but contributes specificity zero. The selector still selects, but it can be overridden by literally any other rule. This is the killer feature for design-system foundations: you can target :where(button, input, select) with default styling that's trivially overridable.

The project's reset (src/lib/styles/reset.css) uses bare * selectors, not :where(*):

csssrc/lib/styles/reset.css (lines 3–11)
*,*::before,*::after {  box-sizing: border-box;}* {  margin: 0;  padding: 0;}

This is a deliberate choice. Bare * has specificity 0-0-0 (universal selector) but it's a real selector — it contributes one to the type-selector count if you ever combine it. Wrapping in :where(*) would make absolutely sure these rules can be overridden by anything else, even other 0-0-0 rules. The trade-off:

Bare * (the project's choice)
Simple, what every other reset on the internet uses, instantly recognizable. Specificity 0-0-0 in the reset layer. Because the layer is the lowest-priority one, the reset will always lose to anything else anyway — wrapping in :where() would be redundant insurance against an already-impossible situation.
:where(*) (the more defensive option)
Belt and braces — even if layers were somehow disabled or stripped, the reset rules still couldn't beat anything. Costs nothing at runtime. Reads slightly more abstract.

Since the layer architecture is the cascade guard, the bare * is the right call. The "redundant insurance" of :where() would imply a lack of trust in the layer order — and trusting the layer order is exactly what this chapter argues for.

The trade-off matrix · five files vs the alternatives

One last architectural question worth addressing explicitly: why five separate CSS files instead of one big file, Tailwind, CSS-in-JS, or BEM? The honest answer is that all four work; the choice depends on team shape and rate of change. Here's the matrix that justified five files for this project:

Five files won here because: (1) separation by concern, not by component (tokens never change at runtime, reset is set once, typography is content-driven, utilities are layout, animations are motion — five orthogonal domains); (2) Svelte gives you scoped <style> blocks for free, so per-component CSS-in-JS solves a problem the framework doesn't have; (3) Tailwind wins for teams that change designers monthly — this is a single designer-developer where the cognitive cost of utility-class strings outweighs the win; (4) BEM was a 2010s answer to a 2010s problem (no native scoping), and Svelte's scoped styles solve it natively.

What this chapter taught (and what the rest of the volume builds on)

The cascade is solvable. @layer from CSS Cascade Level 5 makes the cascade declarative instead of source-order roulette. The project's six-layer order — reset → tokens → base → utilities → components → overrides — is the spine that everything else hangs on.

But the layer architecture has a non-obvious failure mode: the priority statement must ship in every CSS chunk the browser might encounter first, not just in the canonical one. Code-splitting bundlers (Vite, Rolldown, webpack, esbuild) all make this guarantee hard to maintain without intervention. The project's fix is a Svelte preprocessor that injects the foundation into every component <style> block.

The rest of the volume builds on this. Chapter 5 explains the @custom-media declarations that share the same injection point. Chapter 6 adds @container queries to the components layer. Chapter 11 explains the build-pipeline trick that makes Lightning CSS tolerate Svelte's :global() syntax. Every one of those chapters depends on the cascade architecture being correct — which is why this chapter comes first.

Chapter 02

Length units · px, rem, em, %, ch, vw/vh, dvh/svh/lvh, cqi

CSS has more than a dozen length units. Most projects use three of them. The choice is not stylistic — it determines whether the design scales with a user's accessibility preferences, whether mobile browsers clip content under their address bars, and whether a component remains correct when it's reused in a sidebar half the width of its original context. This chapter is the decision tree, anchored to the exact units this project uses in src/lib/styles/tokens.css.

The five categories that matter

Every CSS length resolves to a number of CSS pixels at render time. What varies is what determines the multiplier. Five categories cover everything you'll use in 2026:

Absolute · px
A fixed CSS pixel. Doesn't scale with anything. Use for things that genuinely have a fixed physical meaning at any zoom level: hairlines, the corner radius of small components, media-query breakpoints.
Root-relative · rem
Resolves to a multiple of the <html> element's font size. Per MDN: "For font-relative units that are root-based (such as rem), the font size is relative to the size of the font used by the <html> (root) element." Critical for accessibility — if a user sets their browser's default font size to 20px instead of 16px, every rem in your design scales proportionally.
Font-relative · em, ch
em resolves relative to the current element's font size. ch resolves to the width of the digit "0" in the current font. Both compound across nesting — useful for tightly-coupled spacing (padding around a label that should scale with the label's own size) but dangerous as a default.
Parent-relative · %
Resolves to a fraction of the parent's matching dimension. width: 50% means half the parent's width; height: 50% means half the parent's height (and may behave unexpectedly if the parent has no explicit height).
Viewport-relative · vw, vh, dvh, svh, lvh
Resolves to a fraction of the viewport's dimension. The newer variants (dvh/svh/lvh) account for mobile browser chrome that expands and retracts during scroll.
Container-relative · cqw, cqh, cqi, cqb
Resolves to a fraction of the nearest containment context's dimension. Introduced with CSS Containment Level 3. Cover in detail in Chapter 6 (container queries); mentioned here so the unit family is complete.

The rem vs em trap (with the MDN example)

This is the most common length-unit mistake. The MDN documentation makes the trap explicit:

Quoted from MDN · font-size

"One important fact to keep in mind: em values compound."

"rem values were invented in order to sidestep the compounding problem. rem values are relative to the root html element, not the parent element. In other words, it lets you specify a font size in a relative fashion without being affected by the size of the parent, thereby eliminating compounding."

What "compounding" actually means: if you put a span with font-size: 1.6em inside another span with font-size: 1.6em, the inner span renders at 1.6 × 1.6 = 2.56× the root size — not 1.6×. Each level of nesting multiplies. In a Svelte component tree, where deeply-nested layouts are normal, em-based sizing produces invisible compounding bugs the moment a component is moved one layer deeper.

This is why every spacing token in the project's tokens.css uses rem, not em:

csssrc/lib/styles/tokens.css · spacing scale
/* === Spacing (4px base) === */--space-0: 0;--space-1: 0.25rem;--space-2: 0.5rem;--space-3: 0.75rem;--space-4: 1rem;--space-5: 1.25rem;--space-6: 1.5rem;--space-7: 1.75rem;--space-8: 2rem;/* ... up to --space-48: 12rem */

The "4px base" comment is a designer convention — each step is a multiple of 0.25rem, which at the default 16px root equals 4px increments. But the rem unit means: if the user sets a 20px root size, the entire spacing scale grows proportionally. --space-4 becomes 20px instead of 16px. The rhythm stays consistent; only the absolute size changes. Use px for the same scale and accessibility scaling silently breaks.

When em is right · context-relative spacing

The compound-scaling trap doesn't mean em is always wrong. There's one common case where em is actively the better choice: spacing that should scale with the element's own font size. Padding inside a button is the classic example:

cssexample · em for context-relative padding
/* This button scales its padding with its font size automatically. */.btn {  font-size: 1rem;  padding: 0.5em 1em;}.btn-lg {  font-size: 1.25rem; /* padding auto-scales to 0.625rem 1.25rem */}

If the padding were declared in rem instead, every button size would need to redeclare its padding. With em, the padding is bound to the font-size — change one, get the other for free. The same logic applies to line-height values when expressed as unitless multiples, to border-radius when you want the curve to scale with text size, and to icon sizing inside a button where the icon should grow with the surrounding text.

The project's Button.svelte doesn't use this pattern — it uses fixed --h: 44px heights with hardcoded sizes. That's a defensible choice for a design system where button sizes are explicitly enumerated (sm/md/lg/xl). The em pattern wins when sizes are open-ended.

When px is right · three specific cases

rem is the default for spacing, type, and most properties. There are three places px is the correct call:

  1. Hairlines. border: 1px solid var(--border-default). A 1px border at a 20px root size is still 1px — and that's what you want; the user setting a larger font doesn't want thicker borders. Using 0.0625rem here would scale the border with the user's font preference, producing visibly thicker outlines than the designer intended.
  2. Small border-radius values where the curve matters. The project's radius scale uses px:
    csssrc/lib/styles/tokens.css · radii
    /* === Radii === */--radius-xs: 4px;--radius-sm: 6px;--radius-md: 10px;--radius-lg: 16px;--radius-xl: 22px;--radius-2xl: 32px;--radius-full: 9999px;
    A 6px corner on a small badge would become 7.5px at a 20px root — a visible difference. The 4px radius would become 5px, the curve subtly tighter. For small components that designer-tested at specific sizes, px is the unit of truth.
  3. Media-query breakpoints. @media (min-width: 768px). Browsers compute media queries against actual CSS pixels, not against root-relative rem values, and the breakpoint should fire at the same viewport width regardless of the user's font preference. The project's breakpoint tokens are px:
    csssrc/lib/styles/tokens.css · breakpoints
    --bp-sm: 480px;--bp-md: 768px;--bp-lg: 1024px;--bp-xl: 1280px;--bp-2xl: 1536px;--bp-3xl: 1920px;--bp-4xl: 2560px;
Why this matters · the accessibility math

The default browser font size is 16px. About 15% of desktop users override it to 18px, 20px, or larger — sometimes for vision reasons, sometimes because they prefer it. If your spacing tokens are px, those users get the same tight padding regardless of how they configured their browser; if your spacing tokens are rem, those users get a roomier interface that matches their preference. Choosing rem over px for spacing is the single highest-leverage accessibility decision in a design system. It's not visible to most users — and that's the point.

The viewport-units evolution · vh was wrong on mobile

Before 2022, the canonical "make this section fill the screen" pattern was min-height: 100vh. It shipped in millions of sites and broke in subtle ways on mobile browsers. The reason: mobile browsers have URL bars and toolbars that appear and disappear during scroll. When the user starts scrolling, the URL bar collapses, and the viewport grows by 50–100px. CSS had to decide: does 100vh mean the smaller viewport (URL bar showing) or the larger one (URL bar hidden)?

The answer the spec settled on: vh = the large viewport. Which means using 100vh as a section height on mobile typically clips the bottom of the content behind the URL bar on first paint, then "uncovers" it as the user scrolls. Designers shipped workarounds for years (height: calc(100vh - 60px), JavaScript that recomputed on resize).

CSS Values Level 4 introduced three sibling units that disambiguate this explicitly. Quoting MDN directly:

Quoted from MDN · CSS length

svh (small viewport height): "When you want the smallest possible viewport in response to browser interfaces expanding dynamically… The small viewport size allows the content you design to fill the entire viewport when browser interfaces are expanded."

lvh (large viewport height): "When you want the largest possible viewport in response to browser interfaces retracting dynamically… vh is equivalent to lvh."

dvh (dynamic viewport height): "When you want the viewport to be automatically sized in response to browser interfaces dynamically expanding or retracting."

The project's reset uses dvh on the body:

csssrc/lib/styles/reset.css · body block
body {  -webkit-font-smoothing: antialiased;  -moz-osx-font-smoothing: grayscale;  text-rendering: optimizeLegibility;  line-height: var(--leading-normal);  font-family: var(--font-body);  font-size: var(--text-base);  color: var(--ink-100);  background: var(--surface-0);  overflow-x: hidden;  min-height: 100dvh;}

100dvh means: the body's minimum height tracks the visible viewport at all times. When the user scrolls and the URL bar collapses, the body grows. When the URL bar re-appears, the body shrinks. The dashboard sidebar's height: 100dvh works the same way — the sidebar stays glued to the visible viewport edges regardless of scroll state.

Pitfall · the dvh repaint cost

MDN's own docs flag the trade-off: "using viewport-percentage units based on the dynamic viewport size can cause the content to resize while a user is scrolling a page. This can lead to degradation of the user interface and cause a performance hit." If you have an element that uses dvh AND that element triggers expensive layout (large grids, complex Flexbox, transforms), the page can jank during scroll-induced URL-bar collapse. The fix is usually to switch that specific element to svh (commits to the small size; no resize during scroll) or lvh (commits to the large size). Use dvh where the visible-area-tracking is genuinely needed (full-screen overlays, sticky chrome); use svh/lvh where the size only needs to be correct at one of the two states.

The ch unit · readable paragraph widths

One unit gets almost no attention but is the right answer for a specific common problem: setting a maximum reading width for text blocks. The ch unit equals the width of the digit "0" in the current font. For most proportional fonts, a column of 65ch works out to roughly 65 characters per line — which research has held up for decades as the comfortable reading length for English prose.

cssexample · ch for readable paragraphs
article p {  max-width: 65ch; /* ~65 characters per line, font-aware */}

Compare with the alternatives: max-width: 600px is font-blind (a wide font like Georgia hits 50 chars at 600px; a narrow font like Inter hits 80); max-width: 38rem is root-relative but still font-blind. 65ch reads correctly in every font you might swap in. The project doesn't use this everywhere — it could; long-form prose in the legal pages and about page would benefit.

Container-relative units · cqi for component-aware sizing

cqi (container query inline size) is to cqw what vw is to viewport width — except the dimension comes from the nearest containment context, not the viewport. cqi respects the writing direction (inline = horizontal in English, vertical in Japanese); cqw is purely physical.

The use case: typography that scales with its container, not the viewport. A heading inside a card should be larger when the card is wide and smaller when the card is narrow, regardless of how big the viewport happens to be. Mechanics covered in Chapter 6; the unit family is listed here for completeness.

The decision tree · which unit, when

If you remember nothing else from this chapter:

SPACING (margin, padding, gap) └── rem (scales with user accessibility preference) TYPOGRAPHY (font-size) └── rem (same reason, plus see Ch 3 for fluid type with clamp) ELEMENT HEIGHTS that should match the visible mobile viewport └── dvh (or svh/lvh if dvh causes scroll jank) ELEMENT HEIGHTS that match a parent └── % (with a parent that has an explicit height) CONTEXT-RELATIVE PADDING that should scale with element's own font-size └── em (single-level; never nest) HAIRLINES, small radii, breakpoints └── px (fixed, accessibility-independent) READABLE PARAGRAPH WIDTHS └── ch (font-aware character count) CONTAINER-RELATIVE SIZING └── cqi (see Ch 6)

The whole point of design tokens is that you make this decision once for each kind of value, then every consumer reads from the token. The next chapter shows how clamp(), min(), and max() add a layer on top of these units — making values respond fluidly across viewport sizes without media-query stairstepping.

Chapter 03

The fluid trinity · clamp(), min(), max() · with the math

Before clamp(), "responsive typography" meant a stair-step of media-query overrides — type at 16px below 768px, 18px between 768 and 1280, 20px above. Six values per breakpoint, hand-tuned, often forgotten in spacing tokens. clamp() collapses that into one declaration that smoothly interpolates across the entire viewport range. The project uses it 40 distinct times. Until you can derive the middle argument from a target, you're cargo-culting it.

The three arguments, what each does

Per the MDN reference:

Quoted from MDN · clamp()

"The clamp(min, val, max) function accepts three comma-separated expressions as its parameters."

min: "the lower bound in the range of allowed values. If the preferred value is less than this value, the minimum value will be used."

val (preferred): "the expression whose value will be used as long as the result is between the minimum and maximum values."

max: "the largest (most positive) expression value to which the value of the property will be assigned if the preferred value is greater than this upper bound."

And the formal equivalence: "clamp(MIN, VAL, MAX) is resolved as max(MIN, min(VAL, MAX))."

That last quote is the mental model. The preferred value is what you'd like; the min and max are guardrails. The browser evaluates the preferred at the current viewport, then clamps it to whichever guardrail it would have exceeded.

The middle argument is the only one that does work

The min and max bounds are static — they don't change as the viewport resizes. The middle argument is the only one that's fluid, typically as a function of viewport width (vw) or a sum like 0.95rem + 0.2vw. The shape of every clamp() in this project follows the same template:

clamp( FLOOR , PREFERRED(vw) , CEILING ) │ │ │ │ │ └── biggest the value ever gets │ └── the formula the browser interpolates with └── smallest the value ever gets

The preferred argument is where the engineering happens. Get it right and the value scales smoothly between floor and ceiling. Get it wrong and the value either flat-lines (always at one of the bounds) or flips through the bounds chaotically at small viewport changes.

Deriving the preferred argument · the algebra

The project's type scale is the cleanest example to dissect. Quoting tokens.css directly:

csssrc/lib/styles/tokens.css · fluid type scale
--text-2xs: clamp(0.625rem, 0.6rem + 0.1vw, 0.6875rem);--text-xs:  clamp(0.75rem,  0.72rem + 0.12vw, 0.8125rem);--text-sm:  clamp(0.875rem, 0.84rem + 0.15vw, 0.9375rem);--text-base:clamp(1rem,     0.95rem + 0.2vw,   1.0625rem);--text-md:  clamp(1.0625rem,1rem + 0.25vw,    1.125rem);--text-lg:  clamp(1.125rem, 1.05rem + 0.3vw,  1.25rem);--text-xl:  clamp(1.25rem,  1.15rem + 0.4vw,  1.5rem);--text-2xl: clamp(1.5rem,   1.35rem + 0.6vw,  1.875rem);--text-3xl: clamp(1.875rem, 1.6rem + 1vw,     2.5rem);--text-4xl: clamp(2.25rem,  1.85rem + 1.6vw,  3.25rem);

Take --text-base and walk through what it produces at three viewports (assuming the default 16px root, so 1rem = 16px):

Reading this table: the base font hits the floor (16px) on phones, smoothly grows across tablet and small laptop, and saturates at 17px on anything larger. The growth between the floor and ceiling is exactly 1px over a viewport range of ~512px. That's a 1px increase per 500px of viewport growth — barely noticeable to the user, which is the point. Body text should feel stable, not visibly resize as you scroll or resize the window.

Why the vw coefficient is small (and why it has to be)

The naive "fluid type" tutorial on the internet writes things like clamp(1rem, 5vw, 2rem) — middle term entirely viewport-based. Walk through what that produces:

ViewportPreferred = 5vwClamped to
320px16px16px
400px20px20px

Body text balloons from 16px to 32px over a viewport change of just 320px. That's a 100% font-size growth — visually it's like the page is breathing. The ceiling kicks in at 640px and the text stays glued there for every viewport above. A 1920px monitor renders body text the same size as a 640px viewport. That's not "responsive"; it's flailing.

The L7 derivation: the middle term should barely exit the floor and ceiling at the breakpoints you care about, producing a slow, almost imperceptible scaling between them. The two-component formula floor_value + small_vw achieves this — the constant term anchors the starting size, the small vw coefficient adds gradual growth as the viewport widens.

The exact algebra to derive your own

If you want the value to be exactly Vmin at viewport Wmin and exactly Vmax at viewport Wmax:

slope_per_px = (V_max − V_min) / (W_max − W_min) intercept = V_min − (slope_per_px × W_min) preferred = intercept + slope_per_px × 100vw Example: --text-base V_min = 16px at W_min = 320px (smallest reasonable phone) V_max = 17px at W_max = 1280px (where ceiling should saturate) slope_per_px = (17 − 16) / (1280 − 320) = 1 / 960 ≈ 0.00104 per px intercept = 16 − (0.00104 × 320) = 16 − 0.333 ≈ 15.67px preferred = 15.67px + 0.104vw = 0.98rem + 0.104vw (close to the project's 0.95rem + 0.2vw)

The project's actual values aren't a strict mathematical derivation — they're hand-tuned for visual rhythm. But the shape (small constant in rem plus small vw multiplier) is the pattern. If you understand the algebra, you can reverse-engineer any third-party design system's fluid scale and adjust it to your own anchor points.

Why this matters · the type-scale relationship

Each step in the project's type scale uses a slightly larger vw coefficient: 0.1vw for the smallest, 1.6vw for the largest. That's deliberate — bigger type benefits from more viewport-relative growth, because a hero headline at 36px on a laptop should feel proportionally larger at 52px on a desktop, while body text at 16px should stay at 17px. The ratio of growth (~50% increase in scale steps as you go up) is what gives a fluid type scale its visual harmony. Cut and paste a scale from a tutorial and you get visually-broken hierarchy at the extremes.

min() and max() as standalone

clamp() is the famous one, but its building blocks have independent use. min(a, b) returns whichever of a and b is smaller; max(a, b) returns whichever is larger. Used standalone, they let you write rules that adapt without media queries.

min() for readable paragraphs that never overflow

cssexample · min() for max-width
article p {  max-width: min(100%, 65ch);}

Reads as: "this paragraph's width is whichever is smaller: 100% of its container, or 65 characters of text." In a wide container the paragraph caps at 65ch (readable); in a narrow container it fills 100% (no overflow). One declaration, behavior that adapts to context.

max() for the iOS safe-area + minimum gutter

cssexample · max() for safe-area-aware padding
.container {  padding-inline: max(1rem, env(safe-area-inset-left));}

Reads as: "padding is whichever is larger: 1rem of breathing room, or the iOS notch's safe-area inset." On a regular screen the padding is 1rem; on a notched iPhone in landscape, the padding expands to clear the notch automatically. No media query, no JS, no platform-specific branches.

The project's section padding · fluid spacing matters too

Fluid type without fluid spacing produces visually broken pages — your headings scale up but the gap between sections doesn't, and at 1920px the rhythm collapses. The project applies the same clamp() pattern to spacing:

csssrc/lib/styles/utilities.css · section padding scale
.section {  padding-block: clamp(4rem, 8vw, 8rem);  position: relative;}.section-sm {  padding-block: clamp(2.5rem, 5vw, 4.5rem);}.section-lg {  padding-block: clamp(5rem, 10vw, 10rem);}

Each section variant uses a pure vw-driven middle term (no constant added) because the section padding is intentionally more reactive to viewport width than typography. At 320px a section is 25.6px of vertical padding (clamped to the 64px floor); at 768px it's 61.4px (still clamped to the floor); at 1024px it's 81.9px (preferred wins); at 1600px it's 128px (clamped to the 128px ceiling). The middle term is doing real work between 1024 and 1600px — which is the desktop range where the design needs to breathe more as the canvas widens.

And the container's horizontal gutter follows the same pattern:

csssrc/lib/styles/tokens.css · container gutter
--container-gutter: clamp(1rem, 3vw, 2.5rem);

16px gutter on phones, scaling to 40px on desktop. Every .container on the site reads from this token. Change it once, every page's horizontal padding updates everywhere.

Browser support · this is 2026 CSS

clamp(), min(), and max() shipped in all major browsers in April 2020. dvh/svh/lvh (which combine well with clamp()) shipped in 2022. There's no progressive-enhancement story to worry about — every browser your users have actually runs this. The "what if the browser doesn't support it" callout you'll see in older tutorials no longer applies.

Pitfall · clamp() in negative-margin contexts

If you use clamp() with a negative middle argument (e.g., for negative margins or offsets), the semantics flip in non-obvious ways: min becomes the most negative, max becomes the least negative. The argument names assume positive values. Re-read the MDN math (clamp(MIN, VAL, MAX) = max(MIN, min(VAL, MAX))) and you'll see it still holds, but readability suffers. For negative-direction fluid values, prefer writing negative-margin: calc(0px - clamp(0.5rem, 2vw, 1rem)) — the clamp stays in its natural positive form, and the negation is explicit at the use site.

Summary · what to take away

  • clamp(min, preferred, max) is three arguments: floor, formula, ceiling. The middle one is the only one that does work; the outer two are guardrails.
  • The formula is typically constant_rem + small_vw. The constant anchors the value at the floor; the small vw adds gradual growth.
  • If the value is supposed to be exactly Vmin at viewport Wmin and Vmax at Wmax, the slope is (Vmax − Vmin) / (Wmax − Wmin).
  • Larger type tolerates a larger vw coefficient; body type should stay close to its floor.
  • min() alone is the canonical max-width: min(100%, 65ch) pattern.
  • max() alone is the canonical iOS-safe-area-aware padding pattern.
  • Fluid spacing matters as much as fluid type — without both, the page's rhythm breaks at the extremes.

Chapter 04

Color systems · oklch, color-mix, and why the project still uses hex

The color tokens in this project are 100% hex and rgba — no oklch(), no color-mix(). By 2026 standards that's a defensible regression, not an oversight. This chapter explains why the modern color functions are objectively better for ramp generation, why a one-designer project legitimately picks the older format anyway, and how color-mix() would eliminate half the semantic-color tokens if migration were on the table. The trade-off is real and worth understanding before you copy it.

The state of the project · everything in hex and rgba

csssrc/lib/styles/tokens.css · color tokens
/* === Color: surfaces (dark canvas) === */--surface-0: #0a0a0b;--surface-1: #111114;--surface-2: #17171c;--surface-3: #1f1f26;--surface-4: #2a2a33;--surface-elevated: #1c1c22;--surface-overlay: rgba(10, 10, 11, 0.72);/* === Color: gold/amber primary (9-step ramp) === */--gold-100: #faedcc;--gold-200: #f8e1a8;--gold-300: #f5d08a;--gold-400: #e8b660;--gold-500: #d4a24c;--gold-600: #b0832f;--gold-700: #876320;--gold-800: #5d4416;--gold-900: #34270d;

Nine gold steps, six surface steps, six ink steps, four semantic colors — about 30 hex/rgba tokens total. They were hand-picked in a design tool (probably Figma) and copy-pasted into the file. No automation, no ramp generation, no perceptual-uniformity guarantees. And visually, they work — the dark canvas reads correctly, the gold accents feel premium, the semantic colors are distinguishable.

So why is this a "defensible regression"? Because if you were writing this design system from scratch in 2026, you would use oklch() — and the result would be measurably better in three specific ways. Understanding those three ways is the point of this chapter, even if the project itself doesn't take that path.

What oklch() is, and why it changes the math

Per the MDN reference:

Quoted from MDN · oklch()

"oklch() is the cylindrical form of oklab(), using the same L axis, but with polar Chroma (C) and Hue (h) coordinates."

"The L in oklch() is the perceived lightness, which refers to the 'brightness' we visually perceive with our eyes. This is different from the L in hsl(), where it represents lightness as compared to other colors."

Three arguments: L (lightness, 0–1 or 0%–100%), C (chroma, ~0–0.4 in practice), H (hue, 0–360 degrees). The breakthrough is the L axis — when you write oklch(60% 0.18 75), the 60% means "this color appears 60% as bright as pure white to the human eye." When you write hsl(45 80% 60%), the 60% means "this is 60% of the way between max-dark and max-light in the HSL coordinate system," which has no relationship to perceptual brightness.

The consequence: in HSL or RGB, two colors with the same L (or same average RGB intensity) can look wildly different in brightness to the eye. Yellow and blue at the same HSL lightness — yellow appears much brighter. In oklch(), two colors at the same L value do appear equally bright. That's what "perceptually uniform" means.

What that buys you · principled ramp generation

Take the project's nine-step gold ramp. In hex, each step is a hand-picked value with no enforced relationship to its neighbors. There's no mathematical guarantee that --gold-300 is exactly twice as bright as --gold-600; you'd have to load each into a color analyzer to find out. If a designer ever needs a tenth step — say, --gold-50 for an ultra-light tint — there's no formula. They eyeball it.

In oklch(), you'd write the ramp like this:

csshypothetical · oklch ramp
/* Same hue (75deg, gold), same chroma (0.13), L varies linearly */--gold-100: oklch(95% 0.06 75);--gold-200: oklch(88% 0.10 75);--gold-300: oklch(81% 0.13 75);--gold-400: oklch(74% 0.15 75);--gold-500: oklch(67% 0.16 75);--gold-600: oklch(55% 0.15 75);--gold-700: oklch(43% 0.13 75);--gold-800: oklch(31% 0.10 75);--gold-900: oklch(19% 0.06 75);

Three things become true that weren't true before. First, the hue is locked to one value (75°) so every step is unambiguously the same family of gold. Second, the L axis steps down in roughly equal perceptual units (95 → 88 → 81 → ...) so adjacent steps look like they sit on a uniform gradient. Third, the chroma rises toward the middle and drops at the ends — which mirrors how humans perceive saturation: it's hard to make a near-white or near-black look "vivid," so chroma drops at the extremes.

The killer feature: need a new step? Compute it. A --gold-50 would be oklch(98% 0.03 75) — same hue, slightly higher L than 100, slightly lower chroma. No designer needed. The system extends itself.

So why is this project using hex?

Three reasons, all defensible for the project's specific context, none of which would justify hex over oklch for a multi-designer team:

Reason 1 · designer-tool handoff

Figma's color picker speaks hex and HSL natively. oklch() support arrived in Figma in 2024 but isn't the default — a designer dropping a swatch on the canvas gets hex back unless they actively reconfigure. For a project where one person designs and codes, switching to oklch means an extra translation step every time a color enters the design tool from the code (or vice versa). The translation can be automated (browsers' DevTools convert seamlessly), but it's friction. For a multi-designer team where designers never touch CSS, this argument disappears — give them an oklch-aware tool and they're fine.

Reason 2 · git-diff readability

#e8b660 is a six-character token. oklch(74% 0.15 75) is fifteen characters with three semantically-distinct parameters. In a code review, "the gold-400 changed from #e8b660 to #e6b25e" reads instantly. "oklch(74% 0.15 75) → oklch(72% 0.14 75)" requires a moment to spot which of three values shifted. Hex is the lingua franca of color in design-engineering discussions; oklch is more accurate but less universally legible.

Reason 3 · the rgba escape hatch was already needed

The project's transparency tokens use rgba(), not modern alpha syntax:

csssrc/lib/styles/tokens.css · borders & transparency
--border-subtle:  rgba(248, 246, 241, 0.06);--border-default: rgba(248, 246, 241, 0.1);--border-strong:  rgba(248, 246, 241, 0.18);--border-gold:    rgba(212, 162, 76, 0.32);

Three of those four are the same color (the project's --ink-100: #f8f6f1 as 248/246/241) at four different opacities. In modern CSS, that would be color-mix(in oklch, var(--ink-100) 6%, transparent) — but only if --ink-100 were already in oklch. Mixing hex with oklch produces well-defined results (the browser converts), but the cognitive load of "is this token oklch or hex?" is real. Either commit fully or stay fully — and the project chose to stay fully in hex/rgba.

What color-mix() would eliminate

Even without migrating to oklch, the project could use color-mix() to halve its semantic-color token count. Quoting MDN:

Quoted from MDN · color-mix()

"The <color-interpolation-method> specifies what interpolation method should be used to mix the colors. It consists of the in keyword and the color space the colors should be mixed in."

"This example demonstrates adding transparency to a color by mixing any color with transparent."

The pattern: color-mix(in oklch, var(--base) 12%, transparent). Mix any color with transparent at a given percentage and you get that color at that opacity — but with one critical advantage over rgba(): the source color can be a token reference. Compare:

The cost of this refactor would be one file (rewrite the 8 tokens to 4 + a helper) and a search-and-replace across the codebase (every var(--success-bg) use site becomes color-mix(in oklch, var(--success) 12%, transparent)). About 30 minutes of work for a permanent reduction in maintenance burden. The reason the project hasn't done it yet: the existing tokens work, and the migration is in the "nice to have" backlog, not the "shipping blocker" critical path.

The dark-canvas trick · why ink isn't pure white

One color decision in this project deserves a callout because it's a load-bearing detail most readers won't notice. The body text token:

csssrc/lib/styles/tokens.css · ink scale
--ink-100: #f8f6f1;   /* warm off-white, NOT #ffffff */--ink-200: #d8d3c6;--ink-300: #a39e8f;--ink-400: #74705f;--ink-500: #4f4b3f;--ink-600: #2f2c25;

And the canvas:

csssrc/lib/styles/tokens.css · surface scale (extract)
--surface-0: #0a0a0b;   /* near-black with a hint of blue */

Pure white on pure black (#fff on #000) is uncomfortable to read for long sessions. The white "glows" against the black, creating chromatic aberration on most monitors. Designers have known this for decades — the fix is to soften both ends of the contrast: warm the white slightly (the project uses #f8f6f1, a creamy off-white with red and green channels above blue), and pull the black slightly off pure (the project's #0a0a0b has blue marginally higher than red and green, a cool near-black). The contrast ratio stays high enough for accessibility (we'll check that next) but the eye relaxes.

The chromatic relationship matters too. The off-white leans warm (more red/green, less blue) and the near-black leans cool (more blue). Warm-on-cool is the opposite of "burning" — it's the calmest possible relationship between text and background. Reverse it (cool white on warm black) and the eye strains. This is one of those details that's invisible when done right and obvious when wrong.

WCAG contrast math · the constraint everyone forgets

Every color choice has to clear an accessibility floor. WCAG 2.2 defines two levels:

LevelBody text (14pt regular / 18pt bold)Large text (18pt regular / 14pt bold)UI components / graphics
AA (minimum)4.5:13:13:1
AAA (enhanced)7:14.5:1—

The project's body text — --ink-200: #d8d3c6 on --surface-0: #0a0a0b — has a contrast ratio of roughly 13.5:1, comfortably above even AAA. The eyebrow text — --gold-400: #e8b660 on the same surface — clocks 11.2:1, also AAA. The lowest contrast in the regular text stack is muted body copy — --ink-300: #a39e8f on --surface-0 — at roughly 7.5:1, still passing AAA. --ink-400 is for hints and fineprint only; at 4.8:1 it passes AA for body text but should never be the only carrier of meaning.

Why this matters · the audit you should do once

Every project should run its token pairs through a WCAG contrast calculator once when the design system is locked, and document the pass/fail for each pair. This costs about an hour. The output is a one-page table that every PR can be checked against. When a designer requests "make the muted text a bit lighter," you can immediately answer "that'll drop to 3.9:1 and fail AA." Without the table, those conversations devolve into "it looks fine to me," which is the worst conversation in design.

What about oklch()'s relative-color syntax?

The most powerful oklch feature isn't even mentioned above: relative colors. CSS Color Level 5 lets you derive a color from another color by modifying just one channel:

cssCSS Color Level 5 · relative oklch
/* Take --gold-500's hue and chroma, but force L to 30% → an exact darker variant */--gold-500-dark: oklch(from var(--gold-500) 30% c h);/* Take --gold-500 but desaturate it for a disabled state */--gold-500-muted: oklch(from var(--gold-500) l calc(c * 0.5) h);

The l, c, h identifiers reference the source color's channels — you can use them verbatim, replace them, or compute from them. This eliminates entire categories of token: hover/active/disabled variants don't need their own declarations; they're derived. The project doesn't use this yet. If you were starting fresh today, you would.

Summary · the migration story for a hypothetical Phase-2

  1. The hex tokens stay valid forever. They render correctly. No browser will ever stop supporting them. There's no urgency.
  2. The first realistic migration is color-mix() for the -bg tokens. 30 minutes, halves the semantic-color token count, eliminates a drift surface. High-value, low-risk.
  3. Full oklch() migration is a multi-hour rewrite that produces visually-identical output but unlocks principled ramp generation, relative-color derivation, and perceptual uniformity. Pay for itself the second a designer requests a new color step.
  4. If you're starting a new design system in 2026, start in oklch(). The arguments above ("hex is easier to read") become weaker every quarter as tooling catches up.

Chapter 05

Single-source breakpoints · three failed attempts, the working pipeline

CSS Cascade has had custom properties since 2017. CSS Media Queries Level 5 added @custom-media at-rules so those breakpoints could be referenced from @media queries. The two should obviously connect — declare your breakpoints once, reference them everywhere. They don't. Browsers shipped one half, draft-tracked the other, and the project's first attempt to bridge them produced three consecutive failed builds before the working pipeline emerged. This chapter is the diagnosis, the false starts, and the four-line solution that actually shipped.

Anchor commit

affc7b5 · css(infra): single-source breakpoints via @custom-media — the migration that landed after three failures.

The starting state · "tokens" that aren't tokens

Before affc7b5, the project's tokens.css had this block:

csssrc/lib/styles/tokens.css (BEFORE)
/* === Breakpoints (reference only — use @media directly) === */--bp-sm: 480px;--bp-md: 768px;--bp-lg: 1024px;--bp-xl: 1280px;--bp-2xl: 1536px;--bp-3xl: 1920px;--bp-4xl: 2560px;

The comment said the quiet part out loud: "reference only — use @media directly." Custom properties cannot be used inside @media queries. The CSS spec defines @media conditions as static expressions evaluated at parse time; var(--bp-md) resolves at cascade time, which is after parse. So @media (min-width: var(--bp-md)) is invalid CSS — silently ignored by every browser.

The "tokens" were documentation. Components couldn't use them. Every breakpoint reference across 42 source files was a hardcoded 768px, 1024px, etc. The "tokens" existed to remind developers what the sanctioned values were — but nothing enforced that. An audit found the drift:

Hardcoded value found in componentsSanctioned in tokens.css?Count
640pxNo — closest sanctioned is 480 or 76811 components
1440pxNo — closest sanctioned is 1280 or 15361 component

12 violations of a convention that had no enforcement mechanism. The fix had to do two things: (a) make the tokens actually referenceable from @media, and (b) snap the 12 outliers to the closest sanctioned values. The harder problem is (a).

The right answer · @custom-media, but with caveats

CSS Media Queries Level 5 (still a working draft as of 2026) defines @custom-media:

cssCSS Media Queries Level 5 syntax
@custom-media --bp-md (min-width: 768px);/* Now this works: */@media (--bp-md) {  .card { grid-template-columns: 1fr 1fr; }}

The browser substitutes the named declaration into the @media query at parse time. Exactly what we want. The catch: no browser implements @custom-media natively as of 2026. It has to be compiled away by a build tool before the browser sees the CSS. The project uses Lightning CSS via Vite, which supports the draft behind a feature flag. The flag exists. The path to actually enabling it across this SvelteKit + Svelte 5 + Vite + Lightning CSS stack — that's what took three attempts.

Attempt 1 · the obvious solution that doesn't compile

First attempt: declare the custom-media map directly in Vite's config, the way you'd configure any other Lightning CSS option.

typescriptvite.config.ts (attempt 1 — FAILED)
export default defineConfig({  css: {    transformer: 'lightningcss',    lightningcss: {      drafts: { customMedia: true },      customMedia: {  // ← does this work?        '--bp-md': '(min-width: 768px)',        /* etc */      }    }  }});

TypeScript complained immediately: customMedia is not a valid property on Vite's LightningCSSOptions type. Reading Lightning CSS's own type definitions on disk:

typescriptnode_modules/lightningcss/node/index.d.ts (extract)
export interface Drafts {  /** Whether to enable @custom-media rules. */  customMedia?: boolean}

The boolean drafts.customMedia enables Lightning CSS's parser to recognize @custom-media declarations found in CSS source files. There is no JavaScript-side map for declaring custom media programmatically. The TypeScript type wasn't wrong — the runtime really doesn't accept that field. Attempt 1 was a category error: I was looking for a feature that doesn't exist.

Attempt 2 · a custom Vite plugin that prepends declarations to every CSS file

If Lightning CSS only resolves @custom-media declarations that appear in the same CSS source, the workaround is to inject them at the top of every CSS file. A custom Vite plugin can hook the transform step:

typescriptvite.config.ts (attempt 2 — FAILED)
function customMediaPlugin(): Plugin {  return {    name: 'tradeflex:custom-media',    enforce: 'pre',    transform(code, id) {      if (!id.endsWith('.css') && !id.includes('?svelte&type=style')) return null;      return { code: CUSTOM_MEDIA + code, map: null };    }  };}

The plugin matched both real .css files and Svelte's per-component virtual style imports (the ?svelte&type=style&lang.css query string). It should have worked. The build failed instantly with the same error as before:

terminal — build error

  at src/lib/components/marketing/Hero.svelte (during preprocess)

The diagnosis took a while. The reason the plugin didn't fire: SvelteKit's vitePreprocess runs Lightning CSS inside Svelte's own preprocessor pipeline, which executes before Vite's plugin transform hooks see the file. By the time Vite would call my plugin on a .svelte file, Svelte had already invoked Lightning CSS on the <style> block, and Lightning CSS had already errored on the unknown --bp-md.

The lesson: vite-plugin-svelte processes .svelte files through its own LSP-aware pipeline that doesn't intersect with Vite's standard plugin hooks. Anything that needs to modify the CSS inside a <style> block must happen inside Svelte's preprocessor chain, not in a Vite plugin.

Svelte preprocessors can be chained. If vitePreprocess runs Lightning CSS, and I need the @custom-media declarations to be available during that Lightning CSS pass, then a custom preprocessor needs to inject them before vitePreprocess runs.

The first version got the chain order backwards:

Same error. Svelte's preprocess array runs in order — but the order is left-to-right. With vitePreprocess() first in the array, Lightning CSS still saw the raw source before my preprocessor injected anything.

Swap the order:

Adding the @custom-media declarations to the source had succeeded — but it had also moved Lightning CSS earlier in the pipeline than the Svelte compiler.

The working pipeline · three preprocessors, surgically ordered

The solution that landed is conceptually different from the three failures. Instead of moving Lightning CSS around, it brackets Lightning CSS with two custom preprocessors — one before, one after — that mask the problematic syntax in each direction.

javascriptsvelte.config.js · CSS_FOUNDATION (the shared injection)
const CSS_FOUNDATION = `@layer reset, tokens, base, utilities, components, overrides;@custom-media --bp-sm (min-width: 480px);@custom-media --bp-md (min-width: 768px);@custom-media --bp-lg (min-width: 1024px);@custom-media --bp-xl (min-width: 1280px);@custom-media --bp-2xl (min-width: 1536px);@custom-media --bp-3xl (min-width: 1920px);@custom-media --bp-4xl (min-width: 2560px);@custom-media --bp-sm-down (max-width: 479.98px);@custom-media --bp-md-down (max-width: 767.98px);@custom-media --bp-lg-down (max-width: 1023.98px);`;

Note the dual responsibility: the foundation block declares both the @layer priority order (per Chapter 1) and the breakpoint @custom-media declarations. Both are needed in every CSS chunk; both share the same injection mechanism. Bundling them is a maintenance win.

The two preprocessors that bracket Lightning CSS:

javascriptsvelte.config.js · the bracket pattern
/* Pre-pass: inject foundation + mask :global() */const cssPrePreprocessor = {  name: 'tradeflex-css-pre',  style({ content }) {    const masked = content.replace(/:global\b/g, ':--svelte-global');    return {      code: `${CSS_FOUNDATION}\n@layer components {\n${masked}\n}\n`    };  }};/* Post-pass: unmask :global() so Svelte's compiler sees it */const cssPostPreprocessor = {  name: 'tradeflex-css-post',  style({ content }) {    return { code: content.replace(/:--svelte-global\b/g, ':global') };  }};export default {  preprocess: [cssPrePreprocessor, vitePreprocess(), cssPostPreprocessor]};

The pre-pass does two things: prepend the foundation block (gives Lightning CSS what it needs) and rewrite :global( to :--svelte-global( (a custom pseudo-class Lightning CSS accepts silently). The post-pass undoes the masking before Svelte's compiler runs. Lightning CSS in the middle sees: known custom-media declarations, no unknown pseudo-class. Svelte's compiler at the end sees: clean CSS with :global() intact, ready to be scoped.

Why the mask works · CSS Selectors L4 forward-compatibility

CSS Selectors Level 4 specifies that any pseudo-class starting with :-- is reserved for "custom pseudo-classes" and parsers should accept them silently if not recognized. That's the design escape hatch for adding new pseudo-classes without breaking older parsers. Lightning CSS honors this — it sees :--svelte-global(svg), doesn't know what it means, and passes it through unchanged. Then the post-pass rewrites it back. Net effect: zero Lightning CSS warnings, zero output difference, zero Svelte compiler confusion.

Vite config · enable the draft, document the noise

typescriptvite.config.ts (current)

Without `transformer: 'lightningcss'` the   * minifier-only mode runs too late in the pipeline and   * `@media (--bp-md)` references never expand. */  transformer: 'lightningcss',  lightningcss: {    drafts: { customMedia: true }  }},

Three lines of actual config. The comment block is longer than the config — that's intentional. The choice between cssMinify: 'lightningcss' (Vite default; runs Lightning CSS only at the final minify stage) and transformer: 'lightningcss' (runs Lightning CSS during the preprocess pass) determines whether @custom-media resolves. The minifier-only mode runs after Vite has already finalized the CSS, so any unresolved @media (--bp-md) queries get passed through to the browser as literal text — which the browser then ignores.

The --bp-* custom properties · still in tokens.css for a reason

The original CSS-variable tokens didn't go away. They live alongside the @custom-media declarations:

csssrc/lib/styles/tokens.css (current)

So both forms coexist: the @custom-media declarations for CSS authors, the CSS variables for JavaScript readers.

The triple-source-of-truth (custom-media in breakpoints.css, custom-media in svelte.config.js's CSS_FOUNDATION, custom properties in tokens.css) is the one open weakness of this design. Three files have to stay in sync. The comment block at the top of breakpoints.css spells this out:

/* Keep three sources in sync: *   • this file (global stylesheets) *   • svelte.config.js (Svelte component blocks) *   • the --bp-* CSS variables in tokens.css (for JS-side reads) */

The honest "this could be better" footnote: a single source-of-truth file (e.g., breakpoints.json) consumed by both build configs and the CSS via a custom Vite plugin would eliminate the drift surface entirely. That refactor is on the backlog, not on the critical path.

What this chapter is really about

The technical content above — Lightning CSS draft flags, Svelte preprocessor chains, the :--svelte-global mask — is honest documentation of a working system. But the chapter is also a case study in how problem-solving actually goes when you're integrating bleeding-edge tooling.

The three failed attempts weren't wasted; each one taught the constraint that ruled it out:

  1. Attempt 1 (Vite config map) → "the feature you want doesn't exist." Lesson: always check the actual library's TypeScript types before assuming an option exists. The TypeScript error wasn't wrong; the feature really wasn't there.
  2. Attempt 2 (Vite plugin) → "vite-plugin-svelte has its own pipeline you can't intercept." The fix is masking, not reordering.

The Principal-L7 move isn't to predict the working solution on the first try — it's to not commit half-baked attempts while you're still in the failure cycle. The final pipeline shipped as one clean commit (affc7b5) after the failures had been narrowed down on a scratch branch. The git history shows a working migration, not the journey. That's a deliberate hygiene choice — the journey lives in this chapter, where it serves educational purpose, not in the commit log where it would just be noise.

Chapter 06

Container queries · component-relative responsiveness on 4 real components

The 30-year assumption that "responsive design = viewport-relative" turned out to be wrong. A ProductCard should respond to its container, not the user's window — because the same card might render in a wide marketing grid on one page and a narrow sidebar slot on another. @container queries (Containment Level 3, shipped in all major browsers in 2023) make that possible. This chapter walks the project's four real container-query uses, then narrates the self-reference bug that broke PricingCard for an entire turn before it was diagnosed.

Anchor commits

557eee3 · css(components): add @container queries to 4 reusable components — the introduction

65132cd · fix(home): featured PricingCard no longer overflows wrapper — the self-reference bug + fix

The mental model · containers are local viewports

A standard media query asks: "how wide is the user's browser?" A container query asks: "how wide is the box I'm currently sitting inside?". Two changes make this possible:

  1. An ancestor element declares container-type: inline-size (and optionally container-name: foo). That element becomes a "query container" — its dimensions can be measured.
  2. Descendant rules use @container foo (min-width: ...) to react to the named container's width.

The component author no longer has to assume anything about where the component will live. A card declares its own breakpoints; if it's placed in a 24rem-wide column on a 1920px monitor, it renders in single-column mode anyway. If it's placed in a 60rem-wide layout on a 768px tablet, it renders multi-column.

The four real components

1. ProductCard · highlights collapse based on card width

csssrc/lib/components/commerce/ProductCard.svelte (extract)
.card {  container-type: inline-size;  container-name: product-card;  /* ... rest of card styles ... */}/* Highlights become 2 columns once the card has room. The card stacks * to 1 column in a sidebar slot at the same viewport width. */@container product-card (min-width: 24rem) {  .highlights {    display: grid;    grid-template-columns: 1fr 1fr;    column-gap: var(--space-4);  }}

The card's highlights list (4-6 bullet points per product) becomes a 2-column grid when the card is ≥24rem (384px) wide. In the home page's 3-column featured grid, the card is roughly 450px → 2-column highlights, balanced layout. In a hypothetical dashboard sidebar slot at 280px, the card stays single-column. The viewport doesn't enter the calculation.

2. StatCard · sparkline hides on narrow cards

csssrc/lib/components/dashboard/StatCard.svelte (extract)
.stat-card {  container-type: inline-size;  container-name: stat-card;}/* Below ~15rem (240px) the sparkline can't fit alongside the value * without overlapping; hide it. The card still tells the same * story via the delta percent in the footer. */@container stat-card (max-width: 15rem) {  .spark {    display: none;  }}

The dashboard overview shows 4 StatCards in a row at desktop widths (~280px each) — sparklines visible, story complete. On a tablet where the row collapses to 2-column, each card is ~360px wide — still sparklines visible. If the dashboard ever puts a StatCard in a narrow KPI strip at the top of a detail page (say, 200px wide), the sparkline auto-hides and the card gracefully degrades. The component is correct in every context without any per-page code changes.

3. DataTable · two-stage toolbar collapse

csssrc/lib/components/admin/DataTable.svelte (extracts)
/* Stack the toolbar vertically when the table is narrow so the * search box keeps full width instead of squeezing to a sliver. */@container data-table (max-width: 32rem) {  .toolbar {    flex-direction: column;    align-items: stretch;  }  .search { max-width: none; }}/* In narrow tables the pager's count + buttons collide; stack * them so the buttons get their own row and remain tappable. */@container data-table (max-width: 26rem) {  .pager {    flex-direction: column;    gap: var(--space-3);    align-items: stretch;  }}

Two breakpoints, both container-relative. At 32rem (512px) the toolbar stacks; at 26rem (416px) the pager also stacks. The table works in any width — admin pages typically render it at full container width, but a narrow dashboard widget could embed the same table and the layout still degrades gracefully.

4. .compare-wrap · the dual-layout swap from Ch 7's section

The fourth use isn't on a reusable component — it's on a route-specific element. The subscription page's comparison table swaps between a wide-container <table> layout and a narrow-container stacked-cards layout. The breakpoint is on the wrapper, not the viewport:

csssrc/routes/subscription/+page.svelte (extract)
.compare-wrap {  container-type: inline-size;  container-name: compare;}@container compare (max-width: 760px) {  .compare { /* visually-hidden, SR-only */ }  .compare-cards { display: grid; }}

Below 760px container width, the <table> goes to the SR-only pattern and the stacked-card layout becomes visible. The table stays in the DOM for screen-reader access (one canonical semantic representation); the cards are aria-hidden when the table shows, and vice versa. Same data, two layouts, no JavaScript.

The self-reference bug · a Principal-L7 trap

The four components above all share a structural pattern: the container declaration is on an ancestor element, and the @container rules target descendants. ProductCard's .card declares the container; .highlights inside it is the descendant that responds. StatCard's .stat-card declares the container; .spark responds. DataTable's wrapper declares; .toolbar and .pager respond. .compare-wrap declares; .compare and .compare-cards respond.

This is not a stylistic choice. It's required by the CSS Containment Module Level 3 spec.

Quoted from CSS Containment Module Level 3 (W3C draft)

"For each element, the query container to be queried is selected from among the element's ancestor query containers..."

Read literally: when matching an @container rule against an element, the browser walks up the ancestor chain looking for query containers. The element itself is not in that chain. An element cannot query its own container's dimensions.

The bug · PricingCard's broken featured-lift

Before the recent fix, PricingCard had this pattern:

csssrc/lib/components/commerce/PricingCard.svelte (BROKEN)
.pricing-card {  container-type: inline-size;  container-name: pricing-card;}.is-featured {  transform: scale(1.02);  /* lift effect for the featured plan */}/* Disable the lift when the card is narrow (single-column stack) */@container pricing-card (max-width: 28rem) {  .is-featured {    transform: none;  /* ← supposed to fire below 448px */  }}

The .is-featured class is applied to the same element that declares container-type: pricing-card (they're both on the card's root <article>). The @container rule asks: "is my container ≤28rem wide?" Per the spec quoted above, the browser walks the element's ancestor query containers — and finds none, because the element itself is the only container in the chain. The rule never matches. .is-featured keeps its transform: scale(1.02) at every viewport.

Discovered via direct DOM measurement at 500px viewport:

The card was 457px wide — well under the 448px threshold (28rem × 16px) — but transform was still scale(1.02). The container query never fired. The scale(1.02) made the card visually overflow its .featured-plan-card wrapper by ~5px on each side. The user reported the symptom: "left, right, and bottom borders are missing". The card's borders were rendering outside the wrapper's bounds and getting clipped.

The fix · drop the @container, use a viewport @media instead

csssrc/lib/components/commerce/PricingCard.svelte (FIXED)

So a self-referential @container query never fires. */@media (--bp-lg-down) {  .is-featured {    transform: none;  }}

The viewport media query does fire. The lift is now correctly suppressed at viewport widths below the --bp-lg (1024px) threshold, where the featured-grid layout is single-column anyway. At wider viewports where the multi-column grid kicks in, the lift returns. The behavior the original code intended now actually happens.

Pitfall · the three ways out of the self-reference trap

If you need an element to react to its own dimensions, you have three options. (1) Move the rule to a viewport @media query — works if "what the element looks like" correlates with viewport width in your design (the project's fix). (2) Wrap the element in a parent that declares the container, and put the responsive rule on the element — adds a div, but keeps the local-responsiveness story. (3) Use container-type: size on a parent and check both axes — more complex, rarely needed. Of the three, option 1 is the simplest and the project's design accommodated it. Option 2 is the right choice when the element really does need to respond to its own size regardless of viewport.

Dead container declarations · the audit follow-up

One honest find from re-auditing the codebase while writing this chapter: two components still declare containers they don't use. PricingCard kept container-type: inline-size; container-name: pricing-card after the broken @container rule was deleted in 65132cd. RoadmapCard declares container-type: inline-size; container-name: roadmap-card and never had a @container rule at all.

Dead container declarations aren't free. The browser still performs containment on the element (forcing it into its own layout context, which affects how percentages and child styles resolve). For these two components the impact is negligible — small cards, no children that depend on percentage sizing — but the principle is wrong.

The cleanup: delete both. PricingCard doesn't need the container anymore. RoadmapCard never did. Two trivial diffs that the original commits should have included. This is the kind of thing that gets caught only when you re-read the codebase chapter by chapter — which is one reason writing a documentation volume after shipping the work is more valuable than writing it concurrently. The act of explaining forces a re-audit.

When media queries still win

Container queries are not a strict upgrade. Three categories where @media remains the right tool:

Page-level layout decisions
Whether the navbar shows hamburger or full-width links, whether the sidebar is fixed or hidden, whether the page uses 2-column or 3-column overall layout. These are decisions about the page itself, not about a reusable component, and viewport width is the right input.
Design-token scaling
The project's --container-max grows from 1280px to 1440px at --bp-2xl (1536px viewport). That's a viewport decision, not a component-relative one. Per the project's tokens.css: @media (min-width: 1536px) { :root { --container-max: 1440px } }.
User-preference media queries
prefers-reduced-motion, prefers-color-scheme, prefers-contrast — these aren't about layout at all. They're user-state signals. Container queries have no equivalent.

The cqi unit · type that scales with the card

The container-relative length units (cqi, cqb, cqw, cqh) are the natural complement to @container queries. cqi stands for "container query inline-size" — 1cqi = 1% of the nearest containment context's inline dimension. Use case:

cssexample · cqi for container-aware type
.stat-card {  container-type: inline-size;  container-name: stat-card;}.stat-card .value {  /* Grow the KPI number with the card, capped at sane bounds */  font-size: clamp(1.5rem, 12cqi, 3rem);}

The project doesn't use this yet. It could — the StatCard's KPI numbers could scale with the card's width, looking proportional in every column count. The reason it's not in the current codebase: the design used fixed type sizes via --text-3xl, which works fine and saved the implementation cost. The unit is in the toolkit for when the design system calls for container-relative typography.

Browser support · this is universal in 2026

Container queries shipped in Chrome 105, Safari 16, and Firefox 110 — all in 2022–2023. Container units (cqi et al.) shipped in the same releases. No major browser version released after early 2023 lacks support. Like the rest of Vol 5's modern-CSS features, there's no progressive-enhancement story to worry about; if a user is running a browser old enough to lack container queries, your authentication form's modern color-mix() won't work either.

Summary · the rule

  • Container queries let descendants react to their nearest ancestor's container size — not the viewport.
  • Declare container-type: inline-size on the wrapper; query @container name (max-width: ...) on descendants.
  • An element cannot query its own container. Per W3C spec, container resolution walks ancestor chains only. If you need self-responsive behavior, either move to a viewport media query or wrap the element in a parent that declares the container.
  • Dead container declarations cost containment overhead. If you remove the @container rule, remove the container-type declaration too. Two cleanups outstanding in this codebase.
  • Media queries still win for page-level layout, design-token scaling, and user-preference signals (prefers-reduced-motion, etc.).
  • cqi and friends are the natural pairing — container-relative lengths for container-relative layouts.

Chapter 07

Progressive enhancement · @supports and the backdrop-filter defense

In 2026, ~95% of users get backdrop-filter: blur(20px) rendering correctly. The other 5% — older Firefox, some Linux GPU stacks, certain enterprise-managed installations — see nothing. The navbar that was supposed to be a frosted-glass panel becomes a transparent strip with the page content readable through it; the modal's dimming backdrop becomes a clear pane. This isn't a bug Lighthouse catches. The fix is a four-line CSS pattern applied at every site, and the project does it 13 times.

Anchor commit

53bf081 · css(a11y/compat): @supports fallbacks for every backdrop-filter site — the audit that hardened 10 components + the .glass utility.

The class of bug that ships silently

Run the project's home page in a current Chrome on a desktop. The navbar is a beautifully translucent strip with a 20px blur of the page content scrolling underneath. Now run the same page in a browser without backdrop-filter support. Without an explicit fallback, the navbar's CSS declares:

cssnaive · what shipped before commit 53bf081
.navbar {  background: rgba(10, 10, 11, 0.4);  /* 40% opaque — supposed to be the blur's base */  backdrop-filter: blur(20px) saturate(140%);}

In a browser that supports backdrop-filter, the 40% opaque background combines with the blur to produce frosted glass. In a browser that doesn't, the backdrop-filter declaration is silently dropped and you're left with a 40% opaque background without blur — a translucent strip with sharp content scrolling readably underneath. The navbar's text disappears against whatever color happens to be beneath it. Forms with sticky chrome become unreadable. Modal backdrops become panes of clear glass.

The user doesn't see "an error." They see a broken-looking site and bounce. You don't see "an error" in your monitoring. The CSS console doesn't warn — undeclared support is silent dropping, by spec design.

What @supports is, exactly

Quoted from MDN · @supports

"The @supports CSS at-rule lets you specify CSS declarations that depend on a browser's support for CSS features. Using this at-rule is commonly called a feature query."

"@supports ( <supports-condition> ) { /* If the condition is true, use the CSS in this block. */ }"

"The or operator creates a new expression from the disjunction of two shorter expressions. It returns true if one or both of the shorter expressions is also true."

The semantics are simple: ask the browser "can you do this property: value combination?". If yes, apply the rules inside the block. If no, ignore the block entirely. The atomicity is the point — there's no partial application.

The canonical pattern · opaque default, upgrade on support

Every site in the project follows the same four-line structure. The .glass utility in utilities.css is the cleanest expression:

csssrc/lib/styles/utilities.css · .glass utility
.glass {  background: var(--surface-2);  border: 1px solid var(--border-default);}@supports (backdrop-filter: blur(1px)) or (-webkit-backdrop-filter: blur(1px)) {  .glass {    background: rgba(23, 23, 28, 0.6);    backdrop-filter: blur(20px) saturate(140%);    -webkit-backdrop-filter: blur(20px) saturate(140%);  }}

The default (outside @supports) is a fully opaque --surface-2 background plus a border. Looks like a solid card. Readable from byte one. Only when the browser confirms support does the rule upgrade: background becomes a 60% opaque rgba, and backdrop-filter adds the blur. The fallback users get a flat card; the modern-browser users get the glass effect. Both render correctly.

Why or with -webkit-backdrop-filter

The condition is verbose for a reason. Safari shipped backdrop-filter behind the -webkit- prefix for years and only dropped the requirement in Safari 18 (late 2024). Older iOS devices stuck on iOS 16 — common on hand-me-down hardware — still need the prefixed form to detect support correctly. The or operator covers both:

Browserbackdrop-filter-webkit-backdrop-filter@supports matches?
Chrome 130+✅ supported✅ aliasedYes
Safari 18+✅ supported✅ aliasedYes
Safari 16 (iOS 16)❌✅ supportedYes (via the OR branch)

Older Firefox / GPU-fallback Linux❌❌No → fallback wins

The or isn't decoration. Drop the second condition and Safari 16 silently falls into the fallback branch — losing the glass effect on the audience most likely to expect it (Apple users with old phones).

The fallback isn't always the same

The trick that took the most thinking in 53bf081 wasn't how to write the pattern — it was what to use as the fallback at each of the 10 sites. There are three categories:

Site type 1 · already-mostly-opaque surfaces

Why opaque-default beats translucent-default

An alternative pattern exists: declare the translucent background as the default, and put the fallback behind @supports not (backdrop-filter: ...). Looks symmetric on paper. Three reasons it's worse:

  1. Cascade clarity. The base rule (outside any feature query) is what readers reach for first. Putting the "safe" version there means: "every browser gets a readable surface by default." Putting the risky version there means: "every browser gets a possibly-broken surface unless feature-detection saves them." The first mental model is the L7 default; the second is the bug.
  2. @supports not has more compatibility caveats than @supports. A browser too old to support @supports at all (very rare in 2026 but possible for stale enterprise installs) treats unknown @-rules as failed — meaning neither branch applies, and the user gets no styling. The opaque-default pattern fails gracefully (you keep your default); the translucent-default pattern fails completely (no rules apply, no styling).
  3. Diff readability. Reviewing a PR that adds a new translucent surface, the diff with opaque-default puts the fallback in the base rule (visible at a glance) and the enhancement in the conditional block (clearly labeled as enhancement). The reverse pattern requires the reviewer to mentally parse the negation.

The principle generalizes · feature-detect every CSS feature that "fails silent"

backdrop-filter is the canonical example because the silent-fail is so visually devastating, but the same pattern applies to any CSS feature where the absence-of-support is functionally different from the support-present case. Three other features worth wrapping:

scrollbar-gutter: stable
Reserves space for the scrollbar even when not scrolling, eliminating layout-shift on scroll-trigger. Without support, the content reflows when a scrollbar appears. Fallback: explicit padding-right.
overscroll-behavior: contain
Prevents scroll chaining (the body scrolling when you reach the end of a modal's scroll). Without support, modals scroll the body. Fallback: JavaScript scroll-lock on body.
color-mix()
If you use it for a semantic background (e.g., color-mix(in oklch, var(--success) 12%, transparent)), unsupported browsers get the un-mixed token or nothing. Fallback: an explicit rgba() declaration in the base rule.

The check: if the feature is "polish on top of working baseline," it's safe without @supports. If the feature is load-bearing for legibility, readability, or layout, wrap it. backdrop-filter ended up in the "load-bearing" bucket because the design used translucency to such an extreme degree that the fallback wasn't just "less pretty" — it was unreadable.

Chapter 08

Logical properties · margin-inline-start over margin-left

CSS shipped logical properties in 2020. Most projects still write margin-left and padding-right. Both work in left-to-right English; only one of them works in Arabic, Hebrew, or any vertical writing mode. The migration costs nothing — same character count, same visual result in LTR — and earns the project automatic correctness in 30+ writing systems for free. This chapter is the 9-file audit that converted the codebase, and the one case (margin-left: auto for flex push-to-end) that requires a moment of thought.

Anchor commit

8db73b9 · css(rtl): replace 10 raw margin/padding-left/right with logical properties — 9 files touched, 10 declarations updated, identical LTR visual, mirrored automatically in RTL.

The mental model · inline axis vs block axis

Per MDN's logical properties reference:

Quoted from MDN · CSS logical properties

"Inline dimension: parallel to the flow of text within a line, i.e., the horizontal dimension in horizontal writing modes, and the vertical dimension in vertical writing modes. For standard English text, it is the horizontal dimension."

"Block dimension: perpendicular to the flow of text within a line, i.e., the vertical dimension in horizontal writing modes, and the horizontal dimension in vertical writing modes. For standard English text, it is the vertical dimension."

"The start of a line is not always the left side of a line. Different writing systems operate in various directions."

Logical properties are named relative to the flow of text, not the user's screen. margin-inline-start is the margin on whichever side the text starts — left in English, right in Arabic. margin-block-end is the margin on whichever side the next block of text comes — bottom in English, bottom in Arabic (the block axis works the same in both horizontal writing modes), but rightward in traditional vertical Japanese.

The mapping for standard horizontal-tb writing mode:

margin-rightmargin-inline-endmargin-inline-start
margin-topmargin-block-startmargin-block-start

The block-axis mappings (top/bottom) stay physical-ish because both LTR and RTL still flow lines downward. Logical only matters when you switch to a vertical writing mode (writing-mode: vertical-rl), which is rare. The big leverage from logical properties is the inline axis — the one that flips between LTR and RTL.

The flex push-to-end case · the one site that needed thought

The dashboard subscription banner had this:

csssrc/routes/dashboard/subscription/+page.svelte (diff)
  .banner-action {-   margin-left: auto;+   margin-inline-start: auto;    background: var(--warning);    /* ... */  }

The margin: auto trick on a flex item pushes it to one end of the flex container. margin-left: auto on a flex item in a row pushes it to the right edge — the item is "pushed away from the start." The logical equivalent is margin-inline-start: auto — same intent: "push away from the start."

This is the spot where it's easy to grab the wrong logical property by reflex. "It was on the left, so it must be inline-start" — but the flex semantics are about the side the margin is on, not the direction it pushes. margin-left: auto means "I have a giant left margin that consumes all the space" — the item ends up on the right. margin-inline-start: auto preserves that exactly: the item still ends up on the inline-end side, which is the right in LTR and the left in RTL. The flex layout mirrors correctly.

Pitfall · don't translate by visual position

If you find yourself thinking "the item was on the right, so I need margin-inline-end", stop. The auto-margin is on the side away from where the item ends up. Translate the original property name (left → inline-start, right → inline-end) directly, not the visual outcome. The flex semantics carry over identically — that's the whole point of logical properties.

When physical still wins (the rare case)

Three contexts where physical properties remain correct:

Visual-effect properties that don't care about writing direction
Shadow offsets — box-shadow: 4px 8px 16px rgba(...). The X/Y here are about lighting direction (where the light is coming from), not about reading direction. Even in RTL, the design might want light from the upper-left. Logical isn't an upgrade here.
Transform translations with explicit visual intent
transform: translateX(-50%) for centering. This is purely about pixel-shifting the element; it has nothing to do with text flow. transform: translate(start, ...) doesn't exist; physical X/Y is the only option.
Truly viewport-relative positions
An absolutely-positioned tooltip pinned to top: 50%; left: 50% as a centering hack. top and left here are about screen coordinates. inset-block-start would still work, but the physical names read clearly for this specific intent and the design isn't being mirrored.

The discipline: directional spacing and borders always go logical; visual effects that aren't text-flow-relative can stay physical. The 10 sites in the project's commit were all in the first bucket.

What the migration actually buys you

The honest answer: nothing visible until the day you ship an Arabic locale. The project doesn't currently render in RTL. The migration didn't change a single pixel for any user today.

What it bought instead is option value. If the project ever adds an Arabic translation (the trading desk has Middle Eastern members) or a Hebrew one, the codebase mirrors automatically. There are no per-component RTL stylesheets to maintain, no html[dir="rtl"] .foo { margin-right: 4px } override blocks. The toolchain doesn't even need an RTL processor (the canonical PostCSS RTL plugins exist precisely because most codebases haven't migrated to logical properties — they're a back-fix for projects too far gone to update individually).

The cost was 10 lines of diff in 9 files. Lowest-leverage migration in the audit by raw line count; highest-leverage in terms of future flexibility per character of CSS changed.

Why this matters · the convention that pays once it lands

Every new component added to the codebase after 8db73b9 is now written in logical properties by reflex. The team's CSS muscle memory shifted. Two months from now, no one will remember the migration — they'll just write margin-inline-end automatically. That's the real value: the migration was a one-time cost to change a habit, and the habit pays compound interest in every subsequent component. The Arabic locale never has to happen for the value to be realized; the value is the consistency.

Chapter 09

Motion · easing curves, prefers-reduced-motion, GSAP vs CSS

Motion is the thinnest layer of polish that separates a competent interface from a memorable one — and the thinnest line between memorable and nauseating. This chapter is the project's motion stack: three hand-picked cubic-bezier easing tokens (trimmed from seven after an audit), four duration tiers, eight CSS keyframes, and a GSAP layer that honors prefers-reduced-motion at three independent enforcement points. None of these decisions are stylistic; each has a math reason and an accessibility reason that show up in the code.

The easing curves · what cubic-bezier numbers actually mean

Every motion in the project ultimately resolves through one of three easing tokens. They're declared once in tokens.css:

csssrc/lib/styles/tokens.css · easing
--ease-out:    cubic-bezier(0.16, 1, 0.3, 1);--ease-in-out: cubic-bezier(0.65, 0, 0.35, 1);--ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1);

A cubic-bezier(x1, y1, x2, y2) defines an animation's progress over time as a Bézier curve. The four numbers are two control points (x1,y1) and (x2,y2) on the curve from (0,0) to (1,1). The X axis is time (0% → 100%); the Y axis is animation progress (0% → 100%). The control points pull the curve toward themselves, shaping how fast the animation accelerates and decelerates.

What each token feels like:

--ease-out: cubic-bezier(0.16, 1, 0.3, 1)
Decelerates fast, then settles slowly. The first control point pulls Y immediately to 1 (the animation reaches near-final value almost instantly), while the second control point keeps Y at 1 (the tail glides into place). Feels confident — like the element knew where it was going. The project's default for entrance animations (80 uses across the codebase, the most-consumed easing token by far).
--ease-in-out: cubic-bezier(0.65, 0, 0.35, 1)
Slow start, fast middle, slow end. The classic "natural" S-curve. Used for transitions where the element exists before and after (a panel sliding, a tab indicator moving). Reads as "this thing is moving with deliberate purpose." Single use in the codebase — the gold-pulse infinite animation.
--ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1)
Overshoots past the target, then settles back. The second value of the first control point is 1.56 — above 1, which means the Y coordinate goes past the final animation value during the curve, then returns. Used for the "scale-in" animation where a card or modal appears: it scales up to slightly larger than 100%, then snaps back to 100%. Reads as "that thing landed" — the bounce communicates impact.
Why this matters · easing carries meaning, not just smoothness

Pick the wrong easing and the same animation reads completely differently. A modal fading in with linear feels mechanical. The same modal with --ease-out feels confident. With --ease-spring it feels playful. Three tokens cover most of the vocabulary the project actually needs; the CSS keyword linear handles infinite loops (spin, shimmer, marquee — where any easing would create perceived stops at the cycle boundaries).

Audit footnote · why only three easing tokens

An earlier version of this file declared seven tokens (added --ease-linear, --ease-in, --ease-out-expo, --ease-in-out-quart as "design-system breadth"). A token audit found four were never consumed — no var(--ease-in), no raw cubic-bezier duplicates, no comment references. --ease-linear was a duplicate of the CSS keyword linear, which is what every infinite-loop animation in the codebase used directly. The four unused tokens were deleted in the same audit that produced this volume — the design-system-breadth justification didn't survive the "is anyone actually consuming this?" question. If you ever do need a spring-out or a sharp-quart curve, derive the value from your anchor points (Ch 3's slope formula applies to easing too) and add it back here first.

The duration scale · four tiers, perceptually-spaced

csssrc/lib/styles/tokens.css · durations
--dur-fast:   160ms;--dur-base:   240ms;--dur-slow:   420ms;--dur-slower: 680ms;

The numbers map to perceptual categories:

  • 160ms (fast) — short enough to feel responsive, long enough for the eye to track. Button hover lifts, focus ring expansions, color transitions on links. Used 19 times.
  • 240ms (base) — the project's default for state transitions. Cart drawer opens, modal appears, cards lift on hover. Long enough to read as "this thing moved here on purpose." Used 61 times — the workhorse.
  • 420ms (slow) — entrance animations for content panels. Hero fade-ups, section reveals on scroll. Feels deliberate, almost editorial.
  • 680ms (slower) — used for longer sequences where the eye should follow the motion fully. Marquees, complex stagger reveals.

The discipline: if you find yourself reaching for a duration not in this list, you're either tweaking past what the user can perceive or you've identified a missing tier that should be added to the scale. Hand-picking 237ms for one animation is a code smell.

Audit footnote · the deleted duration tiers

An earlier version of this scale included --dur-instant: 80ms and --dur-cinematic: 1100ms. The same audit that trimmed the easing tokens removed these two — neither was ever consumed. The "80ms is the threshold of conscious perception" framing was theoretically true but didn't survive the "is anyone actually using it?" question. The "cinematic" tier was reserved for the headline split-reveal, which turned out to hardcode its own duration via GSAP defaults. If a use case for either tier emerges, add it back — the four-tier scale isn't a hard limit, it's the current honest representation of what the codebase consumes.

The CSS keyframes · eight reusable patterns + animation aliases

Project-wide motion patterns live in animations.css — eight @keyframes + eight utility classes that compose them with the duration/easing tokens. Three worth examining:

csssrc/lib/styles/animations.css · fade-up + .animate-fade-up
@keyframes fade-up {  from {    opacity: 0;    transform: translate3d(0, 16px, 0);  }  to {    opacity: 1;    transform: translate3d(0, 0, 0);  }}.animate-fade-up {  animation: fade-up var(--dur-slow) var(--ease-out) both;}

The element fades in (opacity 0 → 1) while rising 16px from below. The animation shorthand composes the keyframe with --dur-slow (420ms) and --ease-out (the confident decelerating curve). The both keyword applies the keyframe's from state before the animation starts and its to state after it ends — preventing the flash of unstyled content where the element would briefly appear at its post-animation state before animating.

The translate3d is the load-bearing detail. translateY(16px) would work identically visually — but translate3d is the canonical hint to the browser to promote the element to its own compositor layer, which means the animation runs entirely on the GPU. Without that hint, the browser may run the animation on the main thread, contending with whatever else the page is doing. The 3d trick costs nothing (the Z translation is 0) and earns reliable 60fps.

The spring landing

csssrc/lib/styles/animations.css · scale-in + .animate-scale-in
@keyframes scale-in {  from { opacity: 0; transform: scale(0.96); }  to   { opacity: 1; transform: scale(1); }}.animate-scale-in {  animation: scale-in var(--dur-base) var(--ease-spring) both;}

The keyframe goes from scale(0.96) to scale(1) — only 4% growth. Combined with --ease-spring's overshoot, the element actually scales past 1.0 (briefly to ~1.05 around the 75% mark) before settling back to 1.0. The 4% starting offset is small enough not to draw attention; the overshoot adds the "landed" feel without being cartoonish. Used for modals appearing and cards popping into view.

Shimmer · skeleton-loader pattern

csssrc/lib/styles/animations.css · shimmer (loading state)
@keyframes shimmer {  0%   { background-position: -200% 0; }  100% { background-position: 200% 0; }}.animate-shimmer {  background: linear-gradient(    90deg,    var(--surface-2) 0%,    var(--surface-3) 50%,    var(--surface-2) 100%  );  background-size: 200% 100%;  animation: shimmer 1.6s linear infinite;}

A wider-than-element gradient slides across, producing the canonical "pulse of light" skeleton-loader effect. The gradient is 200% wide and travels from -200% to 200% — a full 400% pan — so the light sweep starts off-screen, crosses the visible area once per cycle, and exits before looping. linear easing on infinite loops only; ease curves would create perceived stops at the edges.

The reduced-motion blanket guard

Per MDN:

Quoted from MDN · prefers-reduced-motion

"The prefers-reduced-motion CSS media feature is used to detect if a user has enabled a setting on their device to minimize the amount of non-essential motion."

"Such animations can trigger discomfort for those with vestibular motion disorders. Animations such as scaling or panning large objects can be vestibular motion triggers."

The intent: "removes, reduces, or replaces motion-based animations" — animations should be toned down to less aggressive alternatives, not necessarily removed entirely.

The project's response is the universal blanket guard, declared once in reset.css:

csssrc/lib/styles/reset.css · reduced-motion guard
@media (prefers-reduced-motion: reduce) {  html { scroll-behavior: auto; }  *, *::before, *::after {    animation-duration: 0.01ms !important;    animation-iteration-count: 1 !important;    transition-duration: 0.01ms !important;    scroll-behavior: auto !important;  }}

The universal selector + !important overrides every component-local animation. The user with reduced-motion preferences set gets:

  • Smooth-scrolling disabled — anchor links jump instantly instead of animating.
  • Every animation runs for 0.01ms once — essentially instant, but technically still firing.
  • Every transition runs for 0.01ms — same.
Pitfall · why 0.01ms instead of 0

Setting animation-duration: 0 seems cleaner but breaks animationend event firing on some platforms — older Safari and certain Firefox versions don't dispatch the event for zero-duration animations. Code that relies on animationend to clean up state (focus restoration after modal close, removing a "just-animated" class) silently stops working. 0.01ms guarantees the event fires while being below any perceptible threshold. The non-zero value isn't paranoia; it's compatibility.

The GSAP layer · three enforcement points for reduced-motion

The CSS keyframes handle simple entrance and ambient animations. The richer GSAP-driven attachments — stagger reveals, parallax scrolling, magnetic hover, the headline split-reveal — need their own respect for the user's preferences. The implementation lives in $lib/animations/gsap.ts:

typescriptsrc/lib/animations/gsap.ts · prefersReducedMotion helper
export function prefersReducedMotion(): boolean {  if (!browser) return false;  return window.matchMedia('(prefers-reduced-motion: reduce)').matches;}

Three different ways this helper gates motion in the attachment functions:

Pattern 1 · snap to final state

typescriptattachments.ts · fadeUp (paraphrased)
if (prefersReducedMotion()) {  gsap.set(node, { opacity: 1, y: 0, x: 0, scale: 1 });  return;}

For entrance animations that move an element into place — the element starts off-screen or invisible, and the animation brings it into view. The reduced-motion path uses gsap.set to immediately apply the post-animation state (opacity 1, position 0, scale 1) and return. The user sees the final state from the start. Same content, no motion.

Pattern 2 · skip the entire effect

typescriptattachments.ts · parallax (paraphrased)
if (!gsap) return;if (prefersReducedMotion()) return;

For purely-decorative effects — parallax background scrolling, cursor-following glows. There's no "final state" to snap to; the effect is motion. The reduced-motion path returns before the attachment registers any listeners. The element renders normally; the parallax/glow simply never starts.

Pattern 3 · pre-emptive guard in attachments.ts

Every attachment in the file checks prefersReducedMotion() at the top before doing any work. This is belt-and-braces over the CSS reset's universal blanket — the CSS guard catches CSS transitions and keyframe animations; the JS guards catch GSAP timelines that bypass CSS entirely (GSAP often animates inline styles directly, which can override the CSS reset's animation-duration: 0.01ms rule).

Why this matters · CSS guards don't cover JS animations

The universal * { animation-duration: 0.01ms !important } in reset.css overrides CSS-declared animations and transitions. It does not override JavaScript-driven motion. GSAP animating element.style.transform directly bypasses the CSS cascade entirely — the reset doesn't know that motion is happening. Without the JS-side check, a reduced-motion user with GSAP attachments still sees the full motion. The two guards are independent; both are required for full coverage.

CSS keyframes vs GSAP · when to reach for which

The project uses both — and the boundary between them isn't stylistic. Each has cases where it's the right answer:

Use caseCSS keyframesGSAP
Single-element entrance fade✅ .animate-fade-up — one class, done❌ overkill
Hover-state transitions✅ transition: background var(--dur-fast)❌ overkill
Infinite loops (spin, shimmer, marquee)✅ animation: spin 1s linear infinite❌ overkill, also harder to pause
Stagger reveal across many children⚠️ possible with animation-delay, but timing gets fragile✅ gsap.from(children, { stagger: 0.1 })

The pattern: CSS handles the simple, deterministic, looping cases; GSAP handles the multi-element coordination, scroll-coupled, and value-interpolation cases. The project's home page uses both — the Hero card has GSAP attachments for the parallax + cursor glow + split-reveal, but the shimmer in its loading state is a pure CSS animation.

Bundle cost · GSAP isn't free

GSAP core is ~50KB gzipped; ScrollTrigger adds another ~12KB. SplitText (used for the headline reveal) is a paid GSAP Bonus plugin (the project doesn't use it; the codebase implements its own simpler split-reveal). For a site that uses GSAP on most pages, the cost amortizes — the file is cached after first visit. For a site that uses GSAP on the home page only, it's worth asking whether the motion budget justifies it.

The project's calculus: the home page's first impression carries the brand (trading desk for serious traders — needs to feel premium, not sluggish), and GSAP enables motion polish that's measurably difficult to recreate in CSS. The trade-off is accepted explicitly. Internal admin pages that load GSAP via the same shared chunk pay the cost without using the features — that's the marginal waste, but the chunk caches and the admin user has typically already visited the home page, so the cost is paid once per session.

The principal-engineer move · always test with motion off

Every time you ship a new animation, set prefers-reduced-motion: reduce in DevTools (Chromium: Rendering panel → "Emulate CSS media feature prefers-reduced-motion") and confirm the page is still fully usable, fully visible, and fully navigable. The bug to avoid: an animation that fades content in is gated by prefersReducedMotion(), returns early, but never sets the final visible state — so reduced-motion users see permanently-invisible content. The fix is always the pattern shown above: gsap.set(node, { opacity: 1, ... }) before returning.

Pitfall · the most common reduced-motion bug

An entrance animation gated by prefersReducedMotion() looks like this in pseudo-code: start at opacity 0, animate to opacity 1. If reduced-motion, skip the animation. If you skip the animation without setting opacity to 1, the element stays at opacity 0 forever. The user sees invisible content. This is genuinely the most common reduced-motion accessibility bug — and it ships easily because the author tests with reduced-motion off and the bug only appears with it on. The fix is to always set the final state in the reduced-motion branch, not just return. Code reviews catch this exactly when the reviewer asks: "what does this look like with motion disabled?" Add that question to your review checklist.

Summary · the motion stack at a glance

  • Three easing tokens (--ease-out, --ease-in-out, --ease-spring) — the audit-survived vocabulary the codebase actually uses. The CSS keyword linear covers infinite loops directly.
  • Four duration tiers from 160ms (fast) to 680ms (slower). If you reach outside the scale, add a new tier here first.
  • Eight CSS keyframes in animations.css + their .animate-* utility classes. All use translate3d for GPU compositing.
  • Universal reduced-motion blanket guard in reset.css — 0.01ms not 0 for animationend event compatibility.
  • JS-side prefersReducedMotion() helper in gsap.ts, called at the top of every attachment in attachments.ts. CSS guards don't cover JS-driven motion.

  • CSS for simple/looping motion; GSAP for stagger/scroll/value-interpolation — the boundary is functional, not stylistic.

Chapter 10

Accessibility through CSS · focus, contrast, forced-colors, skip-link

Most of a web app's accessibility lives in HTML — proper semantics, ARIA roles, label associations. But a handful of accessibility decisions are made in CSS, and getting them wrong means keyboard users can't tell where they are on the page, Windows High Contrast Mode renders the design unreadable, or users with low vision get the same washed-out grays everyone else gets. This chapter is the four CSS-side accessibility surfaces the project handles, the user-preference media queries it doesn't yet, and the honest assessment of which deferrals are defensible.

Surface 1 · the focus-visible discipline

Keyboard users navigate the page by pressing Tab. The browser highlights whichever element currently has focus with an outline. That outline is non-negotiable — without it, a keyboard user cannot tell which link they're about to follow, which button they're about to press. Removing the outline (outline: none) without replacing it is one of the most common accessibility regressions on the web.

The CSS spec's modern answer is :focus-visible, a pseudo-class that only applies when the focus is keyboard-driven (or otherwise meaningful), not when it's a side-effect of a mouse click. Per MDN:

Quoted from MDN · :focus-visible

":focus-visible applies while an element matches the :focus pseudo-class AND the User Agent determines via heuristics that the focus should be made evident on the element."

"When a button is clicked using a pointing device, the focus is generally not visually indicated, but when a text box needing user input has focus, focus is indicated."

"While focus styles are always required when users are navigating the page with the keyboard or when focus is managed via scripts, focus styles are not required when the user knows where they are putting focus, such as when they use a pointing device such as a mouse or finger to physically set focus on an element."

The project's reset establishes the global pattern in nine lines:

csssrc/lib/styles/reset.css · focus
/* Focus */:focus {  outline: none;}:focus-visible {  outline: 2px solid var(--gold-400);  outline-offset: 2px;  border-radius: var(--radius-sm);}

Two rules. The first kills the browser's default focus outline (which varies between browsers and rarely matches the brand). The second restores it — but only when the browser decides the focus is meaningful. Mouse-clicking a button doesn't trigger :focus-visible; keyboard-tabbing to it does. Both rules are needed:

  • Without the first rule, mouse users see ugly browser-default focus rings every time they click — distracting and ugly.
  • Without the second rule, keyboard users see nothing when they Tab. Game over for accessibility.

The 2px gold outline + 2px offset is a deliberate choice. The offset means the outline doesn't touch the element's own border, so it reads as a clearly separate "this is focused" indicator rather than a thicker version of the element's edge. The gold matches the brand. The width is enough to be visible without being garish.

Pitfall · component-local :focus { outline: none } without restoring focus-visible

A component that wants to customize its focus look writes :focus { outline: none } and then forgets to add the :focus-visible restoration. The global reset's :focus-visible rule has lower specificity than the component's :focus rule (selector specificity counts pseudo-classes equally), so the component's rule wins and there's no focus ring at all. Every component that overrides focus needs both halves. A grep audit (grep -rn ":focus {" src/ | grep "outline: none" + checking each match for a sibling :focus-visible rule) is the canonical check.

Surface 2 · the skip-link pattern

A long page with a navbar, hero, content, and footer means a keyboard user pressing Tab from the page top has to step through every navbar link, every hero CTA, every social icon, just to reach the main content. The standard fix is a skip link — the very first focusable element on the page is an anchor that jumps straight to #main-content. The link is visually hidden by default but becomes visible when focused, so mouse users never see it and keyboard users see it on the first Tab.

The HTML side is one line in app.html:

htmlsrc/app.html (line 19)
<a class="skip-link" href="#main-content">Skip to content</a>

The CSS side is in the reset — the off-canvas → on-canvas slide on focus:

csssrc/lib/styles/reset.css · skip-link
.skip-link {  position: fixed;  top: -100px;             /* off-screen by default */  left: var(--space-4);  z-index: var(--z-tooltip);  padding: var(--space-3) var(--space-5);  background: var(--gold-500);  color: var(--surface-0);  border-radius: var(--radius-md);  font-weight: var(--weight-semibold);  transition: top var(--dur-base) var(--ease-out);}.skip-link:focus {  top: var(--space-4);    /* slides into view on focus */}

Three things make this implementation correct:

top: -100px (not display: none or visibility: hidden)
Both display: none and visibility: hidden would remove the link from the keyboard focus order. The element needs to be focusable while not being visible. Off-canvas positioning is the only technique that preserves focusability — the link is fully in the document, fully in the tab order, just rendered above the viewport. Tab moves to it; CSS slides it into view via the top transition.
:focus not :focus-visible
Unusual for this codebase — most focus styling uses :focus-visible. The skip link is the deliberate exception. If a mouse user somehow gives this link focus (programmatically, or via accessibility tooling), they should also see it. The skip link's purpose is "I want to go to main content" — it doesn't matter how focus arrived; the affordance should be visible.
Transition on top
The slide-down is a one-liner, but it matters — without it, the link teleports into view, which reads as broken. With it, the slide explicitly communicates "this thing just appeared because you tabbed to it." 240ms (--dur-base) with the --ease-out curve is the project's "responsive but graceful" combination.

Surface 3 · the .sr-only visually-hidden utility

Some content should be available to screen readers but not visible to sighted users — section heading hints, button labels supplementing icon-only buttons, status announcements. The canonical pattern, in utilities.css:

csssrc/lib/styles/utilities.css · .sr-only
.sr-only {  position: absolute;  width: 1px;  height: 1px;  padding: 0;  margin: -1px;  overflow: hidden;  clip: rect(0, 0, 0, 0);  white-space: nowrap;  border: 0;}

Every property in this rule earns its place — this is the WCAG-recommended pattern and each line solves a specific edge case:

position: absolute + width/height: 1px + margin: -1px
Together these collapse the element to a single pixel and offset it by one pixel — effectively making it 0×0 visually, but with non-zero dimensions so screen readers still recognize it as content.
overflow: hidden + clip: rect(0, 0, 0, 0)
The clip property is deprecated but still respected; the overflow is the modern equivalent. Both prevent text content from rendering past the 1×1 box. The deprecated clip stays for legacy AT compatibility.
white-space: nowrap
Prevents the 1×1 element from being shifted by line breaks in its parent.
border: 0
Defensive reset — if some component-level style added a border, it would be visible even in a 1×1 element.

Used in the project's dual-layout compare table (Ch 7's anchor): the <table> becomes .sr-only on narrow viewports while the visual stacked-card layout takes over. Screen readers still navigate the table semantically; sighted users see the cards. One semantic representation; two visual presentations.

Surface 4 · contrast as math, not vibes

WCAG 2.2 defines accessibility-level color contrast ratios. The project's actual ratios (computed in Ch 4) sit comfortably above the AAA floor:

Token pairContrast ratioWCAG level
--ink-200 on --surface-0 (body text)~13.5:1AAA + plenty of headroom
--gold-400 on --surface-0 (eyebrow text)~11.2:1AAA
--ink-300 on --surface-0 (muted body)~7.5:1AAA

The --ink-400 row is the discipline boundary: 4.8:1 passes AA for body text but the project never uses it for anything load-bearing. Hints, timestamps, fineprint — anywhere the information is supplementary. Status badges and CTA labels always use --ink-100 or --ink-200, both of which clear AAA at any size.

The Ch 4 callout on this still applies: run your token pairs through a contrast calculator once when the design system is locked, document the pass/fail in a one-page table, and PRs get checked against that table. The hard part is the discipline, not the math.

The honest gap · three user-preference media queries the project doesn't handle

This is the part of the chapter that doesn't have a green checkmark. The project handles prefers-reduced-motion (Ch 9). It doesn't handle three other user-preference signals that 2026 accessibility hygiene calls for:

prefers-contrast: more · users who need higher contrast

Some users — low vision, certain visual disorders, bright sunlight on a phone screen — explicitly set their OS to prefer higher contrast. The OS reports this preference to browsers; browsers expose it to CSS via the prefers-contrast media feature. The project doesn't have a single @media (prefers-contrast: more) rule. The "safe" fix is a single block in tokens.css:

csshypothetical · prefers-contrast escalation
@media (prefers-contrast: more) {  :root {    --border-default: rgba(248, 246, 241, 0.25);  /* was 0.10 */    --border-strong:  rgba(248, 246, 241, 0.40);  /* was 0.18 */    --ink-300:        #c0bbab;                  /* was #a39e8f — lifted toward ink-200 */  }}

Three token overrides cover the entire UI because every component reads through the tokens. The borders become 2.5× more visible; the muted text shifts toward the primary text contrast level. The cost: ~5 lines of CSS. The win: the project becomes WCAG AAA-equivalent for users who explicitly opted into high-contrast.

forced-colors: active · Windows High Contrast Mode

Per MDN:

Quoted from MDN · forced-colors

"forced-colors is used to detect if the user agent has enabled a forced colors mode where it enforces a user-chosen limited color palette on the page. An example of a forced colors mode is Windows High Contrast mode."

"When forced colors mode is active... color, background-color, border-color, outline-color... are forced to browser-specified system colors. box-shadow is forced to 'none'. background-image is forced to 'none'."

On Windows with High Contrast Mode active (used by ~3% of Windows users — those with low vision or sensitivity to glare), every project color token is overridden by the OS. The careful gold palette becomes the user's chosen "Highlight" color. Background images are stripped. Box shadows disappear.

This is mostly fine — Microsoft designed High Contrast Mode to provide a usable baseline regardless of the page's design. But two things break: (1) icon-only buttons that relied on a custom background to indicate state become invisible (the icon is the only signal, but icons rendered as SVG fill get forced too); (2) the focus outline, normally --gold-400, becomes the system's "Highlight" color — which on most High Contrast schemes is fine, but in some it's the same color as the button background, making the focus indicator invisible.

The defensive pattern:

csshypothetical · forced-colors hardening
@media (forced-colors: active) {  /* Use a system color for the focus ring so it adapts to user's palette */  :focus-visible { outline-color: Highlight; }  /* Buttons get an explicit border so they're visible without bg */  .btn { border: 2px solid ButtonText; }}

System colors like Highlight, ButtonText, Canvas, CanvasText are special keywords that resolve to the user's chosen colors. Using them keeps the design in sync with the user's preference instead of fighting it.

prefers-reduced-data: reduce · users on metered connections

The least-supported of the three (Safari and Firefox don't ship it yet) but worth wrapping if you have heavy decorative backgrounds. The project's .bg-grid and .bg-noise utilities apply pseudo-element-based grid lines and dot patterns over containers. They're polish, not content. On a metered connection, every pseudo-element repaint costs battery.

csshypothetical · reduced-data dropout
@media (prefers-reduced-data: reduce) {  .bg-grid, .bg-noise { background-image: none; }  .bg-grid::before, .bg-noise::after { display: none; }}

Why the project hasn't shipped these yet (and when it should)

The honest answer: the audit identified them, the project hasn't gotten to them, and the priority is correctly behind shipping product. Three reasons the deferral is defensible:

  1. The baseline is strong. Body-text contrast is 13.5:1 — three times the AAA floor. Users with prefers-contrast: more aren't getting a broken experience; they're getting "fine" when they could get "tailored." Different from getting "broken."
  2. Windows High Contrast Mode users get a usable baseline by default. Microsoft's color forcing is conservative — it strips backgrounds and shadows but doesn't make pages unusable. The project's lack of forced-colors rules means HCM users get the OS's standard treatment, which is functional. They miss the project's specific aesthetic, but they can use the app.
  3. The fixes are cheap when the work happens. The three media queries together would be ~15 lines of CSS in tokens.css. A half-day of work. The reason it's not "ship now" is opportunity cost — the same half-day shipped Ch 13's perf chapter, which moved 250k of JS off every page load. Both are accessibility wins; the latter is a bigger one (slow page loads are an accessibility issue too).

The honest "when this becomes a shipping requirement": (a) the first user complaint about contrast or HCM rendering, (b) the first enterprise customer whose accessibility audit requires AAA + forced-colors compliance, or (c) the first scheduled accessibility audit. None has happened yet; the work goes into the backlog with the audit document attached.

Why this matters · "we'll add it later" is not the same as "we haven't done it"

The L7 distinction: knowing what you haven't done is itself a deliverable. A team that doesn't ship prefers-contrast because they never considered it has an accessibility gap they don't know about — and the first user report blindsides them. A team that consciously deferred it, with a written audit and an estimate, has a planned omission. When the user report comes in, the team's response is "yes, we know, here's the 5-line fix; shipping today." The visible work is the same; the difference is whether you're surprised.

The text-wrap typography aside · accessibility-adjacent

Two newer CSS properties that aren't strictly accessibility-classified but visibly improve readability:

csssrc/lib/styles/typography.css (extracts)
h1, h2, h3, h4 {  /* ... font, weight, line-height ... */  text-wrap: balance;}/* In the lede pattern: */.lede { text-wrap: pretty; }

text-wrap: balance tells the browser to look at the entire heading and choose line breaks that produce visually-balanced line lengths — no single word orphaned on the last line. text-wrap: pretty applies a softer version of the same logic to body paragraphs — preventing single-word orphans but not trying to perfectly equalize line lengths.

Both shipped in all major browsers in 2023 and 2024. The accessibility connection: dyslexic readers and users with low working memory have measurably worse comprehension when single-word orphans break the visual rhythm of paragraphs. The fix is one CSS property per type role. The project sets text-wrap: balance on all heading levels and text-wrap: pretty on the lede pattern. Standard body paragraphs still default to the browser's normal wrapping — adding text-wrap: pretty globally would be a one-line improvement.

  • .sr-only utility. WCAG-canonical 1×1 + clip + overflow + margin pattern. Used by the dual-layout compare table to keep one semantic <table> available to screen readers while visual cards display.

Step 1 · vite.config.ts · enable the Lightning CSS transformer

typescriptvite.config.ts (CSS section)

Two Lightning CSS invocations, doing different jobs:

  • css.transformer: 'lightningcss' — runs Lightning CSS during the preprocess phase, on the CSS as Vite first sees it. This is where @custom-media expansion happens, where future-CSS features get compiled, where vendor prefixes get added. The drafts.customMedia: true flag opts into the draft-spec at-rule that Ch 5 narrated.
  • build.cssMinify: 'lightningcss' — runs Lightning CSS again at the very end of the build, on the final CSS chunks, to minify them. Same engine, different stage. The minifier stage is an optimization — minified output is ~15% smaller than esbuild's, mainly because Lightning CSS knows about modern CSS shorthands and can collapse them more aggressively. The two stages are independent; you can use either alone. The project uses both because they're complementary.

Step 2 · the CSS_FOUNDATION block · what gets injected into every chunk

javascriptsvelte.config.js · CSS_FOUNDATION
const CSS_FOUNDATION = `@layer reset, tokens, base, utilities, components, overrides;@custom-media --bp-sm (min-width: 480px);@custom-media --bp-md (min-width: 768px);@custom-media --bp-lg (min-width: 1024px);@custom-media --bp-xl (min-width: 1280px);@custom-media --bp-2xl (min-width: 1536px);@custom-media --bp-3xl (min-width: 1920px);@custom-media --bp-4xl (min-width: 2560px);@custom-media --bp-sm-down (max-width: 479.98px);@custom-media --bp-md-down (max-width: 767.98px);@custom-media --bp-lg-down (max-width: 1023.98px);`;

One JavaScript constant, two responsibilities — both established in earlier chapters:

  1. The @layer priority declaration ensures every CSS chunk the browser parses establishes the same cascade order. Without this in every chunk, the cascade-ordering bug from Ch 1 ships — utilities start beating components, layouts break intermittently. Re-declaring layer priority in subsequent chunks is a no-op per CSS spec, so the cost of injecting it everywhere is zero.
  2. The @custom-media declarations let every @media (--bp-md) reference resolve. Lightning CSS only sees @custom-media within a single file's scope; if the declarations aren't in the same file as the consumption, the consumption fails to compile.

The triple-source-of-truth caveat from Ch 5 still applies: this block is mirrored in src/lib/styles/breakpoints.css (for global imports that don't go through the Svelte preprocessor) and the values are mirrored in tokens.css as --bp-* CSS variables (for JS-side reads via matchMedia). Step 3 · the cssPrePreprocessor · injection + masking in one pass

javascriptsvelte.config.js · cssPrePreprocessor
const cssPrePreprocessor = {  name: 'tradeflex-css-pre',  style({ content }) {    const masked = content.replace(/:global\b/g, ':--svelte-global');    return {      code: `${CSS_FOUNDATION}\n@layer components {\n${masked}\n}\n`    };  }};

Eight lines. Three operations in order:

  1. Mask :global( to :--svelte-global( — the regex /:global\b/g matches the literal text :global followed by a word boundary (so it doesn't accidentally match a hypothetical :globally class). Every match becomes :--svelte-global, a custom CSS pseudo-class that Lightning CSS accepts silently per CSS Selectors Level 4's forward-compatibility rules.
  2. Prepend CSS_FOUNDATION — the layer priority + breakpoint declarations go first in the resulting CSS, so Lightning CSS sees them before any @media (--bp-*) reference.
  3. Wrap the (masked) user content in @layer components — every Svelte component's local styles land in the components layer of the cascade declared in Ch 1.

The output of this preprocessor for any Svelte file looks roughly like:

csswhat Lightning CSS receives (typical component)
@layer reset, tokens, base, utilities, components, overrides;@custom-media --bp-sm (min-width: 480px);/* ... other breakpoint declarations ... */@layer components {  /* the component's original styles, with :global masked: */  .card { padding: 1rem; }  @media (--bp-md) {    .card { padding: 1.5rem; }  }  .card :--svelte-global(img) { max-width: 100%; }}

Lightning CSS now has everything it needs: known custom-media references, a properly-declared layer order, no unrecognized pseudo-class syntax. It processes the file without warnings.

Step 4 · what Lightning CSS produces

After Lightning CSS runs, the same chunk looks like:

csswhat comes out of vitePreprocess
@layer reset,tokens,base,utilities;@layer components{  .card{padding:1rem}  @media (min-width:768px){.card{padding:1.5rem}}  .card :--svelte-global(img){max-width:100%}}

Three things changed and one thing didn't:

  • The @custom-media declarations are gone — they were only needed for Lightning CSS to do its work; the output uses the expanded form ((min-width: 768px)) directly.
  • The @media (--bp-md) reference expanded to @media (min-width: 768px). This is the whole point of enabling the transformer — the browser doesn't understand --bp-md at all, only the expanded form.
  • Whitespace collapsed — newlines removed, spaces between tokens dropped where the spec allows. Minifier-style compaction.
  • The layer order declaration was trimmed from six layers to four (reset,tokens,base,utilities). Lightning CSS knows that components is re-declared inline by the @layer components { ... } block, and overrides is never used in this chunk, so it elides them. The cascade semantics are preserved because components is implicitly added when the inline block is parsed.
  • The masked :--svelte-global(img) passes through unchanged — Lightning CSS doesn't recognize the custom pseudo-class but accepts it silently. This is the load-bearing trick.

Step 5 · the cssPostPreprocessor · unmask before Svelte sees it

javascriptsvelte.config.js · cssPostPreprocessor
const cssPostPreprocessor = {  name: 'tradeflex-css-post',  style({ content }) {    return { code: content.replace(/:--svelte-global\b/g, ':global') };  }};

The trivial counterpart. Take the masked-and-Lightning-CSS-processed output, restore :--svelte-global back to :global. Svelte's compiler runs next and sees :global(img) in the form it expects.

Svelte's compiler then does its scoping work: every selector that doesn't sit inside a :global() gets a .svelte-HASH class appended to it (so .card becomes .card.svelte-abc123). Selectors inside :global() stay unscoped (so :global(img) just becomes img, applying to every <img> descendant of the scoped element). The mask/unmask round trip is invisible to Svelte — the compiler sees the same input it would have seen if Lightning CSS hadn't been in the pipeline at all.

Why the mask works · CSS Selectors Level 4 forward-compatibility

The mask trick depends on a specific guarantee in the CSS Selectors Level 4 spec:

Quoted from CSS Selectors Level 4 (W3C)

"Pseudo-classes with the -- prefix are reserved for future extensions; user-defined custom pseudo-classes are not permitted. Implementations must treat unrecognized pseudo-classes with the -- prefix as not matching anything, rather than as a syntax error."

That's the contract that makes :--svelte-global(img) safe. Lightning CSS sees a pseudo-class it doesn't recognize, observes the -- prefix, and complies with the spec by accepting it silently (treating it as "matches nothing"). The selector parses successfully even though the pseudo-class is gibberish from Lightning CSS's perspective.

The post-preprocessor restores the real :global name before Svelte's compiler runs, so Svelte's compiler never sees the masked form. The masked form exists only during the ~10 milliseconds while Lightning CSS is processing the file. After that, the original syntax is restored and continues through the rest of the pipeline.

The upstream bug this works around

vite-plugin-svelte issue #1074 documents the underlying problem: when vitePreprocess routes CSS through Lightning CSS (as it does in transformer mode), Lightning CSS encounters Svelte-specific syntax (:global(), scoped selector hashes) that it has no knowledge of. The maintainers' acknowledged-but-unfixed plan is to integrate vite-plugin-svelte to process Svelte CSS before Lightning CSS sees it. Until that lands, the mask trick is the workaround.

The issue was opened in 2024. As of mid-2026 the cleanup is on the SvelteKit team's backlog but hasn't shipped. The mask trick is therefore expected to remain necessary for at least the lifetime of this project's current vite-plugin-svelte version. When the upstream fix ships, the entire cssPostPreprocessor can be deleted and the mask line removed from cssPrePreprocessor — a one-commit cleanup that will be obvious from the issue tracker.

The cost · build-time overhead

Adding three custom preprocessors to the Svelte build pipeline isn't free. The pre-preprocessor runs a regex replace + a template literal concatenation per file. The post-preprocessor runs one more regex replace. Lightning CSS runs twice (transformer + minifier) instead of once. Concrete numbers from a clean build of this project:

ConfigBuild time
Bare minimum (no transformer, no custom preprocessors)~6.5s
Transformer enabled, no custom preprocessors (breaks)—

The transformer + custom-preprocessor pipeline adds ~500ms to a full production build. For a one-time build, that's invisible. For HMR, the cost is per-file rather than per-build, and on a single component change it's ~5-10ms — well below perceptual threshold. The trade-off (single-source breakpoints + cascade ordering + Svelte syntax compatibility, in exchange for ~500ms / build) is unambiguously worth it.

Where this all comes from · the failed-attempt summary

The pipeline above is the fourth design that was attempted. The earlier three (narrated in Ch 5) were:

  1. Configure custom-media map in vite.config.ts — failed because Lightning CSS doesn't expose a JS-side map.
  2. Custom Vite plugin to prepend declarations to every CSS file — failed because vitePreprocess runs Lightning CSS before Vite's plugin transform hooks see .svelte files.
  3. Single Svelte preprocessor running after vitePreprocess — failed because Lightning CSS needed to see the declarations during its pass, not after.

Each failure narrowed the constraint set until the working design was the only one left: a preprocessor that runs before vitePreprocess to inject the foundation block and mask the Svelte syntax, plus a counterpart preprocessor that runs after to undo the masking.

What this chapter teaches beyond the specific pipeline

The principles transfer to any framework + CSS-tooling integration:

  1. Tools that own a file extension wrap standard build hooks. Vite plugins targeting .svelte files run after vite-plugin-svelte has already done its preprocessing. The same is true for .vue, .astro, .mdx. If you need to modify content inside a framework-specific file, you almost always need to go through the framework's preprocessor system, not Vite's plugin system.
  2. CSS processors don't understand framework-specific syntax. Lightning CSS doesn't know :global(). PostCSS doesn't know Svelte's :has(:focus-visible) rewriting. CSS Modules don't know about Astro's scoping. Whenever you put a generic CSS processor in a framework's pipeline, the framework-specific bits become a compatibility problem.
  3. Custom pseudo-classes with the :-- prefix are a generic escape hatch for hiding framework-specific syntax from generic CSS processors. The CSS Selectors Level 4 spec guarantees they parse silently. Any time a framework introduces a syntax extension that processors choke on, a mask/unmask pair is a workable fix.
  4. Bundler emission order is non-deterministic by default. Code-splitting bundlers (Vite, esbuild, Rolldown, webpack) all make this guarantee hard to maintain without intervention. Architecture decisions that depend on chunk-load order — including CSS cascade — need their invariants enforced in every chunk, not just the canonical one.

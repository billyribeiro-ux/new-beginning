Chapter 01

The Reactivity Mental Model

Before we touch a single rune, let's understand the problem reactivity actually solves. Once that clicks, every $state, $derived, and $effect in this codebase becomes obvious. Without it, runes feel like magic.

The problem: keeping the UI in sync with data

Every web app, at its heart, is the same loop:

data changes → UI must reflect the change ↑ │ │ ↓ user interacts ←──── browser renders new UI

For decades, web developers wrote this loop manually:

javascriptthe old way
let count = 0;function increment() {	count++;	document.querySelector('#display').textContent = count;}

This works for one element. But what if five places in the page depend on count? You have to update all five every time it changes. Forget one and the UI lies to the user. Multiply across thousands of variables and components, and the bugs become uncountable.

The solution: declarative reactivity

Modern UI frameworks invert the model. Instead of you imperatively syncing data → DOM, you declare what depends on what, and the framework keeps it in sync:

sveltethe new way
<script>	let count = $state(0);</script><p>{count}</p><button onclick={() => count++}>+1</button>

Three lines. The {count} in the template is bound to the reactive variable. When you mutate count, Svelte automatically re-renders every place that reads it. You never touch the DOM.

What is a rune?

⟁ Definition · Rune

A rune is a special symbol — a function-shaped keyword prefixed with $ — that you use to tell the Svelte 5 compiler "this variable participates in reactivity." Runes are part of the Svelte language, not values. You can't import them, can't pass them as arguments, can't assign them to other names.

The five runes you'll use 95% of the time:

RuneOne-line purposeWhere you use it
$state(…)Reactive memory — a variable the UI tracksCounters, form fields, anything that changes
$derived(…)A value computed from other state — recalculates automaticallyTotals, formatted strings, filtered lists
$effect(…)Side effects that run when state changeslocalStorage sync, canvas drawing, third-party libraries
$props()The inputs a component receives from its parentEvery component, top of <script>
$bindable(…)Lets a child component write back to a parent's stateReusable form inputs

Pull vs push reactivity

Older frameworks (early Vue, React class components) used push reactivity — when data changes, the framework eagerly recomputes every downstream value. This is expensive if nothing reads the value.

Svelte 5 uses push-pull: when state changes, dependencies are marked dirty instantly (the push), but derived values aren't recomputed until they're actually read (the pull). The result: zero wasted work.

⚡ Why this matters

You don't have to understand the algorithm to use it. But knowing it exists explains why a $derived that depends on an expensive computation only runs when something on the page actually reads it. The framework is doing less work than you think.

What's not a rune

To anchor your mental model, here's what's not reactive:

  • A plain let count = 0 — values change at runtime but the UI never re-renders.
  • A const — values can't change at all.
  • A function call result stored in a variable — captured once, doesn't update when inputs change.
  • Anything inside a $state object that was destructured — once you destructure, you break the reactive link.
⚠ Common pitfall

This breaks reactivity:

const { name } = $state({ name: 'Alex' });

The name variable holds the string 'Alex' — a primitive, not a reference. When you change the original object's name, your destructured copy is stale. Either keep the object reference (const user = $state({ name: 'Alex' }); user.name) or use $derived (let name = $derived(user.name)).

The runes in this project

Every .svelte.ts store, every component, every reactive helper in this codebase uses runes. By the end of this volume, you'll have read every store file (cart, ui, toast, prefersReducedMotion) and understand exactly how reactivity flows through them.

$state — reactive memory
Chapter 02

$state — reactive memory

Of all the runes, $state is the one you'll write a thousand times. It's the bedrock — every other rune builds on it. We're going to spend this chapter understanding every nuance: primitives, proxies, classes, raw state, and the destructure trap.

$state(initialValue?)
Declares a reactive variable. Reads and writes are tracked; the UI re-renders when the value changes.

The simplest form: a primitive

svelteexample
<script>	let count = $state(0);</script><button onclick={() => count++}>	clicks: {count}</button>

You read it like any number. You assign to it like any number. The UI tracks it.

Deep state: objects and arrays become proxies

When you pass an object or array to $state, you don't get the original back — you get a proxy. A proxy is a JavaScript feature that lets Svelte intercept every property read and write, so it knows what depends on what.

javascriptexample
let todos = $state([	{ done: false, text: 'add more todos' }]);// All of these trigger UI updates:todos[0].done = true;       // mutate a nested propertytodos.push({ done: false, text: 'eat lunch' });  // array methodtodos = [...todos, newTodo]; // full reassignment

That's why cart.lines.push(line) works even though it looks like you're mutating something. Behind the scenes the proxy notifies every subscriber.

⚠ Common pitfall · destructuring breaks reactivity

Once you destructure, the reference is gone:

let { done, text } = todos[0]; // captures values at this moment

todos[0].done = true; // does not affect the local 'done'

Either keep the reference (const line = todos[0]; line.done) or use $derived to keep the binding live.

Classes: $state on fields

This is the pattern used by every store in our codebase. Classes can declare reactive fields with $state in their body:

typescriptpattern
class Counter {	value = $state(0);	increment() {		this.value++;	}}export const counter = new Counter();

The compiler turns value into a getter/setter pair. Now counter.value is reactive everywhere it's read — including across files when you import counter.

𝓣 TypeScript Spotlight · typing class state

When the initial value is undefined at declaration time but you'll assign it in the constructor, use as:

count = $state() as number;

This tells TypeScript "trust me, this will be a number by the time anyone reads it."

$state.raw — opt out of deep reactivity

Sometimes you have a giant object you'll never mutate, only replace. Wrapping it in a deep proxy is wasted work. $state.raw gives you a shallow reactive variable — reassignment triggers updates; property mutations don't:

javascriptexample
let person = $state.raw({ name: 'Heraclitus', age: 49 });person.age++;          // 🚫 does nothingperson = { ...person, age: 50 };  // ✅ triggers update

Useful for large catalog data, configuration blobs, or anything immutable.

Cross-file state sharing

You can declare state in a .svelte.ts file and import it from anywhere. But there's a rule: you cannot export a reassignable state variable directly. The compiler can only intercept reads/writes within the file that declares it.

❌ This does not work

export let count = $state(0);

Importers see the original number, not the reactive binding. Updates won't propagate.

✅ This works

export const counter = $state({ value: 0 });

The reference is exported; mutations to counter.value propagate.

This is exactly why every store in our codebase exports a class instance: the variable holding the instance is const (never reassigned), but the instance's $state fields are mutable.

What you'll see in the project

You're about to see $state in three patterns:

  • Class fields in cart.svelte.ts, ui.svelte.ts, toast.svelte.ts — most common.
  • Component-local primitives in .svelte files for form fields, hover state, expand/collapse — Chapter 7+.
  • Imported singletons (e.g., cart) shared across components — backbone of cross-page features.

$derived — computed values
Chapter 03

$derived — computed values

If $state is reactive memory, $derived is reactive math. Any value you can express as a pure function of other state belongs in a $derived. The compiler then re-evaluates it automatically whenever its inputs change.

$derived(expression)
A read-only value computed from reactive state. Dependencies are tracked automatically by what you read inside the expression.

The classic example

svelteexample
<script>	let count = $state(0);	let doubled = $derived(count * 2);</script><p>{count} doubled is {doubled}</p>

Every time count changes, doubled recomputes — but only if something on the page actually reads it. The compiler tracks the dependency automatically: by reading count inside the expression, you registered it as a dependency.

Real example from this codebase

From cart.svelte.ts:

typescriptsrc/lib/stores/cart.svelte.ts
lines = $state<CartLine[]>([]);count = $derived(this.lines.reduce((n, l) => n + l.quantity, 0));subtotalCents = $derived(this.lines.reduce((n, l) => n + l.priceCents * l.quantity, 0));isEmpty = $derived(this.lines.length === 0);

Three derived values, all dependent on this.lines. Add a line → all three recompute (when read). Remove a line → same. Change a quantity → same. You never have to remember to update count when lines changes — the framework does it.

$derived.by — multi-statement expressions

If the expression doesn't fit on one line, use $derived.by with a function body:

javascriptexample
let total = $derived.by(() => {	let sum = 0;	for (const n of numbers) {		sum += n;	}	return sum;});

$derived(x) and $derived.by(() => x) are equivalent. Use .by when you need loops, branches, or multiple steps.

Rules of $derived

It must be pure
No mutations inside. The compiler will error if you try to update state inside a derived expression. (For that, use $effect — Chapter 4.)
Dependencies are automatic
Any reactive value you read synchronously inside becomes a dependency. Read it conditionally, and it's only a dependency on the path that read it.
It's lazy
The expression doesn't run until something reads the derived value. If no one reads doubled, the multiplication never happens.
You can override it
Since Svelte 5.25 you can temporarily reassign a $derived for optimistic-UI patterns (e.g., show a like count +1 before the server confirms). Reverts the next time a dependency changes.
⚡ Why this matters in our cart store

The cart's count badge in the navbar reads cart.count. The drawer's "subtotal" line reads cart.subtotalCents. The "Empty cart" placeholder reads cart.isEmpty. Three different pieces of UI in three different files — and they all stay in sync without us ever writing the word "update."

$effect — side effects
Chapter 04

$effect — side effects

$effect is the escape hatch. When you need to do something in response to state changing — write to localStorage, draw on a canvas, call an analytics endpoint — that's an effect. It is the most misused rune in Svelte 5, so we'll spend time learning when not to reach for it.

$effect(callback)
Runs the callback after every change to its tracked dependencies. Runs only in the browser, not during SSR.

The shape

svelteexample
$effect(() => {	// any reactive value read here becomes a dependency	document.title = `${cart.count} items in cart`;	// optional cleanup — runs before next re-run, and on destroy	return () => {		document.title = 'TradeFlex Trading';	};});

Two pieces: the work, and an optional cleanup function. Cleanups run before the next re-run (so they can undo what the previous run did) and when the component/effect is destroyed.

Real example: localStorage persistence in the cart

The cart writes to localStorage every time lines changes. Sneak peek of code we'll read in detail in Chapter 8:

typescriptsrc/lib/stores/cart.svelte.ts (excerpt)
$effect.root(() => {	$effect(() => {		try {			localStorage.setItem(STORAGE_KEY, JSON.stringify(this.lines));		} catch {			/* quota or private mode — ignore */		}	});});

Reading this.lines inside the effect registers it as a dependency. Add a line → effect re-runs → localStorage updated. Refresh the page → cart restored.

📖 Terminology · $effect.root

Normal $effect calls must be inside a tracking context — typically a component or another effect. But a store class isn't a component, so we need $effect.root to create a tracking context. Returns a destroy function we could call to stop the effect. We never call it for our cart because the cart lives for the page's lifetime.

When NOT to use $effect

The number-one mistake new Svelte 5 developers make is using $effect to synchronize state. Don't do this:

❌ Anti-pattern

let count = $state(0);

let doubled = $state(0);

$effect(() => { doubled = count * 2; });

Round-trip: state → effect → state. Inefficient, error-prone, can loop infinitely.

✅ Correct

let count = $state(0);

let doubled = $derived(count * 2);

Direct. Lazy. No effect overhead. The compiler stops you from doing it the wrong way.

⚠ Common pitfall · update loops

If an effect reads and writes the same state, you get an infinite loop. Svelte will detect and bail with an error in dev, but the lesson is structural: effects produce side effects, not state derivations. Reach for $derived first; only use $effect when there's a real side effect (DOM, network, storage, third-party library).

The legitimate use cases

  • Browser APIs: localStorage, sessionStorage, IndexedDB, the History API.
  • Third-party libraries: initializing a charting library, mounting a video player.
  • Canvas / WebGL drawing: redrawing on state change.
  • Subscriptions: WebSocket setup, EventSource, matchMedia listeners.
  • Analytics: firing a pageview when the route changes.
  • DOM measurements: reading element size after layout.

$effect.pre — before DOM updates

Normal $effect runs after the DOM has been updated. Rarely, you need to run code before the DOM updates (e.g., to read a scroll position before the new content shifts it). That's $effect.pre:

javascriptexample
$effect.pre(() => {	// runs before the DOM is updated	previousScroll = container.scrollTop;});

We don't use it in this codebase. Mentioned for completeness — you may see it elsewhere.

$props & $bindable
Chapter 05

$props & $bindable

Components communicate with their parents through props. In Svelte 5, you declare them with $props(). Sometimes a child needs to send a value back to the parent — that's what $bindable is for. These two runes shape every component file in the project.

$props() — declaring component inputs

$props()
Returns the object of inputs the parent passed. You almost always destructure it.

The pattern you'll see in every .svelte file in this project:

sveltepattern
<script lang="ts">	type Props = {		variant?: 'primary' | 'secondary';		size?: 'sm' | 'md' | 'lg';		disabled?: boolean;		onclick?: (e: MouseEvent) => void;		children?: Snippet;	};	let {		variant = 'primary',		size = 'md',		disabled = false,		onclick,		children	}: Props = $props();</script>

Three things happen here:

  1. The Props type defines the shape with TypeScript — required vs optional, allowed values.
  2. Destructuring at the top picks out each prop into a local variable.
  3. Default values (= 'primary') fire when the parent doesn't pass that prop.

Rest props

To capture "everything else the parent passed" (useful for wrapper components that forward unknown attributes):

sveltepattern
let { variant, size, ...rest }: Props = $props();<!-- spread the rest onto the underlying element --><button {...rest}></button>

This is how a custom Button can accept aria-label, data-* attributes, type, etc. — without explicitly declaring each one.

Props flow downward — and only downward

By default, props are one-way: parent → child. The child can read a prop but should never mutate the parent's state through it. Svelte will warn if you try.

When you need the child to communicate back, you have two clean options:

  1. Callback props — the parent passes a function, the child calls it. (onclick in the example above.)
  2. $bindable — for true two-way binding (next section).

$bindable — letting a child write back

$bindable(fallback?)
Marks a prop as two-way bindable. Parent uses bind:value={…}; child can write to it.

Used in our custom form inputs. From Input.svelte (sneak peek — full coverage in Vol 3):

sveltepattern
<script lang="ts">	type Props = {		value?: string;		label?: string;	};	let { value = $bindable(''), label }: Props = $props();</script><label>{label}</label><input bind:value={value} />

Now a parent uses it like this:

svelteparent
let email = $state('');<Input bind:value={email} label="Email" />

Typing into the input updates the parent's email directly. No callback plumbing.

⚡ Why this matters

Use $bindable sparingly. It's powerful but it makes data flow harder to trace — anyone reading the parent has to know which props mutate state and which don't. Reserve it for true input wrappers (text fields, checkboxes, switches) where two-way binding is the whole point.

Typing props with snippets

Snippets (next chapter) are the way to pass content into a component. They get their own type from svelte:

typescriptpattern
import type { Snippet } from 'svelte';type Props = {	children?: Snippet;            // content the parent puts inside <Comp>...</Comp>	iconLeft?: Snippet;            // optional named slot	row?: Snippet<[Item]>;         // snippet that takes an Item param};

The Snippet<[Item]> form is for snippets that take parameters — like the row snippet in a generic table.

Chapter 06

Snippets & {@render}

Snippets are how you pass markup into a component — the way you'd pass a function to higher-order code. Combined with {@render}, they replace slots from older frameworks and enable powerful composition patterns. Every component in this project uses them.

Defining a snippet

svelteexample
{#snippet greeting(name)}	<p>Hello, {name}!</p>{/snippet}{@render greeting('Alex')}{@render greeting('Priya')}

A snippet is a reusable chunk of markup with optional parameters. {@render} calls it like a function and inlines the result.

The implicit children snippet

This is the killer pattern. Any content between a component's opening and closing tags becomes a prop called children automatically:

Parent
svelteApp.svelte
<Button>Click me</Button>
Child
svelteButton.svelte
let { children } = $props();<button>{@render children?.()}</button>

The ?.() optional chaining is important — if the parent didn't pass any children, calling children() on undefined would crash. With ?. it just renders nothing.

Named snippets — multiple "slots"

Need an icon on the left of a button? An optional header on a table? Use named snippets:

svelteButton.svelte (parent usage)
<Button>	{#snippet iconLeft()}		<IconShoppingCart size={16} />	{/snippet}	Add to cart</Button>

Inside Button.svelte, you receive iconLeft as a prop and render it where you want:

svelteButton.svelte (child)
let { children, iconLeft, iconRight } = $props();<button>	{#if iconLeft}<span class="icon">{@render iconLeft()}</span>{/if}	{@render children?.()}	{#if iconRight}<span class="icon">{@render iconRight()}</span>{/if}</button>

This is the exact pattern our project's Button component uses. We'll see the full file in Vol 3.

Snippets with parameters

The killer feature. A snippet can take arguments, which lets a parent describe "render a row for each item" without the child knowing what fields exist:

svelteparent
<DataTable data={orders}>	{#snippet row(order)}		<td>{order.id}</td>		<td>{order.customerName}</td>		<td>{formatPrice(order.totalCents)}</td>	{/snippet}</DataTable>

DataTable doesn't know what an order looks like — it just calls {@render row(item)} for each item. Generic, type-safe, and zero boilerplate.

Why snippets > slots

FeatureSnippetsOld slots
Pass parameters✅ Any number❌ Limited
Type-safe✅ Via Snippet<[T]>⚠️ Partial
Reusable in same component✅ Define once, render anywhere❌
Passable as regular props✅ Just a value❌
⚡ Why this matters

Snippets are the single biggest reason to choose Svelte 5 over React, Vue, or earlier Svelte. They solve component composition cleanly without the contortions of "render props" or "named slots." Once you internalize them, you'll see them as the obvious solution.

.svelte.ts files
Chapter 07

.svelte.ts files — runes outside components

By default, runes only work inside .svelte files. But many things — shared state, custom reactive logic, the cart — need to live outside a component. The .svelte.ts extension is the answer: any TypeScript file with that suffix can use runes.

The rule

Svelte's compiler only processes runes in three file types:

  • *.svelte — the obvious one
  • *.svelte.ts — TypeScript modules with runes
  • *.svelte.js — JavaScript modules with runes

If you put a rune in a regular .ts file, the compiler ignores it and you get a runtime error. Always remember the .svelte infix.

What goes in them

Our four stores are .svelte.ts files. Each exports a class instance that holds reactive state shared across components. Other use cases:

  • Custom hooks (e.g., a reactive useMediaQuery helper)
  • Wrappers around browser APIs (clipboard, geolocation)
  • Form managers (multi-step wizards)
  • Anything that needs reactivity but isn't a component

The store class pattern

Every store in this project follows the same skeleton:

typescriptpattern
import { browser } from '$app/environment';class SomeStore {	field1 = $state(0);	field2 = $state(false);	computed = $derived(this.field1 * 2);	constructor() {		if (browser) {			// browser-only setup: load from localStorage, etc.			$effect.root(() => {				$effect(() => { /* persist */ });			});		}	}	action(input: number) {		this.field1 = input;	}}export const someStore = new SomeStore();

Three architectural choices encoded here:

Why a class?
You can't export let x = $state(...) and have it stay reactive across imports. A class instance is a single object — the reference doesn't change — but its $state fields can. So everyone who imports someStore shares the same reactive state.
Why the browser guard?
SvelteKit renders pages on the server too. localStorage doesn't exist there. The browser import from $app/environment is a boolean that's true in the browser, false during SSR. Wrapping browser-only setup in if (browser) prevents SSR crashes.
Why $effect.root?
A bare $effect needs a parent tracking context. A class isn't one. $effect.root creates one, so we can set up reactive side effects that live for the page's lifetime. We never call the returned destroy function — the store outlives the request.
⚡ Why this matters

Once you understand this pattern, the four store files in this project are 95% the same. The only differences are which fields they hold and what side effects they run. We're about to look at all four — but you already know the shape.

The cart store
Chapter 08

The cart store

Time to write our first real store. The cart holds line items, exposes computed totals, persists to localStorage, and is shared between the navbar badge, the cart drawer, and the cart page. It's the textbook example of cross-component reactive state.

What it needs to do

  • Hold a list of line items (products and subscription plans)
  • Expose count, subtotalCents, isEmpty as live computed values
  • Open / close the drawer (UI state)
  • Trigger a brief pulse animation when an item is added (the badge in the navbar)
  • Survive page refreshes via localStorage
  • Recover gracefully when localStorage is full or disabled

The full file, in three parts

Part 1: Types and storage helper

Create src/lib/stores/cart.svelte.ts and start with:

typescriptsrc/lib/stores/cart.svelte.ts (1/3)
import { browser } from '$app/environment';export type CartLineKind = 'product' | 'plan';export interface CartLine {	id: string;	kind: CartLineKind;	slug: string;	name: string;	subtitle?: string;	priceCents: number;	quantity: number;	maxQuantity?: number;}const STORAGE_KEY = 'tradeflex:cart:v1';function loadFromStorage(): CartLine[] {	if (!browser) return [];	try {		const raw = localStorage.getItem(STORAGE_KEY);		if (!raw) return [];		const parsed = JSON.parse(raw);		if (!Array.isArray(parsed)) return [];		return parsed.filter(			(l): l is CartLine =>				typeof l?.id === 'string' &&				typeof l?.name === 'string' &&				Number.isFinite(l?.priceCents) &&				Number.isFinite(l?.quantity)		);	} catch {		return [];	}}

Notes worth understanding here:

'tradeflex:cart:v1' as the storage key
Namespaced (tradeflex: prefix) and versioned (v1 suffix). If the schema ever changes — say we add a variantId field — bump to v2. Old saved carts are ignored cleanly instead of crashing the new code.
if (!browser) return []
During SSR, localStorage doesn't exist. Without this guard, the page would crash on the server. Return an empty cart on the server; hydrate from localStorage in the browser.
The filter with a type guard
Defensive parsing. localStorage values are strings — anyone could put garbage in. We re-validate each line against the expected shape; bad lines silently drop. This is how you make persistence antifragile.
try / catch { return [] }
localStorage can throw in private browsing mode and on quota-exceeded. We swallow and return an empty cart — better than a broken page.

Part 2: The class — fields, deriveds, and the constructor

typescriptsrc/lib/stores/cart.svelte.ts (2/3)
class CartStore {	lines = $state<CartLine[]>([]);	isOpen = $state(false);	pulseToken = $state(0); // increments on add → animates badge	count = $derived(this.lines.reduce((n, l) => n + l.quantity, 0));	subtotalCents = $derived(this.lines.reduce((n, l) => n + l.priceCents * l.quantity, 0));	isEmpty = $derived(this.lines.length === 0);	constructor() {		if (browser) {			this.lines = loadFromStorage();			$effect.root(() => {				$effect(() => {					try {						localStorage.setItem(STORAGE_KEY, JSON.stringify(this.lines));					} catch {						/* quota or private mode — ignore */					}				});			});		}	}

This is the heart of the store. Three reactive fields, three derived values, one effect that auto-persists. Let me unpack:

lines = $state<CartLine[]>([])
The array is wrapped in a proxy. this.lines.push(...), this.lines[0].quantity = 2, this.lines = [] all trigger updates.
isOpen
Drawer open/closed state. The navbar's "Cart" button calls cart.toggle(); the drawer's backdrop calls cart.close(). One source of truth.
pulseToken
An integer that increments every time something is added. The navbar's cart badge has a CSS keyframe animation keyed to {#key cart.pulseToken} — changing the token remounts the badge, which restarts the animation. Cheap, clever, no setTimeout needed.
The three deriveds
Pure functions of lines. We never set them; they always reflect the current state.
The constructor flow
1) hydrate from localStorage; 2) set up an effect that re-saves on every change. The effect runs once on mount (initial save), then again whenever lines changes.

What's happening

MediaQuery from svelte/reactivity
Svelte 5 ships built-in reactive wrappers for browser APIs that change over time. MediaQuery takes a CSS media query string and an optional SSR fallback, then exposes a reactive .current boolean. Reading .current inside a component / effect tracks it as a dependency; the value updates whenever the OS-level media query changes.
The #mq private field
The # prefix is JavaScript's true-private syntax (different from TypeScript's private keyword — this one is enforced at runtime by the engine). The MediaQuery instance is an implementation detail; only the boolean getter is part of our public API.
A getter, not a field

No browser guard needed
The second argument false is the SSR fallback. During server rendering, .current returns false (no media query matches). In the browser, it returns the real result and updates reactively. The store works on both sides without any conditional code.
⚡ Why this is cleaner than manual matchMedia

The earlier version of this file did window.matchMedia(...), listened to the 'change' event, and stored the result in $state. That worked, but it's seven lines and an event-listener leak waiting to happen. Always prefer the built-in reactive wrappers over hand-rolled subscriptions.

This is the spine of our accessibility story. Every GSAP attachment in animations/attachments.ts (Chapter 13) checks this signal before running. If the user wants reduced motion, they get instant transitions everywhere, without us having to remember to check in every component.

Validators with Zod

User input is the source of all evil. Every email field, every contact form, every password — must be validated before we trust it. We use Zod: write a schema once in TypeScript, get validation on the browser and on the server, plus auto-derived types. One source of truth.

Why Zod, and what makes it different

Zod is a runtime validator with TypeScript integration. You write a schema like z.string().email() and you get:

  • A function that validates input and returns either parsed data or detailed errors
  • A TypeScript type inferred automatically from the schema
  • The same code working in browser and server (we use it in both)

This eliminates the classic split where you have form validation on the frontend, separate validation on the backend, and they drift apart. With Zod, the schema is the single source of truth.

That first config({ jitless: true }) line

Zod v4 ships an optional JIT compiler that speeds up validation by code-generating a per-schema fast path the first time you call it. The JIT relies on new Function(), which is blocked by Content-Security-Policy headers that forbid 'unsafe-eval' — the kind of CSP a security-conscious deployment uses. jitless: true opts out of the JIT, paying a small performance cost for full CSP compatibility. We do this once at the top of the file so every schema in the module follows suit.

Composable schemas

Notice emailSchema, passwordSchema, nameSchema are declared once at the top — then reused inside leadCaptureSchema, contactSchema, loginSchema, signupSchema. This is the architectural payoff: change the password rules once, every form that requires a password gets the update.

Two patterns worth pointing out

The honeypot field

Notice leadCaptureSchema and contactSchema have a website field with max(0) — "must be exactly the empty string." This is a honeypot: in our HTML form we render a hidden <input name="website">. Real users never see it (it's CSS-hidden), but bots fill in every field they find. If website comes back non-empty, Zod rejects with "Bot detected" and we silently drop the submission.

⚡ Why honeypots beat captchas

Captchas are bad UX (slow, accessibility-hostile, increasingly bypassed by AI). Honeypots are invisible to humans and catch ~95% of spam bots with zero friction. We add them everywhere user submissions are accepted.

Cross-field validation with .refine

In signupSchema and resetPasswordSchema, we chain a top-level .refine that compares password against confirm. The path: ['confirm'] tells Zod to attach the error to the confirm field specifically, so the message renders next to the right input.

GSAP attachments

This chapter is where the site goes from "static" to "cinematic." GSAP — GreenSock Animation Platform — is the most powerful JavaScript animator. Wrapped inside Svelte 5 attachments, it gives us seven reusable scroll effects we'll sprinkle across every page in Vol 3.

The plan

Two files:

  1. src/lib/animations/gsap.ts — lazy initialization of GSAP and ScrollTrigger (so they only load on the client)
  2. src/lib/animations/attachments.ts — seven attachment factories: fadeUp, stagger, splitReveal, magnetic, numberTicker, parallax, cursorGlow

1. gsap.ts — singleton initialization

GSAP and ScrollTrigger must be registered exactly once per page. We hide that bookkeeping behind two helper functions. Create src/lib/animations/gsap.ts:

typescriptsrc/lib/animations/gsap.ts
import { browser } from '$app/environment';import { gsap } from 'gsap';import { ScrollTrigger } from 'gsap/ScrollTrigger';let registered = false;export function getGsap(): typeof gsap | null {	if (!browser) return null;	if (!registered) {		gsap.registerPlugin(ScrollTrigger);		registered = true;		gsap.defaults({ ease: 'expo.out', duration: 1.1 });	}	return gsap;}export function getScrollTrigger() {	if (!browser) return null;	getGsap();	return ScrollTrigger;}export function prefersReducedMotion(): boolean {	if (!browser) return false;	return window.matchMedia('(prefers-reduced-motion: reduce)').matches;}

What it does

registered module-level flag
Set once on first getGsap() call. Subsequent calls skip the registration. Without this, calling registerPlugin(ScrollTrigger) multiple times would log a warning.
SSR guard at the top of every export
GSAP touches window. We return null during SSR and let each attachment handle the null case (typically: skip the animation, leave the element in its end state).
gsap.defaults({ ease: 'expo.out', duration: 1.1 })
Project-wide motion DNA. Every animation without explicit ease or duration uses these. Change them once → every attachment respects it.
Why a synchronous prefersReducedMotion() here too
The reactive store from Chapter 9 is for components. Inside an attachment factory we just want a quick "is reduced motion on right now?" check before constructing a tween — synchronous is simpler.

⟁ Attachment recap

An Attachment is a function (element: T) => void | (() => void). You pass it to an element with {@attach myAttachment}. The function runs when the element mounts; its returned cleanup runs when the element unmounts. We covered the rune in Volume 1's Chapter 6 sneak peek; here we put it to industrial use.

Attachment 1: fadeUp — scroll-triggered fade and rise

typescriptsrc/lib/animations/attachments.ts (fadeUp)
/** Fade-up reveal on scroll entrance. */export function fadeUp(opts: RevealOpts = {}): Attachment<HTMLElement> {	return (node) => {		const gsap = getGsap();		const ST = getScrollTrigger();		if (!gsap || !ST) {			node.style.opacity = '1';			node.style.transform = 'none';			return;		}		if (prefersReducedMotion()) {			gsap.set(node, { opacity: 1, y: 0, x: 0, scale: 1 });			return;		}		const ctx = gsap.context(() => {			gsap.fromTo(				node,				{ opacity: 0, y: opts.y ?? 28, x: opts.x ?? 0, scale: opts.scale ?? 1 },				{					opacity: 1,					y: 0,					x: 0,					scale: 1,					duration: opts.duration ?? 1.1,					delay: opts.delay ?? 0,					ease: 'expo.out',					scrollTrigger: {						trigger: node,						start: opts.start ?? Three guards (no GSAP available → bail with element visible; reduced motion → set to end state instantly; otherwise → run the animation). The gsap.context(() => {...}, node) wrapping is the magic: every tween created inside is registered against this node, and the returned ctx.revert() cleans all of them up when the element unmounts. No leaked tweens, no zombie ScrollTriggers.

⚡ Why scroll-triggered reveal matters

Server-rendered HTML appears fully formed on first paint — which means the user sees the whole page at once and their eye doesn't know where to land. A subtle fade-up as each section enters the viewport directs attention. The reader's gaze follows the motion. Done right, it's the difference between a static brochure and a living document.

Why two-layer spans (wrap + inner)

The trick is the clip mask. The outer span.split-word has overflow: hidden; the inner span starts at y: 110% (110% of its own height below the visible area). When we tween y: 0%, the inner span slides up from below the outer's clipping rectangle. The effect: each word appears to rise into existence rather than fade in. Pure CSS would not give you this — it requires DOM manipulation, hence the helper.

⚠ The original-HTML restore matters

Notice the cleanup: ctx.revert() AND node.innerHTML = original. We mutated the DOM (replaced the text with spans). On unmount we restore the original markup so SvelteKit's hydration tracking doesn't get confused, and so users with reduced motion who toggle it on after page load see normal text.

/** Magnetic hover effect on premium CTAs and pricing cards. */export function magnetic(opts: { strength?: number } = {}): Attachment<HTMLElement> {	return (node) => {		if (prefersReducedMotion()) return;		const gsap = getGsap();		if (!gsap) return;		const strength = opts.strength ?? 0.35;		let raf = 0;		const onMove = (e: PointerEvent) => {			const rect = node.getBoundingClientRect();			const cx = rect.left + rect.width / 2;			const cy = rect.top + rect.height / 2;			const dx = (e.clientX - cx) * strength;			const dy = (e.clientY - cy) * strength;			cancelAnimationFrame(raf);			raf = requestAnimationFrame(() => {				gsap.to(node, { x: dx, y: dy, duration: 0.6, ease: 'expo.out' });			});		};		const onLeave = () => {			gsap.to(node, { x: 0, y: 0, duration: 0.8, ease: 'elastic.out(1, 0.4)' });		};		node.addEventListener('pointermove', onMove);		node.addEventListener('pointerleave', onLeave);		return () => {			node.removeEventListener('pointermove', onMove);			node.removeEventListener('pointerleave', onLeave);			gsap.set(node, { x: 0, y: 0 });		};	};}
